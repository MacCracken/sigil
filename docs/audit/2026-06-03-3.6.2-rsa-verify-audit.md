## Threat model / timing (load-bearing framing)

RSA **verification** computes `m = s^e mod n` from the signature `s`
and the PUBLIC key `(n, e)` and checks the encoding. There is **no
secret operand**, so:
- **Constant-time is not required** here. `bn_mod`'s data-dependent
  compare/subtract and `bn_modexp`'s exponent-bit branches act only on
  public values; there is nothing to leak. This is called out
  explicitly in both module headers so the engine is **not**
  mis-reused for RSA *signing* (private exponent `d`), which will need
  blinding + a constant-time ladder in its own bite.
- No private key material is handled, so there is no zeroization
  obligation on this path.
- Not on the parallel batch-verify path (that is Ed25519), so the
  static working buffers are plain single-threaded scratch — the same
  posture as sigil's HMAC/HKDF/PRF surface. Documented; if RSA ever
  joins a parallel path it needs the 3.6 bank treatment.

## Arithmetic correctness

`bn_modexp` was validated against Python `pow(base, exp, mod)`:
- textbook `4^13 mod 497 = 445`;
- a 256-bit `s^65537 mod n`;
- a **2048-bit** `s^65537 mod n` (full RSA-2048 width, 32 limbs);
- edge cases `base^0 = 1` and `0^e = 0`.
Serialization (`bn_from_be`/`bn_to_be`) round-trips. The reduction's
running remainder uses `m_limbs + 1` limbs so the pre-subtract `<< 1`
cannot overflow; `bn_bitlen`-driven loop bound makes `x mod n` exact
for `x < n` (produces `x`) and for `x = 0` (produces 0).

## PKCS#1 v1.5 verify hygiene (the CVE-class concern)

The check **reconstructs the full expected** `EM = 0x00 01 || PS ||
0x00 || DigestInfo || H` and compares all `k` octets (`ct_eq_bytes`),
rather than parsing the recovered EM and skipping fields. This is the
defense against the classic RSA-verify forgery family:
- **Bleichenbacher'06 / "BERserk"** low-exponent (`e=3`) forgeries and
  variants rely on lax verifiers that locate the hash and ignore
  trailing/garbage bytes or accept a short `PS`. Rebuilding and
  comparing the entire EM rejects any extra data, wrong `PS` length,
  missing `0x00` separator, or shifted hash.
- **Algorithm substitution** is prevented by fixing the DigestInfo
  prefix per hash; a SHA-256 signature is rejected by the SHA-384
  verify entry (tested) and vice-versa.
Additional input validation (RFC 8017 §8.2.2):
- `sig_len != k` rejected (signature must be exactly `k` octets);
- `s >= n` rejected (RSAVP1 range check, §5.2.1);
- minimum 8-octet `PS` enforced (`k >= |T| + 11`), else reject;
- `n_len > 512` (i.e. > RSA-4096) rejected — bounds the buffers.
Validated against a real RSA-2048 key (pure-Python keygen + manual
PKCS#1 v1.5 encode, each signature self-checked `pow(s,e,n)==EM`):
valid SHA-256 and SHA-384 signatures accepted; one-bit-tampered sig,
correct sig over a different message, wrong-length sig, and
hash-algorithm mismatch all rejected.

## Buffer safety

- All bignum buffers are sized for RSA-4096: numbers `512` bytes (64
  limbs), products `1024` bytes (128 limbs), the reduction remainder
  `520` bytes (65 limbs). `bn_mul` writes exactly `2*limbs`; `bn_mod`'s
  shift/compare/subtract run over `m_limbs+1`. `_rsa_*` buffers are
  `512` bytes; `n_len > 512` is rejected up front, and `limbs =
  (n_len+7)/8 ≤ 64`. The EM reconstruction writes exactly `n_len`
  bytes (`2 + ps_len + 1 + prefix_len + hash_len = n_len`).
- `bn_from_be` drops bytes beyond the buffer width (caller passes
  matched widths); `bn_to_be` left-zero-pads.

## Findings

- **INFO-1** — performance: schoolbook `O(n^2)` multiply + `O(bits)`
  reduction. A 2048-bit verify is correct but unoptimized; a
  Montgomery/Barrett bite is queued (does not block verify, which is
  one modexp with a tiny public exponent).
- **INFO-2** — `.bss` growth from the engine + RSA static scratch. No
  heap / no per-call alloc; fine for a trust daemon.

Zero CRITICAL / HIGH / MEDIUM / LOW. Audit-floor LOWs from prior
cycles are unaffected.
