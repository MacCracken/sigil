## Step 1 — Input validation

### `src/x509.cyr:_xp_parse_spki`

- AlgorithmIdentifier outer SEQUENCE tag verified
  (`x509.cyr:495`).
- `id-ecPublicKey` OID required (`x509.cyr:511`).
- Parameters OID must be **exactly** prime256v1 or secp384r1
  (`x509.cyr:519-528`). Any other curve OID → -1.
- BIT STRING length validated against the per-curve expected
  size: `2 + pk_len` where `pk_len ∈ {64, 96}`
  (`x509.cyr:540`). Off-by-one between curves cannot slip
  through.
- Unused-bits byte must be 0 (`x509.cyr:541`).
- Uncompressed-point marker (0x04) required (`x509.cyr:542`).
- Compressed (0x02/0x03) and hybrid (0x06/0x07) point forms
  rejected.

### `src/x509.cyr:_x509_verify_link`

NEW guard at `x509.cyr:850`: issuer cert must have
`X509_CURVE_P256`. Without this, a P-384 issuer would be passed
to `ecdsa_p256_verify` whose pubkey-load step would silently
consume the lower 64 bytes of the 96-byte P-384 pubkey slot —
the verify primitive cannot detect the curve mismatch. The
explicit guard rejects this combination at the chain-walk layer.

### `src/sev_snp.cyr:snp_report_verify_full`

- `vcek_chain_pem_len <= 0` rejected first (`sev_snp.cyr:300`).
- `ark_root_der_len <= 0` rejected (`sev_snp.cyr:301`).
- Re-verify `report.signature_algo == ECDSA_P384_SHA384`
  (`sev_snp.cyr:307`) — defence-in-depth: caller might mutate
  the parsed struct between parse and verify.
- `pem_decode_certs < 1` → reject (`sev_snp.cyr:315`).
- Each `x509_parse` failure → reject (`sev_snp.cyr:325`).
- Root `x509_parse` failure → reject (`sev_snp.cyr:331`).
- `x509_verify_chain` failure → reject (`sev_snp.cyr:354`).
- **NEW curve gate**: leaf must be P-384 (`sev_snp.cyr:362`)
  AND pubkey_len must be 96 (`sev_snp.cyr:363`). A P-256 leaf
  parses cleanly via the new SPKI path, so this gate prevents
  silently chaining the lower 64 bytes of the pubkey slot into
  `snp_report_verify`'s P-384 path.

## Step 2 — Buffer safety

### `src/x509.cyr:_xp_parse_spki`

- `pk_len` is computed from validated curve (`64` or `96`); no
  attacker-controlled width.
- Pubkey write loop is bounded `i < pk_len` (`x509.cyr:551`).
  The slot at `cert + 96` is exactly 96 bytes (per the new
  struct layout) — `pk_len ≤ 96` always.
- The 32-byte P-256 zero-pad (upper bytes of the slot) comes
  from the `memset(c, 0, 256)` in `x509_cert_alloc` —
  re-verified inert.
- `qx_limbs[32]` / `qy_limbs[32]` for P-256 → 32-byte arrays.
  `qx_limbs6[48]` / `qy_limbs6[48]` for P-384 → 48-byte arrays.
  `u256_load_be` reads 32 BE bytes; `u384_load_be` reads 48 BE
  bytes; both write 6/4 i64 limbs into the corresponding stack
  arrays. Bounds-clean.

### `src/sev_snp.cyr:snp_report_verify_full`

- `chain[64]`, `parsed[4]`, `inters[4]` arrays are
  function-frame statics (CLAUDE.md quirk #1 — static-array
  semantics). `memset(&chain, 0, 64)` /
  `memset(&inters, 0, 32)` at entry zero them. Writes bounded
  by `count < max_certs (4)` for chain entries and `inters_n <
  4` for intermediates.

### Cert struct layout

Old layout used bytes 0–200; new layout uses bytes 0–248. The
remaining 8 bytes (248–256) stay zero from `memset(c, 0, 256)`.
No write-past-end risk.

## Step 3 — Constant-time audit

- The new curve-dispatch branch (`if curve == X509_CURVE_P256
  ... else ...`) operates on a PUBLIC value derived from the
  cert's parameter OID. Not a secret-data branch.
- The P-384 on-curve check (`pt_p384_is_on_curve`) is audited
  as constant-time in 3.2.4 — same shape as `pt_is_on_curve`
  for P-256.
- `_x509_verify_link`'s new `x509_cert_curve(issuer_cert) !=
  X509_CURVE_P256` check operates on a PUBLIC value (curve type
  derived from the cert).
- `_snp_cert_is_self_issued` operates on PUBLIC DN bytes; not
  constant-time, deliberately.

## Step 6 — Pointer validation

- `pem_decode_certs` returns DER pointers into a bump-allocated
  pool that lives for the process lifetime; sigil writes nothing
  to those bytes after parsing. The caller cannot influence the
  DER pool's lifetime via the verify_full surface (the LOW
  finding).
- `vcek_chain_pem` and `ark_root_der` are caller-owned
  read-only buffers; sigil reads exactly the documented lengths.
- The 96-byte pubkey extracted from a P-384 cert is stored at
  `cert + 96..192`. The accessor `x509_cert_pubkey(c)` returns
  the address; consumers must use `x509_cert_pubkey_len(c)` to
  determine the bytes to read. NEW gate in
  `snp_report_verify_full` enforces this matches snp's 96-byte
  expectation.

## Step 9 — Known CVE check

- **CVE-style EC SPKI parameter confusion** (e.g., naming
  P-256 but encoding P-384 bytes): the SPKI BIT STRING length
  check (`bs_len != 2 + pk_len`) ties the byte width to the
  parameter OID-derived `pk_len`. A cert claiming P-256 with a
  98-byte BIT STRING is rejected at parse.
- **Cofactor / wrong-curve attack** (Lim-Lee / invalid-curve):
  the on-curve check uses curve-specific primitives
  (`pt_is_on_curve` for P-256, `pt_p384_is_on_curve` for P-384)
  — each verifies `y^2 = x^3 + ax + b mod p` with the curve's
  own `a, b, p`. A P-256 cert with attacker-chosen Qx, Qy
  cannot satisfy the P-384 equation and vice-versa.
- **Trust-anchor confusion**: the new curve gate in
  `_x509_verify_link` blocks the only path where a P-384 cert
  could be silently used as an issuer (which would
  mis-dispatch the signature verify primitive). A consumer
  passing a P-384 root via `*_verify_full(_, root_der, _, _)`
  has the parse succeed (the root struct holds curve=P384)
  but the first chain link's `_x509_verify_link` invocation
  rejects.

## Step 10 — Findings

### LOW-1 (NEW) — `snp_report_verify_full` bump-alloc drift

**Where**: `src/sev_snp.cyr:snp_report_verify_full`.

**Note**: Same shape as the 3.4.0 LOW for sgx/tdx verify_full:
(a) `pem_decode_certs` DER pool sized to `vcek_chain_pem_len`
(~1.3 KB for our synthesized 2-cert chain; real AMD chains
~3–5 KB), (b) up to 4 × 256-byte `x509_cert` structs, (c) 4 ×
64-byte raw_sig buffers inside `x509_parse`, (d) 1 × 256-byte
root cert + 1 × 64-byte raw_sig for the root. Per-call: ~5–8 KB
bump drift.

**Action**: Joins the existing six LOWs. All seven close at
3.6 via the unified `_into` API.

### INFO-1 — Mixed-curve chain restriction

**Where**: `src/x509.cyr:_x509_verify_link` (new
`X509_CURVE_P256` issuer gate at line 850).

**Note**: Chain-link signatures must be ECDSA-SHA256 (the only
`X509_SIG_*` value sigil's x509 recognises). This implies the
issuer's pubkey must be P-256 (the curve `ecdsa_p256_verify`
operates on). The 3.4.1 work lets the LEAF be P-384, but every
issuer in the chain — root, intermediates, anything that signs
a subsequent link — must remain P-256.

**Implication**: Real AMD VCEK chains use RSA-signed ARK/ASK
links and an ECDSA-signed VCEK leaf. Sigil can verify the
VCEK leaf signed by an ECDSA-SHA256-P256 issuer (synthetic test
shape) but cannot walk the real AMD ARK/ASK chain (RSA not
supported). Real-deployment integrations should hand sigil only
the ASK → VCEK fragment after an external pre-step verifies
ARK → ASK.

**Action**: Documented in the new `X509_CURVE_P256` /
`X509_CURVE_P384` constant block in `src/x509.cyr` and in
`src/sev_snp.cyr`'s file header. A future cycle that adds RSA
or P-384 chain-link verify would relax this — out of scope for
3.4.1.

### INFO-2 — In-quote ARK cert treatment

**Where**: `src/sev_snp.cyr:_snp_cert_is_self_issued`,
`src/sev_snp.cyr:snp_report_verify_full:342-345`.

**Note**: Mirrors the SGX/TDX shape. If the VCEK chain delivered
by the caller includes a copy of the AMD Root Key (self-issued
ARK), sigil drops it and anchors verification on the
caller-supplied `ark_root_der`. The in-quote ARK is never
compared to or trusted on its own merits.

**Action**: None. Matches the trust-establishment posture
documented for SGX/TDX in 3.4.0 INFO-2.

### INFO-3 — X509Cert struct footprint expansion

**Where**: `src/x509.cyr` — cert struct layout comment block
(lines 191–223).

**Note**: The pubkey slot grew from 64 to 96 bytes; subsequent
fields (sig_algo, sig_off, sig_len, is_ca, path_len) shifted
+32; new fields (curve, pubkey_len) added. Used footprint grew
from ~200 of 256 bytes to ~248 of 256 bytes. Reserved tail
shrunk from 56 to 8 bytes.

**Implication**: Any future addition to X509Cert (issuer key
identifier, AKI/SKI extensions, custom-extension flags) must
fit in the remaining 8 bytes or trigger another struct expansion.
The current 256-byte struct allocation budget is approaching
saturation.

**Action**: Note in audit only; not a blocker. When the next
field surfaces, the layout will either need an 8-byte slot
slice (single i64) or a heap-extension pointer (16 bytes).
