## 10-step Security Hardening checklist

1. **Input validation.** Both entry points reject negative
   `aad_len` / `pt_len` / `ct_len` with `ERR_INVALID_INPUT`. Zero
   lengths are valid (empty AAD and/or empty message) and handled
   by the pad/loop arithmetic.
2. **Buffer safety.** `_cp_tag` sizes the mac_data buffer
   `mac_len = aad_len + pad16(aad_len) + ct_len + pad16(ct_len) +
   16` and `memset`s it to zero (filling both pad regions). Writes:
   AAD at offset 0 (`aad_len` bytes), ciphertext at
   `aad_len + pad` (`ct_len` bytes), two le64 length fields in the
   final 16 bytes — total reach exactly `mac_len`. `otk` is 32
   bytes copied from the 64-byte keystream block; `expected` is 16
   bytes; `ct_out`/`pt_out` are caller-sized to the message length.
3. **Constant-time.** Tag verification uses the stdlib
   `ct_eq_bytes` (no early exit). The one data-dependent branch in
   decrypt is on the *tag-valid bit* — the accept/reject decision
   the AEAD must expose — not on key or plaintext content.
   Plaintext is derived only on the accept path.
4. **Key material zeroization.** The one-time Poly1305 key `otk`,
   the `_cp_keygen` keystream `block`, and the `expected` tag are
   `secret var` (zeroized on return). The mac_data scratch holds
   only AAD + ciphertext (public) — not zeroized, by design.
5. **Syscall review.** None.
6. **Pointer validation.** No untrusted deref; the only heap
   buffer is the bounded `fl_alloc` mac_data (see INFO-2).
7. **No command injection.** None.
8. **No path traversal.** None.
9. **Known-CVE check.** (a) **Plaintext release before
   authentication** — the classic AEAD footgun; avoided, decrypt
   returns `ERR_INTEGRITY_MISMATCH` and zeroes `pt_out` on tag
   failure, deriving plaintext only after a valid tag. (b)
   **Variable-time tag compare** (forgery oracle) — avoided via
   `ct_eq_bytes`. (c) **Length-field ambiguity** — the le64(aad)
   ‖ le64(ct) trailer is present and correctly ordered, preventing
   AAD/ciphertext boundary-shifting forgeries. (d) **Nonce reuse**
   — see INFO-1.
10. **Findings filed** — below.

## Findings

### INFO-1 — Nonce reuse is a caller obligation

Reusing a (key, nonce) pair is catastrophic for any AEAD: the
ChaCha20 keystream repeats *and* the derived one-time Poly1305 key
repeats, enabling both plaintext recovery and tag forgery. The
construction derives a fresh Poly1305 key per nonce (satisfying
Poly1305's one-time-key contract for distinct nonces), but cannot
detect reuse. Standard AEAD caveat; the caller must guarantee
nonce uniqueness per key. No code change.

### INFO-2 — mac_data `fl_alloc` is unguarded

`_cp_tag` does not null-check the `fl_alloc(mac_len)` result before
`memset` — an allocator exhaustion would fault. This matches the
existing sigil convention (e.g. `aes_gcm` round-key `alloc`,
ed25519 `ge_alloc` are likewise unguarded); sigil's threat model
assumes allocator success for these small, short-lived buffers.
Noted for consistency, no change. **The allocation uses `fl_alloc`
/ `fl_free` (per-call scratch, freed before return) — NOT the bump
allocator — so it does not extend the v3.7 bump-allocator audit
floor.**

### INFO-3 — Serial-only static scratch

The composition inherits the function-scope static-array shape of
its primitives (poly1305 `block`, chacha20 `st`/`ws`/`ks`); each is
fully written before reading, so serial use is correct. Not yet on
a multi-threaded path (v3.6 per-worker-scratch cycle).
