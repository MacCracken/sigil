# Sigil Security Audit — 2026-04-13

**Auditor**: Internal (Claude/Opus)
**Version audited**: 2.0.1
**Scope**: All src/*.cyr modules and fuzz/ harness
**Method**: 10-step Security Hardening checklist (CLAUDE.md)

| Severity  | Count | Notes                                    |
|-----------|-------|------------------------------------------|
| CRITICAL  | 2     | Silent weak keys, silent pubkey fallback |

## CRITICAL

### C1 — Silent weak keys in `generate_keypair`
**File**: `src/trust.cyr:364-379`
**Issue**: `/dev/urandom` fd open and read return values are unchecked.
If `file_open` returns `-1` (ENOENT, EMFILE) or `file_read` returns
fewer than 32 bytes, the `seed` buffer contains uninitialized stack
memory or partial entropy. Ed25519 keypair is then derived from a
predictable/weak seed — catastrophic key compromise, no error signal.

**Fix**: Return error from `generate_keypair` on fd < 0 or n != 32.
Same pattern in `tpm_random` (`src/tpm.cyr:140-156`).

### C2 — Silent zero public key in verification
**File**: `src/hex.cyr:35-46`, `src/verify.cyr:239`
**Issue**: `hex_decode` silently converts non-hex characters to `0`
via `_hex_nibble` fall-through. Combined with `verify.cyr:239`
hardcoding `hex_decode(pk, 64)` without validating `strlen(pk) == 64`
or that `pk` contains only hex chars, a corrupted or truncated
`public_key_hex` in the keyring decodes to a predictable all-zero or
partially-zero pubkey. An attacker who can tamper with keyring JSON
files could substitute a pubkey for which they hold the matching
secret, or exploit small-subgroup behavior at the all-zero point.

**Fix**: `hex_decode` rejects non-hex chars and odd-length input
(return `0` = failure sentinel). `sv_verify_artifact` validates
`strlen(pk) == 64` and decode success before calling `verify_data`.

## HIGH

### H3 — Ed25519 signature malleability (S ≥ L accepted)
**File**: `src/ed25519.cyr:633-692`
**Issue**: RFC 8032 §5.1.7 and §8.4 mandate rejecting signatures
where the scalar `S` is not in the canonical range `[0, L)`.
`ed25519_verify` loads `S` from `sig[32..64]` and passes it directly
to scalar multiplication without range-checking. An attacker can
produce a second valid signature `(R, S+L)` for the same
`(pk, msg, sig)` tuple. For Sigil (trust boundary), this enables
signature re-use attacks and violates the "single valid signature
per message" invariant consumers may assume.

**Fix**: Before using `_ver_S_scalar`, compare against `_ed_L`;
return 0 if `S >= L`.

### H4 — Path traversal + buffer overflow in `keyring_save`
**File**: `src/trust.cyr:268-316`
**Issue**: Path constructed as `keys_dir + "/" + kid + ".json"` into
a fixed 256-byte buffer. Two issues:
1. No bounds check: `dlen + 1 + kidlen + 5 + 1` may exceed 256,
   causing heap overflow.
2. No validation that `kid` is a safe filename: `kid = "../../etc/passwd"`
   escapes `keys_dir`, allowing arbitrary-path writes when the caller
   controls key IDs (e.g., imported from JSON).

**Fix**: Validate `kid` contains only `[A-Za-z0-9_-]`, reject if
`strlen(kid) > 64`. Compute required path length and bail if it
exceeds buffer.

### H5 — Fuzz harness crashes on clean exit (SIGSEGV)
**File**: `fuzz/fuzz_integrity.fcyr`, `fuzz/fuzz_revocation.fcyr`
**Issue**: Both fuzz binaries return exit 139 (SIGSEGV) after all
assertions pass. The `scripts/check.sh` harness masks this: it treats
any non-124 exit as OK due to `|| true` before `rc=$?`. The segfault
is post-`main` and indicates memory corruption during cleanup — a
latent memory-safety bug in the library under fuzz load.

**Fix**: Investigate crash (likely hashmap or freelist path in
`assert_summary` or final vec destructors). Fix `check.sh` to treat
SIGSEGV as a failure.

## MEDIUM

### M6 — HMAC stack buffers not zeroed
**File**: `src/hmac.cyr:14-58`
**Issue**: `kprime`, `ipad`, `opad` hold key-derived material
(K', K' ⊕ 0x36, K' ⊕ 0x5c) on stack. Never zeroed before function
return. Stack frames can be recovered via process memory reads.
**Fix**: `memset` all three to 0 at end of `hmac_sha256`.

### M7 — Ed25519 secret scalars not zeroized
**File**: `src/ed25519.cyr:502-522, 539-618`
**Issue**: Globals `_kp_scalar`, `_sign_r_scalar`, `_sign_a_scalar`
hold the private scalar `a` and nonce `r`. `_sign_az` is zeroed at
line 616 but the derived scalars persist across calls.
**Fix**: `memset` all three to 0 at end of `ed25519_sign` and
`ed25519_keypair`.

### M8 — `file_read` short-read not validated (other sites)
**File**: `src/tpm.cyr:140-156`, `src/trust.cyr:327-342`
**Issue**: Beyond C1, `tpm_random` and `hash_file` do not verify
`file_read` returned the requested count. `hash_file` tolerates
short reads (continues loop) which is correct; `tpm_random` does
not guarantee `count` bytes were filled.
**Fix**: `tpm_random` returns `n` (already does), but callers must
treat `n < count` as failure. Document or enforce.

### M9 — JSON log injection in persistence paths
**File**: `src/trust.cyr:294-313`, `src/verify.cyr:551-579`
**Issue**: `keyring_save` and `sv_save_trust_store` write
user-controlled strings (`key_id`, `pub_name`, `path`, `content_hash`)
directly into JSON without escaping. A `"` or newline in any field
corrupts the JSON, breaking re-load and potentially allowing forgery
of adjacent records when re-parsed by lenient consumers.
**Fix**: JSON-escape `"`, `\`, and control bytes < 0x20 before
writing.

### M10 — `hex_decode` accepts odd-length and invalid chars
**File**: `src/hex.cyr:35-46`
**Issue**: Beyond C2, odd-length input is silently truncated
(`hex_len / 2`). Partial success produces silent data loss.
**Fix**: Covered by C2's reject-invalid-hex fix.

## LOW

### L11 — `ireport_summary` fixed 128-byte buffer
**File**: `src/integrity.cyr:329-362`
**Issue**: Worst-case numeric width (4×19 digits + fixed text) is
~120 bytes. Tight headroom; a future format change could overflow.
**Fix**: Size buffer to 192 or compute needed length first.

### L12 — `_sv_key_authorized` memeq OOB read
**File**: `src/verify.cyr:127-148`
**Issue**: `memeq(path, prefix, strlen(prefix))` reads `plen2` bytes
from `path`. If `strlen(path) < plen2`, reads past path's allocation.
Cyrius C strings are null-terminated, so OOB read is bounded by the
next heap object, but still UB.
**Fix**: Early-return if `strlen(path) < plen2`.
