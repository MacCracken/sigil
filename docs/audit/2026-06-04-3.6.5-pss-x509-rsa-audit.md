## RSA-PSS

### Correctness
- **Cross-implementation KAT**: a pure-Python EMSA-PSS encoder (fixed
  salt) over the same RSA-2048 key produces signatures that sigil's
  `rsa_pss_verify_sha{256,384}` accept — independent confirmation of the
  MGF1 + EMSA-PSS-Verify structure. Sign is randomized (random salt), so
  it is covered by sign→verify roundtrips, not byte-KATs.
- **emBits/emLen**: `embits = modBits − 1`, `emlen = ceil(embits/8)`
  computed from the actual modulus bit length (`_rsa_bitlen_be`). For
  all multiple-of-8 moduli (2048/3072/4096) `emlen == k` and the top
  mask clears exactly 1 bit; the general `8·emlen − embits` mask handles
  the off-by-8 case too. EM is built/recovered right-aligned in the
  k-byte buffer, so `m = OS2IP(EM) < 2^embits < n` always holds (feeds
  `_rsa_raw_sign`'s precondition).
- **Salt-agnostic verify**: EMSA-PSS-Verify recovers the salt from the
  `0x01` separator after the zero PS, rejecting any non-zero octet
  before it (matches OpenSSL `SALTLEN_AUTO`). Sign uses salt = hLen (the
  TLS 1.3 `rsa_pss_rsae_*` convention).

### Buffer safety
- `_mgf_seedc[72]` ≥ hLen(≤48) + 4. `_pss_db[512]`/`_pss_dbmask[512]` ≥
  dbLen = emLen−hLen−1 (≤ 479). `_pss_em[512]` ≥ emLen (≤ 512), written
  only at `emstart = _pss_em + (nlen−emlen) .. +emlen−1`. `_pss_mprime
  [600]` ≥ 8 + hLen + slen worst case (8+48+463 = 519 for the largest
  recoverable salt at RSA-4096). All bounds verified for k ≤ 512.
- MGF1 honours a partial final block (`take = min(hLen, remaining)`),
  never over-writing `mask_out`.

### Timing / key material — INFO-1/2
PSS **verify** operates only on the signature + public key + public
message → no constant-time requirement (mirrors v1.5 verify). PSS
**sign** delegates the secret-exponent operation to `_rsa_raw_sign`
(unchanged constant-time Montgomery ladder + base blinding + CRT +
Bellcore verify-after-sign, audited 3.6.4). The EMSA-PSS encode touches
only the message digest + a random salt, introducing no secret-dependent
branch. The salt is per-signature random and becomes public (embedded in
the signature), so no zeroization is required. The PSS scratch is plain
single-threaded static (RSA is not on the parallel batch-verify path —
same posture as the rest of `src/rsa.cyr` / `src/bignum.cyr`).

## x509 RSA chain-link

### Parse path (attacker-controlled bytes)
- **RSA SPKI**: dispatched on the AlgorithmIdentifier OID
  (`_oid_rsa_encryption`). Extraction reuses the audited
  `rsa_pubkey_from_der` (3.6.3) over the SPKI span `buf[pos..spki_np)` —
  every read is a bounds-checked `der_walk`. The modulus/exponent land
  in a 544-byte side block (`n[512] || e[16] || n_len || e_len`); the
  `n_max=512`/`e_max=16` caps in `rsa_pubkey_from_der` bound the writes.
- **Signature algorithm**: `_xp_parse_sig_algid` accepts only
  ecdsa-with-SHA256, sha256WithRSA, sha384WithRSA; RSA requires the
  explicit PKCS#1 NULL parameters (RFC 4055 §2.1). Inner (TBS) and outer
  signature algorithms are required to **match** (RFC 5280 §4.1.1.2) —
  defeats algorithm substitution.
- **Signature value**: for RSA the BIT STRING content is the raw k-byte
  signature, stored as an absolute pointer into the DER buffer + length;
  `der_walk` guarantees `inner_sig_off + inner_sig_len ≤ der_len`.

### Verify dispatch
`_x509_verify_link` requires the issuer key-type to match the child's
signature algorithm: RSA-SHA256/384 → issuer must be `X509_CURVE_RSA`
with a non-NULL side block, fed to `rsa_pkcs1v15_verify_sha{256,384}`
(which re-checks `sig_len == n_len`). No type confusion: an RSA child
against a P-256/P-384 issuer (or vice versa) returns 0. A P-384 *issuer*
signing a link remains out of scope (documented) — only the leaf may be
P-384. Empirically: openssl-generated RSA-2048 CA verifies both SHA-256
and SHA-384 leaves; tamper, wrong-issuer-key, and DN-mismatch all reject.

### LOW-1 (audit floor)
The 544-byte RSA side block is allocated with bump `alloc` on every
`x509_parse` of an RSA cert — the same per-call bump-allocator-lifetime
shape as the existing **3.2.2 LOW-1** (`x509_parse` raw_sig). It carries
into the v3.7 unified `_into` API cleanup alongside the other seven
floor findings; not independently exploitable (single-shot parse, pool
reset between verification runs by the consumer).

## Disposition

3.6.5 ships clean — **0 CRITICAL/HIGH/MEDIUM**, **1 LOW** folded into the
existing v3.7 `_into` audit-floor cleanup. The RSA PSS + v1.5 surface is
now complete (sign + verify, PKCS#1 v1.5 + PSS, SHA-256/384), and x509
