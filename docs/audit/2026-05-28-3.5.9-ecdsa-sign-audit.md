## The nonce path (load-bearing)

`k` is derived by RFC 6979 §3.2 HMAC_DRBG keyed on the curve hash
(`hmac_sha256` for P-256, `hmac_sha384` for P-384), seeded from
`int2octets(d) ‖ bits2octets(H(m))`. Properties verified:

- **Determinism** — re-signing the same (key, message) yields the
  identical signature (tested: `ct_eq_bytes` over a second signature).
  No `getrandom`/RNG is consulted, so there is no entropy-failure or
  nonce-reuse class for this primitive.
- **Correctness** — the exact `(r, s)` for the RFC 6979 A.2.5
  (P-256) and A.2.6 (P-384) "sample"/"test" vectors match
  byte-for-byte. The vectors were independently reproduced with a
  stdlib-only Python RFC 6979 implementation before embedding, so the
  KATs are not self-referential.
- **Range** — the DRBG loop rejects `k == 0` and `k >= n` and
  regenerates (RFC 6979 step h), bounded at 256 tries.
- **Zeroization** — `k`, the DRBG state (`K`, `V`), the private
  scalar, and the `k^-1` / `(e + r·d)` intermediates live in a single
  per-function `secret var` scratch block, compiler-zeroized on every
  return path (including the early-exit error returns).

## Checklist findings

1. **Input validation** — `d_be` is the caller's 32/48-byte scalar
   (contract: in `[1, n-1]`, produced by `*_privkey_from_der`). The
   message is hashed internally (`sha256`/`sha384`), matching the
   verify entry points. `sign_der` bounds the output against
   `out_max` and returns -1 if exceeded. **Clean.**
2. **Buffer safety** — the sign scratch is a single `secret var
   sc[416]` (P-256) / `sc[624]` (P-384) carved into 13 fixed slots by
   constant offset; every slot is exactly one field width (32/48).
   The DER body buffer is `body[110]`, and the maximum content for
   two 48-byte INTEGERs with sign-pad is 2×(2+1+48)=102 < 110; the
   total is bounded against `out_max` before the SEQUENCE write.
   **Clean.**
3. **Constant-time** — the tag/secret comparisons here are the
   `u256_is_zero` / `u256_cmp` guards on r, s, k against public
   bounds; there is no comparison of one secret against another, and
   no early-exit branch keyed on the private scalar's *value*. The
   add-mod-n (`_ecs_addmod_n_*`) branches only on the public carry/
   compare, not on secret content. **Clean at this layer.** (See
   LOW below for the inherited point-arithmetic property.)
4. **add-mod-n correctness (carry)** — the lib `u256_addmod` drops
   the 2^256 carry from `u256_add`, which is *incorrect* for the
   P-256/P-384 scalar field where `n ≈ 2^256` and `a+b` can exceed
   `2^256`. This module uses a carry-correct `_ecs_addmod_n_p256` /
   `_p384` instead (honors the carry, single conditional subtract).
   Verified indirectly: a wrong `(e + r·d)` would not reproduce the
   RFC 6979 `s` values, which it does exactly. **Clean.**
5. **Key zeroization** — see the nonce-path section; single
   `secret var` block per sign fn, plus the DRBG's own `secret var`
   buffers. The raw signature output (`r‖s`) is public. **Clean.**
6. **Syscall review** — none added (no RNG syscall — the whole point
   of RFC 6979). **N/A.**
7. **Pointer validation** — all internal buffers are fixed-size
   stack slots; the only external pointers are `d_be`, `msg`,
   `sig_out`/`out` per the documented contract. **Clean.**
8. **No command injection / path traversal** — none. **N/A.**
9. **Known-CVE check (ECDSA signing)**:
   - *Nonce reuse / bias (PS3, many)* — eliminated by RFC 6979
     determinism; KAT-verified against the spec.
   - *Low-s malleability* — sigil emits `s` as computed (not
     low-s-normalized); standard ECDSA verify (incl. sigil's) accepts
     any `s ∈ [1, n-1]`. Not a vulnerability; noted for callers that
     require BIP-62-style low-s (none in the TLS path).
   - *Zero r / zero s* — both rejected (`u256_is_zero`) with a -1
     return rather than emitting a degenerate signature.
   - *Fault attacks* — out of scope (no redundant-recompute
     countermeasure); single-tenant threat model.
10. **Findings filed** — this document.

## LOW — inherited non-constant-time scalar mul / inversion

`pt_scalarmul` / `pt384_scalarmul` and the Fermat `fn_p256_inv` /
`fn_p384_inv` are the same routines the verify path uses. Their
timing is not strictly independent of the scalar (the inversion
exponent is the fixed public `n-2`, but the ladder/`fn_*_mul` chain
has data-dependent sub-steps). For *verify* the scalars are public,
so this was benign; for *sign* the nonce `k` and key `d` flow through
`pt_scalarmul(k·G)` and `fn_*_inv(k)`. On AGNOS's single-tenant
deployment this is not exploitable (no co-resident attacker), and it
is the same property already on the audit floor for the verify path.
The v3.6/3.7 work (caller-scratch + Solinas) is the natural place to
revisit constant-time hardening of the shared point code. **Tracked,
not a regression.**
