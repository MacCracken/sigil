## Correctness

- **Word-permutation table** — the nine terms `s1..s9` and their
  coefficients (`+1, +2, +2, +1, +1, −1, −1, −1, −1`) were checked
  term-by-term against the canonical NIST P-256 reduction (FIPS 186-4
  App. D.2 / GECC Alg. 2.29); all nine match.
- **Differential KAT** — `p256_reduce` (Solinas) equals
  `_p256_reduce_longdiv` byte-for-byte over 64 SHA-256-seeded random
  512-bit inputs + edge cases (`2^512−1`, high-half all-ones), and the
  existing `reduce(2^256)` / `reduce(p)` / `reduce(p+1)` KATs and the
  full RFC 6979 P-256 verify vectors all pass (`ecdsa_p256.tcyr` 57/57;
  full suite 51/51 incl. the SGX/TDX/SEV-SNP `*_verify_full` chains that
  exercise P-256 verify transitively).
- **Term reduction** — each `si < 2^256 < 2·p256`, so the single
  conditional subtract in `_p256_sol_red1` brings it into `[0, p)`; the
  combination then uses the audited `fp_p256_add`/`fp_p256_sub`
  (operands and result in `[0, p)`), so the final result is fully
  reduced — what `fp_p256_mul` / point arithmetic require.

## Memory safety

`_p256_wd(prod, idx)` reads `load64(prod + (idx/2)*8)` for `idx` 0..15 →
within the 64-byte product. `_p256_mk` writes exactly 32 bytes to the
32-byte `out` / `_p256_sol_t`. The `(wN << 32)` packing operates on
`_p256_wd` values masked to `0xFFFFFFFF` (no sign-extension). The
`fp_p256_add(out, out, T)` / `fp_p256_sub(out, out, T)` self-aliasing is
safe (`u256_add`/`u256_sub` read each limb before writing it). Scratch
is single-threaded (INFO-2).

## Constant-time posture (INFO-1)

`_p256_solinas_reduce` has data-dependent conditional subtracts
(`_p256_sol_red1` and the `fp_p256_add`/`fp_p256_sub` reductions). The
review specifically checked whether this **regresses** the side-channel
posture versus the replaced `_p256_long_div_reduce` — which itself did a
per-bit, value-dependent conditional subtract on the running remainder.
Conclusion: **posture-neutral.** Neither reduction is constant-time; the
swap introduces no new secret-dependent path. `ecdsa_p256_verify`
operates on public data (signature, public key, message). The
secret-scalar exposure on the `ecdsa_p256_sign` path is **pre-existing**
(the long division had it too) and mitigated by RFC 6979 deterministic
nonces + AGNOS's single-tenant deployment. A constant-time field stack
(and the fixed-base-comb cache-timing item) remain separately tracked in
the roadmap; they are out of scope for the reduction swap.

## Performance (claims backed by `benches/history.csv`)

Row `v3.7.0-p256-solinas` (cyrius 6.0.61, dev host): `ecdsa_p256_verify`
**147.488 ms (long-div) → 26.139 ms (Solinas) = 5.65×**. The ≤ 10 ms
roadmap target is **not** reached by the reduction alone — with the
reduction fast, the schoolbook `u256_mul_full` and the double-and-add
scalar multiplication dominate. That target is carried to the v3.7 **EC
scalar-mult speedup** item (fixed-base comb + wNAF); see the roadmap.
