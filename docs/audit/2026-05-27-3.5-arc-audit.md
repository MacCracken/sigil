## 3.5.0 — Poly1305 one-time MAC (`src/poly1305.cyr`)

26-bit-limb (`poly1305-donna`) reduction mod 2^130−5; every
intermediate product < 2^59, so no 128-bit path. Final freeze is
**constant-time** (sign-bit mask-select, no branch on the
key-derived accumulator). Validated against RFC 8439 §2.5.2 + A.3.

- **Constant-time**: ✓ no secret-dependent branch in the freeze;
  `poly1305_verify` uses `ct_eq_bytes`.
- **Zeroization**: `block` scratch + `computed` are `secret var`.
- **INFO-A — best-effort scalar zeroization.** `secret var` covers
  stack arrays; the key-derived scalar limbs are cleared by explicit
  `= 0` stores (same accepted stance as mldsa module globals). Key
  is caller-owned and one-time.
- **Contract**: ONE-TIME KEY — never authenticate two messages
  under one key (RFC 8439 §2.6); documented in the module header.

## 3.5.1 — ChaCha20 stream cipher (`src/chacha20.cyr`)

20-round ARX permutation in 32-bit-word arithmetic;
`chacha20_block` + `chacha20_xor`. Validated against RFC 8439
§2.3.2 keystream + §2.4.2 encryption vectors.

- **Constant-time**: ✓ by construction (fixed rounds, no
  data-dependent branch on key/plaintext).
- **Zeroization**: `st` (key), `ws`, `ks` are `secret var`.
- **Buffer safety**: 64-byte block / 128-byte state offsets all
  bounded; `chacha20_xor` handles `in_len == 0` and negative.
- **INFO-B — 256 GiB counter cap.** The 32-bit block counter wraps
  after 2^32 blocks; a single (key, nonce) must encrypt ≤ 256 GiB
  (RFC 8439 §2.4). Far above any sigil payload; documented as a
  hard caller obligation in the module header.

## 3.5.2 — ChaCha20-Poly1305 AEAD (`src/chacha20poly1305.cyr`)

RFC 8439 §2.8 composition: one-time Poly1305 key from the counter-0
ChaCha20 block, ciphertext from counter 1, tag over `aad || pad16
|| ct || pad16 || le64(aad_len) || le64(ct_len)`. Validated against
§2.8.2 (ciphertext + tag).

- **AEAD contract**: ✓ decrypt recomputes + compares the tag in
  constant time (`ct_eq_bytes`) and zeroes `pt_out`, returning
  `ERR_INTEGRITY_MISMATCH`, **before** deriving any plaintext.
- **INFO-C — nonce reuse is a caller obligation.** Reusing
  (key, nonce) repeats both keystream and one-time MAC key
  (catastrophic). Standard AEAD caveat; cannot be detected here.
- **INFO-D — mac_data `fl_alloc` unguarded.** No null-check before
  `memset`, matching the existing sigil convention (aes_gcm /
  ed25519 allocs likewise). Uses `fl_alloc`/`fl_free` (per-call
  scratch, freed before return) — **NOT** the bump allocator, so it
  does not extend the v3.7 bump-allocator audit floor.

## 3.5.3 — X25519 key agreement (`src/x25519.cyr`)

RFC 7748 Montgomery ladder reusing the `bigint_ext` Curve25519
field arithmetic (mod 2^255−19); constant-time masked `cswap`;
scalar clamp. Validated against §5.2 vectors 1-2 and §6.1
Diffie-Hellman (both sides agree).

- **Constant-time**: ✓ fixed 255 iterations, `cswap` masked — the
  field-op sequence is independent of secret scalar bits.
- **Zeroization**: clamped scalar + all ladder state share one
  `secret var W` block (zeroized on return).
- **INFO-E — all-zero shared secret not rejected.** RFC 7748 §6.1
  "MAY" check; left to the protocol layer (TLS 1.3 mandates it).
  Deliberate for a primitive library; documented as a caller
  obligation.
- **INFO-F — fp-reduction branches inherited.** `fp_add/sub` /
  `u512_mod_p` end with bounded data-dependent conditional `p`
  subtractions on field values — the same primitives Ed25519 has
  shipped since 3.0.x; the established accepted level for the
  single-tenant threat model. X25519 adds no new secret-dependent
  branch.
- The single-block `secret var` shape was forced by Cyrius's max-8
  `secret`/`defer`-blocks-per-function cap; cap raise filed upstream
  (`cyrius/docs/development/issues/2026-05-27-secret-defer-block-per-fn-cap.md`).

## Cross-cutting

- **INFO-G — serial-only static scratch (all four modules).**
  Working buffers are function-scope `var` arrays = Cyrius static
  globals (quirk #1); each call fully writes before reading, so
  serial use is correct. None is yet on a multi-threaded path; the
  v3.6 caller-provided-scratch cycle addresses concurrency
  structurally.

while ChaCha20-Poly1305 is software-friendly. A useful data point
for consumers choosing an AEAD on hosts without PCLMULQDQ.
