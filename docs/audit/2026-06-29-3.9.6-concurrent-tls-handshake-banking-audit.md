# 2026-06-29 — 3.9.6 concurrent-TLS-handshake banking audit

**Scope**: the concurrent-TLS-handshake crash
([`2026-06-28-concurrent-tls-handshake-global-scratch-race.md`](../development/issues/2026-06-28-concurrent-tls-handshake-global-scratch-race.md))
— make sigil's handshake + AEAD crypto safe under concurrent TLS
handshakes. Toolchain pinned to cyrius **6.3.5**.

## Summary

A TLS 1.3 server crashed (SIGSEGV / ECONNRESET) the moment two
handshakes ran concurrently: the handshake-critical crypto scratch was
process-global and shared across threads. Root cause is two-fold —
(1) the per-worker crypto banks (`src/crypto_scratch.cyr`) existed but
were only ever activated by the batch-verify path, so concurrent TLS
workers all collided on bank 0; and (2) the primitives the TLS 1.3 key
schedule actually drives (HKDF, HMAC, `ed25519_sign`, the one-shot
SHA-2 hashes, and both AEAD suites) were either unbanked or used the
non-thread-safe `fl_alloc` (quirk #7). Fixed by making `cbank()`
auto-assign a per-thread lane (ADR 0007) and banking every affected
primitive (ADR 0004 per-lane discipline).

## Findings

| # | Severity | Site | Finding | Fix |
|---|---|---|---|---|
| 1 | **CRITICAL** | `src/crypto_scratch.cyr` `cbank()` | Lane index only set by `crypto_bank_set` (batch path); all other concurrent threads read bank 0 → cross-thread scratch race → SIGSEGV. | Auto-assign next lane from an atomic counter on first `cbank()`; `SIGIL_CRYPTO_BANKS` 8→64; bank 0 reserved for main. |
| 2 | **CRITICAL** | `src/hkdf.cyr`, `src/hkdf_sha384.cyr` | HKDF-Expand held round state in process-global pointers + an `fl_alloc` concat scratch (the issue's PRIMARY observed crash, in every TLS 1.3 key-schedule step). | Rewritten allocation-free: per-round HMAC streamed over `(T(i-1), info, counter)` segments via a banked `_hkdf_hmac_seg3`; banked per-lane `t_prev`/`t_current`; globals deleted. |
| 3 | **CRITICAL** | `src/hmac.cyr`, `src/hmac_sha384.cyr` | `secret var` K′/ipad/opad/inner_hash are function-scope statics (quirk #1); streaming ctx `fl_alloc`'d. Raced on every key-schedule HMAC. | Banked per-lane (plain `var` + per-lane wipe); streaming ctx → banked `_into` (no `fl_alloc`). |
| 4 | **HIGH** | `src/ed25519.cyr` `ed25519_sign` / `sc_muladd` | Nonce/scalar scratch (`az`/`r_scalar`/`a_scalar`/…, `prod`) unbanked statics on the CertificateVerify path — crash **and** a nonce-corruption hazard. | Banked per-lane + per-lane wipe; streaming SHA-512 → banked `_into`. `prod` (secret product) now wiped (was not). |
| 5 | **HIGH** | one-shot `sha256`/`sha512`/`sha384` | Each `fl_alloc`'d its ctx; concurrent transcript / HKDF / HMAC-long-key hashes raced the freelist. | Converted to banked `_into` ctx — no allocator. Fixes all callers at the source. |
| 6 | **HIGH** | `src/sha384.cyr` `sha384_finalize` | `var tmp[64]` (truncation scratch) unbanked static — raced concurrent SHA-384 transcript/HKDF-SHA384. Caught live by the new race-detector (SHA-384 path failed 4/15 before the fix). | Banked per-lane. |
| 7 | **HIGH** | `src/aes_gcm.cyr` (GHASH/CTR/`_gcm_compute_j0`/encrypt+decrypt impls) | ~20 GCM-mode + software-AES scratch buffers (incl. `secret var` h/y/s/expected_tag) are statics per quirk #1 — raced on every AES-GCM op (encrypted handshake flight + records). The issue's "do NOT chase AES-GCM" note **refuted**: it misread indented arrays as stack locals. | Banked per-lane + per-lane wipe across all sites. |
| 8 | **HIGH** | `src/aes_gcm.cyr` encrypt+decrypt impls | `round_keys = fl_alloc(rk_size)` — non-thread-safe allocator on every AEAD op. Caught live by the race-detector (AES-GCM failed 3/25 after the array banking, before this). | Banked static key schedule (240 B/lane keeps 16-byte AES-NI alignment, quirk #6); `fl_free` dropped. |
| 9 | **MEDIUM** | `src/chacha20poly1305.cyr`, `src/poly1305.cyr` | `secret var` one-time key / block / tag scratch are statics — raced concurrent ChaCha20-Poly1305 (the issue confirmed **both** AEAD suites crash). | Banked per-lane + per-lane wipe. |
| 10 | **MEDIUM** | `src/aes_ni.cyr`, `src/sha_ni.cyr` | One-time CPUID self-test writes module-global `_aes_ni_st_*`/`_sha_ni_st_ctx`; two threads first-probing concurrently could corrupt it and spuriously downgrade NI. | CAS guard so exactly one thread runs the self-test; others spin for the published cache (x86-only paths, x86-TSO visibility). |

## Deferred (tracked → 3.9.7, NOT closed silently)

- **LOW/MEDIUM** — `src/chacha20poly1305.cyr` `_cp_tag` builds Poly1305
  mac_data in an `fl_alloc` buffer (the **only** remaining concurrent-
  path `fl_alloc`). Needs a streaming Poly1305 (init/update/finalize).
  AES-GCM — the issue's repro suite — is fully fixed; the ChaCha suite's
  handshake key-schedule/sign/transcript is safe, only its AEAD tag
  buffer remains.
- **MEDIUM (latent)** — `src/ecdsa_p256.cyr` / `ecdsa_p384.cyr`: ~100
  pointer-global scratch buffers/curve. Not reproduced (probe cert is
  Ed25519; `tls_native` rejects RSA), but an ECDSA-cert server under
  concurrency would hit the same class.
- **LOW (off-path)** — `src/bignum.cyr` (RSA modexp/Montgomery) and
  `src/tls12_prf.cyr` (TLS 1.2 PRF) carry unbanked statics; neither is on
  the TLS 1.3 server path (RSA rejected; 1.3 doesn't use the 1.2 PRF).

## Verification

- **Constant-time / zeroization**: every banked secret buffer keeps its
  zeroization via a per-lane `memset` (ADR 0004); the `secret var`
  compiler guarantee is forfeited by design (quirk #9). `ed25519_sign`'s
  `prod` and the GCM key schedule are now wiped where the pre-3.9.6 code
  was not — a net hygiene gain. No new branch on secret data; tag
  compares unchanged (`ct_eq_bytes`/`ct_eq_bytes_lens`).
- **Alignment**: banked AES-NI round keys use a 240-B lane stride
  (16-divisible) so every lane stays 16-aligned (quirk #6) — the
  `pxor/aesenc` memory-operand requirement holds.
- **Race detector**: new `tests/tcyr/concurrent_tls_handshake.tcyr` — 16
  auto-banked worker threads (no `crypto_bank_set`, exceeding the old
  8-bank cap) recompute sign + HKDF-SHA256/384 + SHA-384 transcript +
  AES-256-GCM + ChaCha20-Poly1305 vs. a serial bank-0 reference. Caught
  findings #6 and #8 as intermittent mismatches; **stable 30/30** after
  the fixes.
- **Correctness regression**: full `.tcyr` suite 1290 passed / 0 failed
  (RFC 5869 HKDF, RFC 8439 ChaCha20-Poly1305/Poly1305, FIPS AES-GCM/SHA
  vectors all green through the rewritten code paths). 9 pre-existing
  compile failures (pem/x509*/tdx/`*_verify_full`) are unrelated
  include-chain drift, failing identically on 6.2.48 and 6.3.5.
