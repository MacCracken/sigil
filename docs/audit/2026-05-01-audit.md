## New findings — 3.0 cycle

### MEDIUM

#### M1 — Thread-safety: crypto modules use shared module-global scratch

**Files**: `src/sha_ni.cyr` (`_sha_ni_buf`, `_sha_ni_K`,
`_sha_ni_cache`), `src/aes_ni.cyr` (`_aes_ni_cache`,
`_aes_ni_probe_out`), `src/aes_gcm.cyr` (`_aes_state`,
`_aes_sbox`, `_aes_rcon`), `src/hkdf.cyr` (`_hkdf_scratch`,
`_hkdf_prk`, `_hkdf_info`, `_hkdf_out`, `_hkdf_produced`,
`_hkdf_counter`), `src/ed25519.cyr` (every `_ver_*`, `_sign_*`,
`_kp_*`, `_ed_*` global), `src/mldsa.cyr` (`_mldsa_ws_*`
workspace), `src/verify.cyr` (`_sigil_batch_mutex`).

**Issue**: Every crypto module promotes its hot-path state to
module globals because Cyrius's single-pass compiler can clobber
locals across deeply nested call chains (CLAUDE.md quirk #1). This
is correct for single-threaded use but means **two threads calling
`hmac_sha256` / `ed25519_verify` / `aes_gcm_decrypt` / `hkdf_expand`
/ `mldsa65_verify` simultaneously would race on the same global
buffers and produce incorrect results (or crash)**.

**Mitigation in 3.0** (current state, not a fix): the parallel
batch verify path serialises every `sv_verify_artifact` call
through `_sigil_batch_mutex` (`src/verify.cyr:877-879`). Workers
acquire the mutex before entering the verify path and release on
return. Because the mutex covers the **full** call chain — from
`hash_file` through `ed25519_verify` and into the SHA-NI / AES-NI
compress paths — no two workers ever touch the crypto globals
concurrently, despite both running on different OS threads.

**Real risk surface**: when the 3.1 "Option 1" alloc-free rewrite
lifts the mutex (so `ed25519_verify` runs concurrently across
workers), the global-scratch shape will need to either:
1. Move every `_ver_*` / `_sign_*` / `_mldsa_ws_*` to per-thread
   scratch passed in by the caller, OR
2. Wrap each individual crypto primitive in its own internal
   mutex (defeating most of the parallelism win), OR
3. Provide a `tls_var` analogue from cyrius once the language
   supports thread-local storage.

**This is the load-bearing constraint behind 3.0 shipping at
"correctness-only, 0.96×–1.04× throughput" rather than the
fan-out speedup.** The mutex-wrap is an *intentional* correctness
choice; the throughput rewrite is intentionally deferred to 3.1.

**Severity rationale**: MEDIUM (not HIGH) because the mutex-wrap
mitigation is sound and correctness is preserved. Promoting to
HIGH would be appropriate only if 3.1's rewrite ships without
addressing this systematically — which the
`docs/development/roadmap.md` § Road to v3.1 entry already calls
out as a hard requirement.

**Action**: documented; carried forward to 3.1 audit with the
"Option 1 rewrite" item.

### LOW

#### L1 — Public verify entry points accept NULL without explicit check

**Files**: `src/ed25519.cyr:723` (`ed25519_verify`),
`src/ed25519.cyr:646` (`ed25519_sign`), `src/mldsa.cyr:374`
(`mldsa65_verify`), `src/mldsa.cyr:256` (`mldsa65_sign`).

**Issue**: `ed25519_verify(pk, msg, msg_len, sig)` and
`mldsa65_verify(sig, msg, msglen, pk)` immediately dereference
their pointer parameters without null-checking. Calling with
`pk == 0` or `sig == 0` triggers a segfault rather than returning
a clean failure.

All sigil-internal callers validate before reaching these entry
points (`sv_verify_artifact` checks `pk != 0`, `strlen(pk) == 64`,
and the `hex_decode` result before calling `verify_data`, which
in turn calls `ed25519_verify`). The exposure surface is therefore
limited to:
- External Cyrius consumers calling the public API directly.
- A future C/Rust FFI binding.
- Hybrid verify (`sigil_verify_hybrid`) where the caller passes
  one of `ed_pk` / `mldsa_pk` as 0 with the wrong `required_alg`.

**Severity rationale**: LOW (defense-in-depth). No crash path is
reachable in current sigil consumers; the audit looked at every
call site of these functions in `src/`.

**Fix recommendation**: add `if (pk == 0 || sig == 0) return 0;`
to the verify entry points. Cost: 2 instructions per call,
unmeasurable on the ~6.4 ms `ed25519_verify` hot path. Apply
during the 3.1 alloc-free rewrite touch-pass.

#### L2 — `alloc(count * 8)` in batch dispatcher trusts caller `count`

**File**: `src/verify.cyr:896-898`.

**Issue**: `sv_verify_batch` does `var slots = alloc(count * 8);`
plus `alloc(workers * 48)` and `alloc(workers * 8)` without
bounds-checking `count`. A negative `count` triggers a wraparound
to a small / negative size, returning a small block, which the
worker loop then writes past — silent memory corruption. A `count`
near `i64::MAX / 8` would overflow to a small allocation
similarly.

`workers` is the constant 4 (`_SIGIL_BATCH_WORKERS`), so its
multiplications are safe. `count` is the only attacker-influenceable
input here, **and only in the sense of "an internal caller built
the batch list incorrectly"** — `sv_verify_batch` is sigil-internal,
no current consumer passes attacker-derived counts.

**Severity rationale**: LOW. Not exploitable today; defense in
depth.

**Fix recommendation**: add `if (count <= 0) { return vec_new(); }`
at function entry. Already implicit in the existing
`while (i < count)` loop for the serial path, but the parallel
path's allocs precede the loop. Also adds a documented contract.

### INFO (defense-in-depth notes, not bugs)

#### I1 — One-time CPUID probe scratch leaks 8 bytes

**Files**: `src/aes_ni.cyr:35` (`_aes_ni_probe_out = alloc(8)`),
`src/sha_ni.cyr:40` (`_sha_ni_probe_out = alloc(8)`).

**Issue**: First call to `aes_ni_available()` or `sha_ni_available()`
allocates an 8-byte cache slot for the probe's output pointer and
never frees it. Per process: 16 bytes lifetime-leaked.

**Severity rationale**: INFO. Functionally a "lazy module-global"
allocation; not a memory leak in the bug sense.

**Fix recommendation**: replace with `var _aes_ni_probe_out[8];`
file-scope arrays now that Cyrius 5.5.21+ guarantees 16-byte
alignment for `var x[N]` with `N > 8`. Trivial; do during the
next touch-pass on those modules.

#### I2 — Convention mismatch in hybrid verify return values

**File**: `src/mldsa.cyr:456` (`sigil_verify_hybrid`).

**Issue**: `ed25519_verify` returns `1` on success, `0` on fail.
`mldsa65_verify` returns `0` on success, `1` on fail (matching the
pq-crystals C reference). `sigil_verify_hybrid` correctly bridges
both conventions internally and returns `1` for success. External
callers reaching for either primitive directly need to know which
convention they're hitting.

**Severity rationale**: INFO. Correctly handled in `mldsa.cyr:464-470`;
no security impact. Footgun risk for future direct callers.

**Fix recommendation**: align the conventions in the 3.1 verify-path
rewrite so all primitives return the same boolean. Document in
module headers until then.

#### I3 — Pre-existing `sys_stat` undefined-function warning on x86_64

**Files**: `lib/agnosys.cyr:9958` (vendored agnosys 1.0.4 bundle),
`src/fuse.cyr:267` upstream in
`https://github.com/MacCracken/agnosys`.

**Issue**: Default x86_64 sigil builds emit
`warning: undefined function 'sys_stat'` /
`error: undefined function 'sys_stat' (will crash at runtime)`
during link. agnosys 1.0.4's `fuse_validate_mountpoint` calls
`sys_stat` which exists in cyrius 5.7.48's stdlib only on aarch64
(`lib/syscalls_aarch64_linux.cyr:346`). Cyrius's DCE marks the
function unreachable for sigil (no FUSE call site), so no runtime
impact in practice.

**Severity rationale**: INFO. Not a sigil-side issue; filed
upstream at
`agnosys/docs/development/issues/2026-05-01-sys-stat-x86-portability.md`
on 2026-05-01. Recommended fix paths (Option A: agnosys-side shim
matching yukti's pattern; Option B: cyrius stdlib parity) documented
there.

**Action**: tracked upstream; no sigil-side change required.

## Audit method coverage

Each step of the CLAUDE.md "Security Hardening" checklist:

1. **Input validation** — covered: `hex_decode`, `keyring_save`'s
   `_is_safe_key_id`, `aes_gcm_decrypt`'s `aad_len < 0` /
   `ct_len < 0` guards, `hkdf_extract` / `hkdf_expand`'s
   `salt_len < 0` / `info_len < 0` / `out_len > 8160` guards. New
   finding L1 captures the verify entry-point gap; new finding L2
   captures the batch dispatcher gap.
2. **Buffer safety** — covered: every `var x[N]` and `alloc(N)`
   walked through; arithmetic sites surveyed (see L2). 16-byte
   alignment for AES-NI / SHA-NI scratch verified (`_sha_ni_align16`,
   `var rk[240]` since 5.5.21).
3. **Constant-time audit** — covered: `ct_eq` used at every
   signature/MAC compare (`ed25519_verify:774`,
   `aes_gcm_decrypt:652`, `mldsa65_verify:426-432` via
   bitwise-OR accumulation). No early-exit on secret data found
   in any compare.
4. **Key material zeroization** — covered: `secret var` adopted
   for HMAC ipad/opad/kprime, HKDF prk/t_prev/t_current/seed.
   Manual `memset` zeroization for ed25519 / aes_gcm scratch
   verified (`src/aes_gcm.cyr:570-578, 678-687`,
   `src/ed25519.cyr:626-627, 703-706`). Roadmap items the ~20
   remaining manual-`memset` sites for in-place `secret var`
   adoption.
5. **Syscall review** — covered: every `syscall(...)` and
   `sys_*(...)` call site inspected. Pattern scan confirmed no
   `sys_system`, no `exec_cmd`, no `popen`. agnosys 1.0.4's
   `sys_open` migration (the work merged in 2.9.5) reviewed —
   correctly uses `sys_open(path, flags, mode)` form.
6. **Pointer validation** — covered: see L1.
7. **No command injection** — covered: zero call sites of
   `sys_system` / `exec_cmd` / `popen` across `src/`.
8. **No path traversal** — covered: `_is_safe_key_id` enforces
   `[A-Za-z0-9_-]` for key-id-derived filenames in `keyring_save`.
   No `../` escape pattern in any string-builder. No hardcoded
   `/etc/`, `/proc/`, or `/tmp/` write paths in `src/`.
9. **Known CVE check** — covered:
   - **Ed25519** — RFC 8032 §5.1.7 / §8.4 `S < L` enforcement
     (H3 fix) closes the malleability class. Constant-time
     `ge_cmov` via stdlib `ct_select`. No small-subgroup checks
     omitted (the `ge_from_bytes` decode path validates against
     curve equation; legacy weak-subgroup attacks not applicable
     to the `[S]B − [h]A == R` form).
   - **SHA-256 / SHA-512** — FIPS 180-4 reference; SHA-NI compress
     cross-tested against software path on every input the test
     suite exercises (CHANGELOG 2.9.3).
   - **HMAC-SHA256** — RFC 2104 ipad/opad construction; not the
     "simplified `H(K || m)`" anti-pattern.
   - **AES-256-GCM** — FIPS 197 + NIST SP 800-38D; tag computed
     before plaintext touch; pt zeroed on tag fail; ct_eq tag
     compare; nonce-reuse detection is the **caller's** responsibility
     per the API contract (sigil offers no internal counter; consumers
     must supply unique 96-bit IVs per `(key, message)`).
   - **ML-DSA-65** — NIST FIPS 204; pq-crystals reference
     parity; ctilde compare uses bitwise-OR accumulation.
10. **File findings** — this document.
