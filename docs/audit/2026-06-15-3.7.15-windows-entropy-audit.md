## Load-bearing confirmations (the security positives)

1. **`_sigil_random_fill` rejects both failure shapes.** `random_bytes` (6.2.12
   `lib/random.cyr`) returns `len` on success, a negative `-errno` on permanent
   error, and a partial `off` (`0..<len`) on a zero-return mid-loop. The single
   `random_bytes(out, n) != n` test catches all three (negative `!= n`; any partial
   `!= n`) and maps them to `-1`. No short / partial / stale fill can reach key
   derivation. The `n == 0` early-return is benign (no key/seed/nonce site passes 0).

2. **Every converted call site treats a non-zero return as a hard abort.**
   `trust.cyr` `generate_keypair` (`!= 0 → return 0`), `ed25519.cyr`
   `ed25519_generate_keypair` (`!= 0 → memset sk_out/pk_out then return 0`),
   `mldsa.cyr` `mldsa65_keypair` (`!= 0 → return 0`), `rsa.cyr` `_rsa_gen_blind`
   (`!= 0 → return -1`). `_rsa_pss_rand` delegates to `_sigil_random_fill` (identical
   `0 / -1` ABI, `n == 0 → 0` guard preserved); both RSA call sites abort on `< 0`.

3. **Secret zeroization intact.** `secret var seed[32]` (Cyrius 5.3.5+ scope-exit
   zeroization) with a defensive init `memset` remains on the ed25519 / mldsa / trust
   keygen paths; the seed is zeroized on every return, including the entropy-failure
   path. No new secret-bearing locals were introduced (the per-call `fd` / `got`
   loop locals are gone).

4. **`_rsa_gen_blind` retry loop still bounded and never proceeds un-filled.** The
   `tries >= 16 → -1` cap is unchanged; `_rsa_blind_rand` is re-filled every iteration
   *before* use and the function returns `-1` immediately on any fill failure. The
   `nlen <= 512` precondition (enforced at all three sign entry points) keeps the fill
   inside the 512-byte `_rsa_blind_rand` buffer.

5. **Behavioral equivalence on Linux is exact.** Every converted site preserves its
   original success/failure ABI (key_id pointer / 0; 1 / 0; -1 / 0). Removing the
   `file_open` / `file_read` / `file_close` loop changed no control flow, retry count,
   byte count, or fill semantics — `random_bytes` loops `sys_getrandom` until `off == len`.
   Sizes correct (`seed[32]` ↔ 32, `MLDSA_SEEDBYTES == 32`, `nlen <= 512`); pointer
   arithmetic is the buffer base in every case (offset loop now internal). Smoke build
   clean; ed25519 / rsa / mldsa suites green end-to-end (keygen, PSS blind, PSS salt).

6. **Build-artifact sync verified.** `src/random.cyr` appears in all three artifacts
   (`cyrius.cyml [lib].modules`, `scripts/regen-dist.sh` MODULES, `src/lib.cyr`) in the
   same relative order (after `error`, before `crypto_scratch`). `dist/sigil.cyr` has
   **zero** `file_open("/dev/urandom")` (only historical-rationale comments) and is
   self-contained (no `src/` includes). All 18 keygen-capable `.tcyr` files plus the
   new `random.tcyr` carry the `lib/random.cyr` + `src/random.cyr` includes (zero
   `undefined function '_sigil_random_fill'` warnings).

7. **Consumer-integrity loop closed for the new `random_bytes` dependency.** README
   documents `lib/random.cyr` as a required opt-in include in all three places (the
   opt-in bullet list, the Usage include-order block, the "why required" section;
   the SIGILL-warning symbol list now includes `'random_bytes'`). Version agreement:
   `VERSION` = 3.7.15, `cyrius.cyml` pin = 6.2.12, dist header = 3.7.15, CHANGELOG top
   = `[3.7.15]`. `cyrius doc --check dist/sigil.cyr` → 0 undocumented (`_sigil_random_fill`
   carries its doc comment).

## Open items (surfaced, not buried)

- **`src/tpm.cyr` `tpm_random` — RESOLVED (converted in-cycle).** It is AGNOS-only
  (TPM RNG primary; one of the four agnosys-wrapping modules excluded from
  `dist/sigil.cyr`), so it was outside the issue's Windows scope and already failed
  *closed* on Windows. Robert chose to convert it for consistency: it now routes
  through `_sigil_random_fill` (`if (_sigil_random_fill(buf, count) != 0) { return 0; } return count;`),
  ABI preserved, making the "single entropy boundary" claim literally true and the
  module Windows-functional. Smoke build clean; `agnosys.tcyr` 26/26 green.

- **Windows (`cass`) acceptance pending.** The issue's acceptance requires a sigil
  keypair-gen + a `tls_native` client nonce on real Windows producing unique values.
  This cannot run on the Linux dev host; the issue stays **open** until verified on
  `cass`. The `tls_native` re-fold/re-verify half is cyrius-owned (downstream of the
  vendored `lib/sigil.cyr` fold).
