## Summary

| Severity | Count |
|---|---|
| CRITICAL | 0 |
| HIGH | 0 |
| MEDIUM | 0 |
| LOW | 2 |
| INFO | 2 |

The ECDSA verify surface is **verify-only**: no secret material is
handled, no syscalls are issued, no filesystem or network access
occurs. The threat surface is parsing-and-arithmetic, fully bounded
by the 256-bit field. The two LOW findings track known limitations
that the consumers of this surface (SGX/SEV/TDX backends in the
3.2.3–3.2.5 bites) will need to factor into their pre-validation.

## Step 1 — Input validation

`ecdsa_p256_verify(pk, msg, msg_len, sig)`:
- `r`, `s` decoded as 256-bit big-endian, then range-checked:
  `is_zero(r)`, `is_zero(s)`, `r ≥ n`, `s ≥ n` all reject
  (`src/ecdsa_p256.cyr:709-714`). Matches SEC 1 §4.1.4 step 1.
- `pk` decoded as `Qx || Qy`. A partial check rejects the
  all-zero point (`src/ecdsa_p256.cyr:720-722`) — see LOW-1.
- `msg_len` is the caller's responsibility; passed through to
  `sha256` which already bounds-checks.

`ecdsa_p256_verify_der(pk, msg, msg_len, sig_der, sig_der_len)`:
- Outer length check: `sig_der_len < 8` rejects (`:794`).
- Sequence tag verified: `load8(sig_der) != 0x30` rejects (`:795`).
- Sequence length: short-form only, `seq_len >= 0x80` rejects (`:797`).
- Body bounds: `2 + seq_len > sig_der_len` rejects (`:798`).
- Each INTEGER decoded via `_ecdsa_der_int` which:
  - Verifies tag `0x02` (`:773`).
  - Caps raw length at 33 bytes — P-256 max (`:774`).
  - Bounds-checks `off + 2 + raw_len > src_len` (`:775`).
  - Validates the optional 33-byte leading-`0x00` is actually zero
    (`:781`).
  - Pads the 32-byte destination with zeros before copy (`:786`).

## Step 2 — Buffer safety

All buffers are fixed-size, declared via `alloc(32)`, `alloc(64)`,
or `alloc(96)` (points). No dynamic sizing. All writes verified
in-bounds:

- Field arithmetic scratch (`_p256_prod` 64 B, `_p256_tmp{1,2}` 32 B,
  `_p256_R` 32 B): only `load64`/`store64` at fixed offsets 0/8/16/24
  for 32 B, 0..56 for 64 B. No out-of-bounds writes possible.
- Point scratch (`_ptd_*`, `_pta_*`, `_ptm_*`): each 32 B for field
  values or 96 B for points; same fixed-offset access pattern.
- DER decoder's `dst` is the caller's 32-byte buffer; `memset(dst, 0, 32)`
  precedes the copy; copy loop bounded by `int_len ≤ 32` (post-strip).

## Step 3 — Constant-time audit

**Threat model**: ECDSA verify operates exclusively on public inputs
(`pk`, `msg`, `sig`, derived intermediates `e`, `u1`, `u2`, `R`,
`v`). Timing leakage on these values is **not** a security defect —
verify outputs are public-decision per spec.

What sigil 3.2.1 does anyway:

- **Scalar multiplication** uses a Montgomery ladder
  (`pt_scalarmul`, `:651-672`) with `pt_cswap` (`:413-424`), which
  is branchless on the point payload. Defense-in-depth: any future
  sign / DH path through `pt_scalarmul` inherits constant-time
  behaviour for free (CLAUDE.md "no timing side-channels"
  invariant maintained for the primitive).
- **`pt_add`** (`:548-628`) has explicit early-return branches on
  P=O, Q=O, P=Q, P=-Q. These short-circuits are not constant-time,
  but the inputs are public, so this is acceptable. **INFO-1**
  tracks the future-sign caveat.
- **`pt_double`** (`:476-535`) has no input-dependent branches —
  the early-out on infinity (`:480`) tests Z=0 which is structural,
  not secret.
- **Final r comparison** (`:737`) uses `u256_eq`, which has
  early-exit branches. `r` and `v` are both public. The
  per-byte `ct_eq_bytes_lens` pattern from `ed25519_verify`
  is not adopted here — see INFO-1.

**Field arithmetic and reduction** (`p256_reduce`, `fp_p256_mul`,
`fp_p256_inv`, etc.) all branch on intermediate field values —
again public for verify. Same INFO-1 applies.

All input pointers (`pk`, `msg`, `sig`, `sig_der`) are caller-
provided. The library API contract documents the expected widths

— same convention as `ed25519_verify`. Caller responsibility, per

## Step 7 — No command injection

No `sys_system` / `exec_*` calls. Pure arithmetic.

## Step 8 — No path traversal

No filesystem access.

## Step 9 — Known CVE check

- **CVE-2022-21449 (Java ECDSA: r=0, s=0 accepted)**: r and s are
  both rejected on `is_zero` (`:710-711`). ✅
- **CVE-2020-0601 (Windows curve parameter spoofing)**: sigil does
  not parse curve parameters from input — they are hardcoded as
  `_p256_p`, `_p256_n`, `_p256_b`, `_p256_gx`, `_p256_gy`
  (`:51-65`). ✅
- **Invalid-curve attack** (Bos, Halderman et al., 2014): a
  malicious peer supplies a public key on a different curve
  (sharing the same field but with different `b`). Verify never
  rechecks `gy^2 ≡ gx^3 - 3*gx + b` for the provided `Q`. The
  attack lets a forged signature pass if the attacker can choose
  `Q` freely. **LOW-1** tracks this for the TEE backends.
- **Lattice-attack on biased nonces**: sign-only concern; N/A.
- **Subgroup confinement attack**: P-256 has prime order n, so all
  non-infinity points are generators of the full group; no
  small-subgroup attacks apply. ✅
- **Out-of-canonical signature malleability**: SEC 1 / FIPS 186-5
  do not mandate low-s for ECDSA (unlike BIP-62 for secp256k1).
  Sigil follows the spec — both `s` and `n - s` are accepted as
  valid. Documented as a non-issue for the AGNOS threat model.

## Step 10 — Findings

### LOW-1 — Public-key Q not validated to be on the curve

**Where**: `ecdsa_p256_verify` (`src/ecdsa_p256.cyr:716-723`).

**Issue**: After decoding `Qx, Qy` from `pk`, sigil only checks
that the point is not the all-zero "infinity" pair. It does **not**
verify `Qy^2 ≡ Qx^3 - 3*Qx + b (mod p)`. A caller supplying a
public key on a curve sharing the P-256 field but with different
`b` could potentially fool verify into accepting a forged
signature (invalid-curve attack, Bos et al. 2014).

**Threat model — 3.2.1**: Today sigil's only ECDSA consumer is
internal test code. The keyring is already trusted boundary so the
defect is dormant.

**Threat model — 3.2.3–3.2.5**: SGX/SEV-SNP/TDX attestation
verifiers consume ECDSA against `pk`s that arrive from an attacker-
controllable channel (the untrusted host runtime). The TEE bites
**must** validate Q-on-curve before calling `ecdsa_p256_verify`,
or sigil must add the check internally before those bites land.

**Recommended fix (deferred to 3.2.3)**: add a `pk_validate` helper
shared with the X.509 walker (3.2.2 will need a similar shape for
parsing certificate subject-public-key-info). Lift the curve
equation check out of the test file's "base point on the curve"
group and expose it as `pt_is_on_curve(x, y)`. Verify wraps that
behind a `pk` decode in 3.2.3.

**Severity**: LOW — currently dormant; documented in code; clean
follow-up gated by the TEE arc landing.

### LOW-2 — DER decoder tolerates non-strict length encoding

**Where**: `_ecdsa_der_int` (`src/ecdsa_p256.cyr:769-792`).

**Issue**: Strict DER requires the leading `0x00` sign byte only
when the high bit of the next byte is set. Sigil accepts a 33-byte
INTEGER with leading `0x00` regardless of the MSB. This means two
distinct DER blobs can encode the same signature, which technically
violates DER's "unique encoding" property.

**Threat model**: DER non-uniqueness is exploitable in scenarios
where signature equality is meaningful (deduplication, anti-replay).
In sigil's current model, the raw `(r, s)` after decode are what
matter; the DER blob itself is not retained or compared.

**Recommended fix (deferred)**: tighten the decoder when a consumer
surfaces a need. Annotate the function with the laxity for now.

**Severity**: LOW — defense-in-depth, no exploitable consequence
in sigil's surface.

### INFO-1 — Future-sign timing-side-channel caveat

**Where**: `pt_add`, `u256_eq` (final r compare), field
arithmetic.

**Note**: 3.2.1 ships verify-only. If a future TEE bite (e.g. SGX
seal in 3.2.6, or a SEV/TDX response signing path that hypothetically
lands later) introduces an ECDSA sign primitive against the same
`fp_p256_*` / `pt_*` surface, every branch on intermediate field
values needs a constant-time audit. The Montgomery ladder
substrate is already constant-time; the rest of the surface is
not.

**Action**: add a paragraph to the 3.2.6 scope doc when that bite
is opened.

---

### INFO-2 — Field reduction is bit-by-bit, ~136 ms per verify

**Where**: `_p256_long_div_reduce` (`src/ecdsa_p256.cyr:172-191`).

**Note**: The 3.2.1 reduction uses textbook long-division over
256 bits per `fp_p256_mul` call (~2 K reductions per verify). The
benchmark row `v3.2.1,ecdsa_p256_verify,136.283,ms` in
`benches/history.csv` is the baseline. Solinas word-level
reduction can drop this 20–50×. Tracked in the 3.2.x roadmap's
"bench tuning" follow-up; not gating any TEE bite ship.

**Action**: open as a roadmap item when the TEE arc closes. Not a
security defect — speed only.
