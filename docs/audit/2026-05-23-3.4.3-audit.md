## Scope: what changed in 3.4.3

### Code changes

Single file: `src/aes_gcm.cyr`. Twelve `var X[16];` declarations
become `secret var X[16];`. The corresponding end-of-function
`memset(&X, 0, 16)` calls are removed (the compiler now emits
zeroization on every return). All init-time `memset` calls
required for program correctness (GHASH-accumulator-starts-at-
zero, tail-pad zero convention) are retained — `secret var`
zeros on scope **exit**, not on declaration.

| Function | Buffers converted |
|---|---|
| `_ghash_mul` | `v` (working copy of H subkey) |
| `_ghash_update` | `block` (PT/CT scratch), `scratch`, `scratch2` (GHASH mul output) |
| `_aes_gcm_ctr_xor` | `ks` (AES-CTR keystream) |
| `aes_gcm_encrypt` | `h`, `y`, `s` |
| `aes_gcm_decrypt` | `h`, `y`, `s`, `expected_tag` |

### Not converted (deliberate)

- **`round_keys`** — 240-byte heap allocation via `fl_alloc`.
  `secret var` is stack-scope only; heap allocations are not
  reachable. Explicit `memset(round_keys, 0, 240)` immediately
  before `fl_free(round_keys)` is retained.
- **`j0`, `counter`, `zero_block`, `len_block`** — IV-derived,
  length-encoded, or constant-zero values. Not key material.
  Existing explicit end-`memset` calls retained as defense-in-
  depth without claiming the `secret var` semantic.

### Test surface

Unchanged: 1178 assertions, 0 failures. The full `tests/tcyr/`
sweep ran clean before and after the conversion.

### Bench surface

| Bench | Pre (3.4.2) | Post (3.4.3) | Delta |
|---|---|---|---|
| `aes_gcm_encrypt_1kb` | 716 µs | 723 µs | +1.0% |
| `aes_gcm_decrypt_1kb_valid` | 715 µs | 729 µs | +2.0% |
| `aes_gcm_decrypt_1kb_forged` | 720 µs | 730 µs | +1.4% |
| `aes256_encrypt_block` | 21 ns | 22 ns | within noise |
| `sha256_4kb` | 12 µs | 12 µs | 0% (control) |

The 1–2% AEAD deltas reflect (a) run-to-run noise and (b) a
small real cost from the compiler-emitted zeroization on every
return path. Acceptable for the security gain. Logged in
`benches/history.csv` as `v3.4.3-secret-var-aesgcm`.

## Step 1 — Input validation

Unchanged from 3.4.2. The conversion did not modify any
length, bounds, or format checks. The early-validation rejects
in encrypt/decrypt (`aad_len < 0`, `pt_len < 0`, `ct_len < 0`)
now run zeroization on the return — strictly better than 3.4.2,
where those returns left stack content in place.

## Step 2 — Buffer safety

Unchanged. `secret var X[N];` produces the same N-byte stack
allocation as `var X[N];`. No buffer-size declaration changed.
The init-time `memset` calls retained at `y` (GHASH accumulator
must start at zero), `len_block` (defense in depth), and
`zero_block` (constant zero input to AES) all match the prior
shape.

## Step 3 — Constant-time audit

Unchanged. No secret-data branches were added; the
compiler-emitted zeroization is a fixed-cost unconditional
sequence executed on return — does not depend on key bytes,
plaintext, or tag value.

## Step 4 — Key material zeroization

**Strict improvement.** Every error-return path in `aes_gcm_encrypt`
and `aes_gcm_decrypt` now zeroes the 3-or-4 secret-derived stack
locals (`h`, `y`, `s`, plus `expected_tag` in decrypt). Prior
shape: only the happy-path end-of-function memsets ran. The
helper functions (`_ghash_mul`, `_ghash_update`, `_aes_gcm_ctr_xor`)
gain the same property for their internal scratch.

Heap-allocated `round_keys` continues to be explicitly zeroed
before `fl_free` — unchanged from prior cycles.

## Step 5 — Syscall review

No syscalls added or modified.

## Step 6 — Pointer validation

Unchanged. No pointer-derefence patterns changed; pointer
parameters still arrive validated by caller.

## Step 7 — No command injection

N/A — no shell execution surface.

## Step 8 — No path traversal

N/A — no filesystem path surface.

## Step 9 — Known CVE check

No new crypto primitives. Reviewed against the AES-GCM CVE
classes:

- **Nonce-reuse class** (CVE-2016-7054 et al.) — not addressed
  by this patch (nonce management is caller responsibility per
  the public API contract). Unchanged.
- **Forbidden-attack / weak-tag class** — sigil's GHASH still
  computes the full 16-byte tag and compares constant-time via
  `ct_eq_bytes_lens`. Unchanged.
- **Timing side-channel on tag compare** — `ct_eq_bytes_lens`
  loops the full length even on mismatch. Unchanged.

## Step 10 — Findings

### INFO-1 — Test-output NUL byte in `*_verify_full.tcyr` summaries

**Where**: `tests/tcyr/sgx_verify_full.tcyr`,
`tests/tcyr/snp_verify_full.tcyr`,
`tests/tcyr/tdx_verify_full.tcyr` final summary lines.

**Note**: The three `*_verify_full.tcyr` files emit a NUL byte
(`\x00`) immediately before their `N passed, 0 failed (N total)`
summary line. This was discovered during the P(-1) step-1
cleanliness check while reconciling the per-file assertion sum
against the state.md baseline of 1178: a naïve `grep -oE`
counter using bash string interpolation drops those lines
because bash treats NUL as a string terminator.

**Implication**: Cosmetic only. No security impact. State.md's
1178 is correct; any test-counting CI helper that doesn't strip
NULs first will undercount by 38 (= 11+11+16). The current state
of the codebase is correct — only ad-hoc shell pipelines are
affected.

**Action**: Filed here as INFO. Closure target: whenever a
future edit touches those three test files (e.g. adding a new
`*_verify_full` test case), fix the print statement to emit a
plain newline before the summary line. Not blocking 3.4.3 ship.
