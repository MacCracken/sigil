## Step 1 — Input validation

`x509_parse(der, der_len, out_cert)` validates every TLV step against
`der_len` before dereference. The walker function `der_walk` is the
single gatekeeper:

- `pos >= buf_len` → `-1` (`:97`).
- tag byte does not match the expected tag → `-1` (`:101`).
- `_der_len` rejects indefinite-length form (`b0 == 0x80`,
  `nbytes == 0`) (`:79`).
- `_der_len` caps long-form length at 4 bytes — no field in a P-256
  cert needs more. Rejects `nbytes > 4` (`:80`).
- `_der_len` bounds-checks the length field itself: `pos + 1 + nbytes
  > buf_len` → `-1` (`:81`).
- `_der_len` rejects negative computed lengths after the bit pattern
  (`:90`) — defense against a hypothetical i64 sign bit getting set
  via a 5+-byte length, redundant with the 4-byte cap but cheap.
- `der_walk` rejects `content_pos + len > buf_len` (`:105`).

Time decoders (`_x509_parse_utctime`, `_x509_parse_gentime`) check:
- exact ASCII string length (13 or 15 bytes);
- trailing `Z` (rejects local-time / offset-time forms);
- each digit pair is in `[0,9]` (`_x509_dec2` / `_x509_dec4`);
- month, day, hour, minute, second ranges (`_x509_date_to_epoch`);
- day-in-month is valid for the year (handles February + leap years).

`x509_verify_chain` validates time windows for every cert (root and
intermediates included) before any signature check.

## Step 2 — Buffer safety

`x509.cyr` allocates:
- One 96-byte month-days table (`_x509_mdays`, initialised once).
- One 256-byte `X509Cert` struct per parsed cert (`x509_cert_alloc`).
- One 64-byte raw-r||s buffer per successful parse
  (`x509_parse`, `:570`) — see LOW-2 below for the lifetime caveat.

No `alloc` inside `der_walk` / `_der_len` / time decoders. The TLV
walker uses two i64 out-params (`out_content_pos`, `out_next_pos`)
that the caller stores in `var` locals; no heap writes during the
walk itself.

Every `store64` / `store8` writes within a known-sized buffer:
- `X509Cert` fields use compile-time-fixed offsets (0..199) into
  the 256-byte struct.
- The 64-byte `pubkey` field is written via a bounded `i < 64`
  loop (`:386-388`).
- The 64-byte `raw_sig` buffer is written via the existing
  `_ecdsa_der_int` helper, which already pads then copies
  `int_len ≤ 32` bytes per integer.

## Step 3 — Constant-time audit

X.509 parsing is **not** constant-time and does not need to be —
the cert chain itself is public material in every TEE attestation
scenario sigil is designed for. The chain walker invokes
`ecdsa_p256_verify` which inherits the 3.2.1 constant-time
discipline for the scalar-mul substrate. No new secret-data branch
is introduced.

The DN equality check (`_x509_dn_eq`, `:670`) is early-exit; DN
bytes are public.

## Step 4 — Key material zeroization

No secret material handled by `x509.cyr`. Subject public keys are
public; storing them in a long-lived buffer is correct.

## Step 5 — Syscall review

No syscalls.

## Step 6 — Pointer validation

The `der` pointer must be a valid buffer of at least `der_len` bytes
— caller responsibility. The walker never reads past `der_len`. The
parser stores raw pointers (`der_ptr`, `raw_sig` heap address) into
the cert struct; consumers MUST keep the underlying DER buffer
alive for the lifetime of the parsed cert. Documented in the cert
struct's field comment (`:213-237`).

`x509_verify_chain` derives the TBS-bytes pointer from
`load64(child + 0) + load64(child + 16)` (`:687-688`); both come
from the parser's own stash and are integers, not foreign pointers.
No untrusted pointer arithmetic.

## Step 7 — No command injection

No `sys_system` / `exec_*` calls.

## Step 8 — No path traversal

No filesystem access. The cert DER buffer comes from the caller; no
file or URL is opened by this module.

## Step 9 — Known CVE check

- **CVE-2008-5077 (OpenSSL OpenPGP signature bypass via ASN.1
  length confusion)**: sigil's `_der_len` rejects malformed length
  encodings and bounds-checks the result against `buf_len`. ✅
- **CVE-2014-1568 (NSS BER signature forgery via padding bytes)**:
  sigil's DER int decoder (`_ecdsa_der_int`, called inside
  `x509_parse` for the signature wrapping) rejects integers longer
  than 33 bytes and requires the leading-zero sign byte to actually
  be zero. ✅
- **CVE-2009-2409 (MD2 + null-prefix collision attacks on legacy
  certs)**: sigil only accepts `ecdsa-with-SHA256`. Any other
  signature algorithm OID is rejected at parse time
  (`_xp_parse_algid_ecdsa_sha256`, `:354`). ✅
- **CVE-2022-3602 / CVE-2022-3786 (OpenSSL X.509 punycode
  buffer overflow in name constraints)**: sigil does not parse
  name constraints, does not decode strings beyond a byte-level DN
  equality check. ✅
- **Algorithm confusion (Frankencerts)**: outer
  `signatureAlgorithm` and inner `tbsCertificate.signature` AlgID
  are BOTH parsed and BOTH must be ecdsa-with-SHA256
  (`x509_parse:539, :553`). Algorithm-substitution attacks are
  blocked at the structural level. ✅
- **Invalid-curve attack (Bos et al. 2014)**: closes LOW-1 from
  2026-05-21 — `_xp_parse_spki` calls `pt_is_on_curve` on the
  decoded `(Qx, Qy)` (`src/x509.cyr:393`). Off-curve public keys
  in SubjectPublicKeyInfo are rejected at parse time, **before**
  the cert reaches the chain walker. ✅

## Step 10 — Findings

### LOW-1 — `x509_parse` allocates per-call without free

**Where**: `x509_parse` (`src/x509.cyr:570`) allocates a 64-byte
`raw_sig` buffer for every successfully parsed cert.

**Issue**: Sigil's `alloc` is a bump allocator; allocations live
for the program's lifetime. A long-running consumer that parses a
high volume of certs (e.g. a kavach attestation backend handling
thousands of requests per second) will accumulate 64 bytes per
cert in retained memory.

**Threat model**: Not a security defect — memory exhaustion only.
For AGNOS's single-tenant trust verification surface this is
dormant.

**Recommended fix (deferred)**: when the kavach backend lands in
3.2.3 / 3.2.4 / 3.2.5, surface a `x509_cert_into(der, der_len,
out_cert, sig_scratch)` API that takes a caller-provided 64-byte
scratch — same pattern as `sv_verify_artifact_into` introduced in
3.2.0. Until then, the bump cost is bounded by call rate.

**Severity**: LOW — long-lived consumers should track allocations;
no exploit path.

### INFO-1 — Minimum-length encoding not enforced

**Where**: `_der_len` (`src/x509.cyr:89`).

**Note**: Strict DER requires that length fields use the
shortest possible encoding (e.g. `0x05` rather than `0x81 0x05`).
Sigil tolerates the long-form variant for any length to match
the reality of widely-deployed CAs that technically violate this
rule but otherwise validate. This is a deliberate laxity, not a
defect; analogous to 3.2.1's INFO-1 for ECDSA DER integers.

**Action**: tighten if a future consumer (e.g. a sigil-side
issuance flow) needs canonical-DER guarantees for hashing
purposes. For verification, the laxity is safe — the TBS bytes
themselves are hashed verbatim and the signature was generated
over those exact bytes, so canonical-form-or-not doesn't change
the verify outcome.

---

### INFO-2 — DN equality is raw byte-for-byte

**Where**: `_x509_dn_eq` / `_x509_issuer_matches`
(`src/x509.cyr:670, :703`).

**Note**: RFC 5280 §7.1 specifies a comparison algorithm for
Distinguished Names that involves whitespace normalisation,
case-folding for `PrintableString`, etc. Sigil compares the raw
DER bytes. For the SGX/SEV/TDX use case — where the cert chain
is machine-generated and the issuer DN in a child cert literally
copies the subject DN bytes from the parent cert — byte-equality
is the correct and stricter check. For interop with
human-administered CAs that re-encode DNs, a future bite may
need a relaxed comparator.

**Action**: track in `docs/development/3.2-tee-arc.md` if any
3.2.3+ integration surfaces a real-world DN that doesn't
byte-match across the chain link.
