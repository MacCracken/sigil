## 10-step Security Hardening checklist

1. **Input validation.** `chacha20_block` reads a fixed 32-byte
   key, 12-byte nonce, 32-bit counter, writes 64 bytes.
   `chacha20_xor` handles `in_len == 0` (loop body never runs) and
   negative `in_len` (the `while (done < in_len)` guard rejects) ‚Äî
   no over-read, no crash.
2. **Buffer safety.** `chacha20_block` state arrays `st`/`ws` are
   128 bytes (16 i64 slots) accessed at `i*8`, i < 16 ‚Üí max offset
   127. Output `out` is written at `i*4`, i < 16 ‚Üí bytes 0..63 (64
   bytes, the contract). `chacha20_xor` keystream `ks` is 64 bytes,
   read at `j < n ‚â§ 64`; `input`/`out` accessed at `done + j` with
   `done + j < in_len` (caller sizes both to `in_len`). The
   overlapping 4-byte loads in setup read within `key`/`nonce`.
3. **Constant-time.** ChaCha20 has no data-dependent branches: 20
   fixed rounds of add/xor/rotate over 32-bit words, identical for
   every input. The only branches (`while` bounds, `if (n > 64)`)
   test public lengths/positions, never key or plaintext bits.
4. **Key material zeroization.** `st` (holds the key), `ws`
   (working state), and `ks` (keystream) are `secret var` ‚Äî
   compiler-zeroized on every return.
5. **Syscall review.** None.
6. **Pointer validation.** No untrusted-pointer deref; all access
   at fixed/derived offsets bounded as in (2).
7. **No command injection.** None.
8. **No path traversal.** No filesystem access.
9. **Known-CVE check.** (a) Rotation / round-constant errors that
   would diverge from the spec ‚Äî excluded by the ¬ß2.3.2 keystream
   vector matching byte-for-byte. (b) Counter overflow ‚Üí keystream
   reuse ‚Äî see INFO-1. (c) Nonce reuse is a caller obligation
   (standard for any stream cipher); the AEAD bite (3.5.2) derives
   a fresh per-nonce Poly1305 key on top.
10. **Findings filed** ‚Äî below.

## Findings

### INFO-1 ‚Äî 32-bit counter usage cap (‚â§ 256 GiB per key/nonce)

`chacha20_xor` increments the block counter mod 2^32
(`ctr = (ctr + 1) & 0xffffffff`). After 2^32 blocks the counter
wraps and the keystream repeats, which for a stream cipher is
catastrophic (XOR of two ciphertexts leaks plaintext). RFC 8439
¬ß2.4 bounds a single (key, nonce) to 2^32 blocks = 256 GiB. This
is far above any sigil payload (signatures, configs, attestation
quotes ‚Äî KB range) and is documented as a hard caller obligation
in the `chacha20_xor` header. Not enforced in code (would require
a length check that never triggers in practice); flagged as a
usage contract.

### INFO-2 ‚Äî Single-call static scratch (serial-only)

`st`, `ws`, `ks` are function-scope arrays, which Cyrius allocates
as static globals (CLAUDE.md quirk #1). Each call fully writes them
before reading, so serial use is correct; concurrent callers would
race. ChaCha20 is not yet on a multi-threaded path (the v3.6
per-worker-scratch cycle covers that). Documented in the module
header.
