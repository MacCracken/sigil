## sha384.cyr — wrapper over sha512_transform

Trivially correct: reuses the FIPS 180-4 §5.3.4 IV and truncates
the 64-byte SHA-512 finalize output to 48 bytes. The
compression core (`sha512_transform`) was already audited at
2.9.x. Vectors against FIPS 180-4 App. B.1 (empty, "abc",
112-char) all pass.

No new findings.

## ecdsa_p384.cyr — structural mirror of ecdsa_p256.cyr

**Step 1 (input validation)**: `ecdsa_p384_verify` checks r, s ∈
[1, n-1] (rejects zero and ≥ n) at `:843-846` and validates Q
on-curve via `pt_p384_is_on_curve` at `:851`. Same shape as
3.2.1's verify, which the prior audit already covered.

**Step 2 (buffer safety)**: All u384 buffers are 48 bytes; all
double-precision buffers are 96 bytes; point buffers are 144
bytes. Every `load64`/`store64` uses fixed compile-time offsets
within these allocations. The schoolbook 384×384 multiply
(`u384_mul_full`) and the long-division reduction
(`_p384_long_div_reduce`) operate on 6-limb arrays with bounded
i (0..5 / 0..383) loops.

**Step 3 (constant-time)**: Same posture as 3.2.1 —
Montgomery-ladder scalar mul + `pt384_cswap` keep the primitive
constant-time for any future sign / DH caller; the verify path
itself operates on public inputs and uses early-exit
comparisons.

**Step 4 (zeroization)**: No secret material flows through
`ecdsa_p384_verify`. If a future bite adds P-384 sign, every
`_p384_*` / `_pt384*` scratch global needs `secret var` or
explicit `memset(_, 0, 48)` — same caveat 3.2.1's audit flagged
for P-256.

**Step 9 (CVE check)**:
- **CVE-2022-21449 (zero r/s ECDSA bypass)**: rejected at
  `:843-844`. ✅
- **Invalid-curve attack (Bos et al. 2014)**: on-curve check
  inside verify at `:851`. ✅
- **CVE-2020-0601 (curve param spoofing)**: P-384 constants are
  hardcoded via `u384_from_hex` at init; not parsed from caller
  input. ✅

RFC 6979 §A.2.6 vectors (both `sample` and `test` messages)
pass; 2G against published SEC 2 reference passes; n·G == O
passes. No CRITICAL / HIGH / MEDIUM findings.

## sev_snp.cyr — AMD attestation report

**Step 1 (input validation)**:
- `snp_report_parse` requires `buf_len == SNP_REPORT_SIZE`
  (1184 B exactly) — `:124`. Anything else rejected.
- `version != 2` rejected (`:127`).
- `signature_algo != 1` (ECDSA P-384 / SHA-384) rejected
  (`:131`). Other algos (e.g. AMD's potential future RSA
  variants) are not handled — by design.
- `snp_report_verify` re-checks `signature_algo` (`:201`) as
  defense-in-depth against a caller mutating the parsed struct
  between parse and verify.
- The signature high-padding zero check (`:213-216`) blocks an
  attacker stuffing extra bits into the 24-byte zero-pad
  region.

**Step 2 (buffer safety)**: `_snp_load_u32` / `_snp_load_u64`
read byte-by-byte; the parser's compile-time fixed offsets all
sit within `[0, 0x2A0)` which is well under the validated 1184-
byte buffer. Per-call allocation: one 96-byte BE-sig scratch
(`_snp_sig_be`) initialised once via `_snp_v_init` — see LOW-1.

**Step 9 (CVE check)**:
- **Algorithm confusion**: signature_algo is enforced strictly
  at both parse and verify time (`:131, :201`). ✅
- **Length-confusion attacks**: report size is exactly 1184
  bytes; no variable-length fields inside the body or the
  signature section (the trailing 368 bytes after r/s are
  required-zero but sigil currently only checks the r-pad
  region; see INFO-2). ✅ for the load-bearing case.
- **VCEK substitution**: 3.2.4 takes a caller-provided VCEK
  pubkey; the ARK → ASK → VCEK X.509 chain walk is the
  caller's responsibility. INFO-1 tracks the deferred
  integrated walk.

## Step 10 — Findings

### LOW-1 — `_snp_v_init` perm-lifetime allocation

**Where**: `src/sev_snp.cyr:189-194`.

**Issue**: Same pattern as 3.2.2 LOW-1 — `_snp_v_init` allocates
a 96-byte BE-sig scratch buffer once and retains it for program
lifetime. Sigil's bump allocator never reclaims it.

**Threat model**: Not a security defect — 96 bytes one-time
cost, the buffer is reused across all subsequent verify calls.
Documented for consistency with the 3.2.2 finding.

**Severity**: LOW — defense-in-depth tracking only.

---

### INFO-1 — VCEK chain walk deferred to caller

**Where**: `snp_report_verify` and the `sev_snp.cyr` file header
(`:25-32`).

**Note**: 3.2.4 takes a pre-validated VCEK pubkey as a 96-byte
raw Qx||Qy buffer. The ARK → ASK → VCEK X.509 chain walk
(typically PEM-encoded, fetched from AMD KDS once per host) is
the caller's responsibility. The `x509_*` surface from 3.2.2
covers the walking; only PEM decoding remains as a future
patch.

**Action**: when an integration consumer (kavach in particular)
surfaces, ship the PEM decoder + an `snp_report_verify_full(
report, ark_root)` wrapper that internalises the chain walk.
Until then, the per-piece API is the supported surface.

---

### INFO-2 — Trailing reserved bytes after signature not validated

**Where**: `src/sev_snp.cyr:213-216`.

**Note**: AMD's spec defines the 368 bytes following sig.s
(offset 0x330..0x4A0) as reserved-zero. Sigil currently
validates the 24-byte high padding inside each of r and s
(defense in depth — see Step 9), but does NOT validate the
trailing 368-byte reserved region. A non-zero trailer is a
spec violation but doesn't affect the signature check itself
(the signed body ends at 0x2A0; the trailer is after the
signature).

**Action**: tighten if a future integration calls for strict-
format enforcement. Currently a deliberate laxity that matches
real-world report variability across AMD firmware revisions.

---

### INFO-3 — Cyrius long-`main()` quirk hit in test development

**Where**: `tests/tcyr/sev_snp.tcyr` (test helper refactor).

**Note**: The initial sev_snp test was structured as a single
long `main()` body with ~150 lines of locals (vars + assert
calls). Cyrius's single-pass compiler segfaulted at the first
parse call inside that shape — the same code split across
multiple helper functions (`run_parse_tests`,
`run_malformed_tests`, etc.) runs cleanly. This is a Cyrius
quirk (likely related to CLAUDE.md quirk #1's local clobbering
at scale), not a sigil bug; the workaround is to keep test
`main()` bodies short and delegate.

**Action**: file a Cyrius upstream issue if a clean repro
isolates further. For now, the helper-split pattern is the
recommended test shape for sigil tests with many locals.
Future tests in this arc and beyond should adopt the same
pattern from the start.

---

### INFO-4 — P-384 reduction speed

**Where**: `src/ecdsa_p384.cyr:300` (`_p384_long_div_reduce`).

**Note**: Same shape as 3.2.1's `_p256_long_div_reduce` — bit-
by-bit shift-subtract over 384 iterations per `fp_p384_mul`
call. Expect ~3× the per-verify cost of P-256 (verify-path
total around 500-700 ms). Documented as a non-defect, optimised
via Solinas as part of the same bench-tuning follow-up the
3.2.1 audit's INFO-2 already tracks.

**Action**: none. Roll into the existing Solinas-tuning item.
