No exploitable findings. The SHA-384 HMAC/HKDF surface mirrors the
audited SHA-256 surface (`src/hmac.cyr` / `src/hkdf.cyr`), correctly
re-based on B=128 / L=48.

## 1. Input validation

- `hmac_sha384`: `key_len > 128` routes through `sha384(key,...)` →
  48-byte K′ (zero-padded to 128); shorter keys zero-padded. No length
  is trusted unchecked; all buffer writes are bounded by the fixed
  128/48 constants.
- `hkdf_extract_sha384`: rejects negative `salt_len` / `ikm_len` (−1).
  Empty salt → 48 zero bytes (RFC 5869 §2.2).
- `hkdf_expand_sha384`: rejects negative lengths (−1) and `out_len >
  255·48 = 12240` (−2, RFC 5869 §2.3 — correctly the SHA-384 cap, not
  the SHA-256 8160). `out_len == 0` returns early.

## 2. Buffer safety

- `kprime384[128]`, `ipad384[128]`, `opad384[128]`, `inner_hash384[48]`:
  block size B=128 and digest L=48 used consistently — the documented
  "wrinkle" (B is 128 for SHA-384, not 64). The `key_len <= 128` branch
  relies on the `memset(&kprime384,0,128)` baseline; verified present.
- `hkdf_expand_sha384` scratch = `48 + info_len + 1` (`fl_alloc`),
  exactly the max "T(i−1) || info || counter" input. `take =
  min(48, out_len − produced)` bounds every output copy; the loop
  terminates because `produced` strictly increases by `take ≥ 1`.

## 3. Constant-time posture

HMAC/HKDF inputs (key, salt, IKM) are secret, but there is **no branch
on secret data**: the only conditionals are on `key_len`/`info_len`/
`out_len`/`counter` — all public lengths. The XOR-pad loops run the full
128 bytes unconditionally. SHA-384 itself is data-independent. The 19
KAT assertions (RFC 4231 vectors, Python-`hmac`-reproduced) confirm
functional correctness end-to-end.

## 4. Key-material zeroization

- `hmac_sha384`: K′, ipad, opad are `secret var` → compiler-zeroized on
  every return (Cyrius 5.3.5+). `inner_hash384` is a public intermediate
  digest, intentionally not cleared — **INFO-2** (it is HMAC output H((K′
  ⊕ ipad)‖m), which the outer hash already commits to publicly; no key
  recovery path).
- `hkdf_extract_sha384`: the local `zero_salt384` is re-`memset` after
  use. PRK is written to the caller's buffer (caller owns its lifetime;
  the `hkdf_sha384` wrapper holds it in a `secret var`).
- `hkdf_expand_sha384`: `t384_prev` / `t384_current` are `secret var`
  (zeroed on exit); the heap scratch is `memset` to 0 before `fl_free`;
  the global pointers are wiped to 0 after use so nothing dangles
  between calls.
- `hkdf_sha384`: the 48-byte PRK is a `secret var`, zeroized on both the
  `rc1` error path and the success path.

## 5. Allocation / lifetime

`hkdf_expand_sha384` uses `fl_alloc` + `fl_free` for the per-call
scratch (correct freelist discipline — not bump `alloc`). The module
globals (`_hkdf384_*`) are a single-threaded register-allocator
workaround documented in-file and wiped after each call — **INFO-1**.
HKDF-SHA384 is not on a parallel path; if it ever joins one it needs the
3.6 bank treatment (same note as the SHA-256 HKDF).

## 6–10. Syscalls / pointers / injection / path traversal / CVE class

No syscalls, no external command execution, no filesystem paths, no raw
pointer deref of untrusted input (all reads bounded by validated
lengths). HMAC length-extension is structurally prevented by the nested
construction. No known CVE pattern applies to a spec-faithful HMAC/HKDF.
