**Scope**: The 3.8.0 ChaCha20 + X25519 per-worker banking — making both modules

- `src/chacha20.cyr` — `chacha20_block` (`st`/`ws`) and `chacha20_xor` (`ks`).

## The bug that was caught (and fixed) in development

The first banking attempt kept the arrays as `secret var` (preserving the
originals' compiler-zeroize-on-exit). But `secret var X[N]` zeroizes the **whole**
array on scope exit — so when worker A's `chacha20_block` / `x25519` returned, it
zeroed **all 8 lanes**, clobbering a concurrent worker B's in-flight lane. The new
`tests/tcyr/banking_concurrent.tcyr` race-detector **caught this immediately**
(intermittent 1–2 mismatches across runs — the classic race signature).

**Fix**: plain `var` (matching the proven `fp_mul`/`fp_inv` banking, which is also
plain `var`) **+ an explicit per-lane `memset` on exit** — wiping only the calling
worker's own lane, never a sibling's. This both eliminates the clobber AND
preserves the secret-zeroization intent (x25519's `W` holds the clamped secret
scalar `k`; chacha20's `st`/`ws`/`ks` hold key-derived keystream state).

## Verification

1. **Race-detector** (`banking_concurrent.tcyr`): 4 worker threads on distinct
   banks (1–4), each recomputing a deterministic x25519 + chacha20 result in a
   tight loop, compared byte-for-byte against the serial (bank-0) reference.
   **5/5 clean runs** post-fix; the pre-fix code failed it. A serial KAT at bank 0
   alone cannot catch a wrong lane stride (offset 0 hides it) — only this
   non-zero-bank concurrent run does. +2 assertions.
2. **Serial KATs**: `chacha20.tcyr` (RFC 8439), `chacha20poly1305.tcyr`,
   `x25519.tcyr` (RFC 7748) all green — banking is transparent at bank 0.
3. **Full suite**: 1483 assertions, 0 failures; cyrlint clean.
4. The per-module tests gained `lib/thread_local.cyr` + `src/crypto_scratch.cyr`
   includes so `cbank()` resolves (the 3.7.8 ud2/SIGILL-footgun class — a
   bundle/test that references `cbank()` without those includes would `ud2`).

## Notes

- **Secret zeroization is preserved, per-lane.** Each banked function memsets its
  own lane before return. This is *stronger* than the proven `fp_*` banking, which
  uses plain `var` with **no** zeroization (a deliberate 3.6 trade-off for the
  transient field scratch); here the original `secret var` intent is kept via the
  explicit per-lane wipe.
- **Defensive only.** Neither ChaCha20 nor X25519 is reachable from
  `sv_verify_batch` today (the batch worker only drives ed25519/sha/ecdsa/mldsa
  artifact verification). The banking closes a *latent* race ahead of any future
  concurrent consumer (e.g. a parallel TLS handshake doing X25519, or batched AEAD).

## Verdict

The 3.8.0 banking is **correct, race-validated, and secret-zeroization-preserving**.
The one real hazard (cross-lane `secret var` clobber) was caught by the new
race-detector and fixed before shipping. Audit floor stays empty.
