## Step 10 — Findings

### INFO-1 — Dist bundle drift root cause

**Where**: `cyrius.cyml [lib].modules` and the retired
`cyrius distlib` subcommand.

**Note**: The original `cyrius distlib` subcommand (referenced
in `dist/sigil.cyr`'s header comment "Generated by: cyrius
distlib") was retired from cyrius's CLI somewhere in the
3.x.x → 5.x.x cyrius evolution. After retirement, sigil's dist
bundle was no longer regenerated by any release-side automation
— the file became frozen at the last cyrius release that had
the subcommand. The src tree progressed across the entire
3.2.x TEE arc (six bites adding ~3,200 lines of new code) plus
the 3.4.x TEE-completion cycle (~485 lines of new code), but
the dist bundle didn't see any of it.

**Implication**: Consumers building against `dist/sigil.cyr`
since 3.2.0 (2026-05-21) would have been missing every public
API added in the intervening ten module additions —
including all of the 3.2.x TEE attestation primitives and all
three `*_verify_full` orchestrators from 3.4. Whether any such
consumer exists is unknown — sigil's downstream consumers
(daimon, kavach, ark, …) build against `src/lib.cyr` directly
against a sibling checkout, not against the dist bundle.
Reading the source-tree integration shape in the audit history
and CHANGELOG indicates the dist bundle is rarely the actual
consumer surface for first-party AGNOS apps, but a third-party
consumer pulling sigil as a single-file `.cyr` would have been
silently missing the entire TEE attestation surface.

**Action**: Closed in this release. The new
`scripts/regen-dist.sh` is the replacement for the retired
cyrius subcommand. CLAUDE.md and CONTRIBUTING.md both reference
the regen flow in the Documentation Structure / closeout
discussions. Future cycles that modify `[lib].modules` (or
modify source files whose changes need to land in the dist)
re-run `scripts/regen-dist.sh` as part of the closeout pass.

### INFO-2 — Regen automation gap (closure target: future cycle)

**Where**: closeout-pass workflow (CLAUDE.md § "Closeout Pass").

**Note**: The regen step is currently manual. A future patch
could add a `make check` target (or extend `scripts/check.sh`)
to invoke `scripts/regen-dist.sh` and diff the result against
the committed `dist/sigil.cyr` — failing the check if the
bundle is stale relative to `[lib].modules`. This is the
"module-list drift gate" candidate listed in
`docs/doc-health.md` under "Programmatic gates (future)".

**Action**: Not blocking for 3.4.2 ship. File as a candidate
programmatic gate in `docs/doc-health.md` (already done as
part of the 2026-05-22 doc sweep). Land when a future cycle
touches the closeout workflow.
