## Montgomery-on-verify

### Performance (claims backed by `benches/history.csv`, row `v3.6.6-rsa-montgomery`)
- `rsa2048_verify_schoolbook` (old `bn_modexp`): **11.680 ms**
- `rsa2048_verify_montgomery` (new `bn_mont_modexp`): **3.401 ms** →
  **3.43×** on the verify modexp; end-to-end `rsa2048_verify_sha256`
  **3.385 ms**.
- Sign: `rsa2048_sign_sha256_crt` **74.82 ms** vs the full-width secret
  modexp `rsa2048_modexp_fullwidth` **242.90 ms** → the CRT path is
  **~3.3×** the full-width cost (the existing 3.6.4 CRT win, now
  measured).

### Correctness
`bn_mont_modexp` is KAT-equal to `bn_modexp` for odd moduli
(`bignum.tcyr`), and the v1.5 + PSS signatures still match the external
byte-for-byte KATs after the swap (`rsa.tcyr`). The three swapped sites
use distinct, correctly-sized operand buffers (512 B = 64 limbs ≥
`limbs`); `bn_mont_modexp`'s internal scratch (`_bn_mont_t[528]` = 66
limbs ≥ limbs+2; `_bn_mont_r2src` ≥ 2·limbs+1) covers the worst case;
calls are sequential and each fully re-initializes its scratch — safe
under the single-threaded RSA posture (ADR 0001 banking not required).

### LOW-A (found by the review; FIXED in 3.6.6)
`bn_mont_modexp` has a hard **odd-modulus** precondition (its
`_bn_mont_n0inv` Newton inverse converges only for odd `n[0]`). On the
verify path the modulus is attacker-controlled (SPKI / caller key), and
the schoolbook `bn_modexp` it replaced accepted *any* modulus — so the
swap silently narrowed the domain with the precondition only asserted in
a comment. The review confirmed (empirically) that an **even** modulus
yields a value diverging from schoolbook.

- **Not exploitable**: an even `n` is not a valid RSA key (no legitimate
  signature exists for it); the recovered garbage EM fails the
  constant-time EM compare, so verify still returns 0 (reject). No
  forgery, no bypass, no OOB (the even-modulus garbage stays within the
  exactly-sized `_bn_mont_t`).
- **Fix**: `_rsa_recover_em` now rejects an even (or zero) modulus
  (`if ((load64(&_rsa_n) & 1) == 0) return -1`) before the ladder —
  enforcing the precondition at the trust boundary per CLAUDE.md §
  Input validation. Regression test added (`rsa.tcyr`: "even modulus
  rejected"). The sign-path sites (`_rsa_raw_sign`) take `n` from a
  parsed private key (`p·q`, always odd) and are additionally guarded by
  verify-after-sign, so they fail closed without a separate check.

## pem_decode_privkey → RSAK

The RSA branches (PKCS#1 `kind==2`, PKCS#8 `algo==RSA`) now call
`rsa_privkey_from_der(pool, der_len, key_out)` and return `RSAK_SIZE`
**only after** confirming `key_max >= RSAK_SIZE` — the guard precedes any
write to `key_out`, so there is no overrun (the review specifically
probed this and confirmed it safe). EC/Ed25519 contracts are unchanged
(still return 32/48 into the caller buffer). `RSAK_SIZE` /
`rsa_privkey_from_der` resolve cleanly (bignum + rsa precede privkey in
the bundle/`lib.cyr` order). The bump-allocated DER `pool` carries the
same per-call allocator-lifetime caveat documented for `pem_decode_certs`
(not a new finding). Tests: `privkey.tcyr` +4 (RSA → RSAK struct, modulus
matches; small-buffer sentinel retained).
