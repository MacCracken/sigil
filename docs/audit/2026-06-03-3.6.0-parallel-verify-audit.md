# Sigil Security Audit — 2026-06-03 (3.6.0 parallel verify / mutex drop)

**Auditor**: Internal (Claude/Opus)
**Version audited**: 3.6.0
**Cyrius toolchain**: 6.0.52 (pin); deps agnosys 1.3.2, sakshi 2.2.6
**Prior audit**: `2026-05-28-3.5.9-ecdsa-sign-audit.md`
**Scope**: The 3.6 parallel-verify refactor — drops
`_sigil_batch_mutex` from `sv_verify_batch` so the batch path runs
the crypto truly in parallel. New module `src/crypto_scratch.cyr`
(per-thread bank selection over cyrius 6.0.52 thread-local storage);
per-thread *banking* of every transient working array on the verify
call path across `src/sha_ni.cyr`, `src/sha256.cyr`, `src/sha512.cyr`,
`src/bigint_ext.cyr`, `src/ed25519.cyr`, `src/trust.cyr`; the
mutex-drop + per-worker bank assignment in `src/verify.cyr`; and an
alloc removal in `ge_identity`. No crypto *algorithm* changed.
**Method**: Work Loop step 7 + the 10-step Security Hardening
checklist, focused on the load-bearing concern for this change —
**data races** on shared working state once the serialising mutex is
gone (a race here corrupts a content hash or a signature check →
wrong trust verdict).

## The race surface (load-bearing)

`var X[N]` inside a function in cyrius is a **static function-scope
global**, not a stack-local (CLAUDE.md quirk #1; re-confirmed under
cycc 6.0.52 via `tests/tcyr/var_array_semantics.tcyr` during this
cycle). Under the old design the batch mutex serialised the entire
`sv_verify_artifact_into` call so these shared arrays never collided.
3.6 removes the mutex, so each was given a per-thread *bank*: the
array is widened to `[N * SIGIL_CRYPTO_BANKS]` and the function
operates on lane `&X + cbank()*N`. Workers take banks `1..workers`
(assigned once at `_batch_worker` entry via `crypto_bank_set`); the
main/serial path is bank 0. Lanes are disjoint by construction, so no
two concurrently-live threads touch the same bytes — no cross-function
offset map, no possibility of overlap.

Every transient working array reachable from `ed25519_verify` /
`hash_file_into` was enumerated and banked:

- `sha_ni.cyr` — `sha256_transform_ni` block scratch (`raw`). (SHA-NI
  is the live path on the audit host; the self-test `raw` and
  `_sha_ni_st_ctx` run only on the main thread during pre-warm and are
  left unbanked, correctly.)
- `sha256.cyr` — `sha256_transform` message schedule `W`.
- `sha512.cyr` — `sha512_transform` message schedule `W` (Ed25519 HRAM
  hash).
- `bigint_ext.cyr` — `u512_mod_p` (`aH,aL,prod,lowp,r,extra`),
  `fp_mul` (`prod`), `fp_pow` (`tmp,result,base_buf`), `fp_inv`
  (8 addition-chain temporaries). **`fp_inv` was missed on the first
  inventory pass** (its arrays are declared multiple-per-line) and
  caught by the mutex-off race test failing — see "Verification".
- `ed25519.cyr` — `ge_add`, `ge_double`, `ge_scalarmult`,
  `ge_scalarmult_base`, `ge_to_bytes`, `ge_from_bytes`, `sc_reduce`,
  and `ed25519_verify`'s own point/scalar buffers + its inline
  SHA-512 context (`hctx`).
- `trust.cyr` — `hash_file_into` file-read buffer (`buf`) and `digest`
  (a shared read buffer would interleave file bytes across workers and
  corrupt the content hash).

Off-path static arrays were verified **not reachable** from a batch
worker and deliberately left unbanked: the `ed25519_sign` /
`ed25519_keypair` / `ed25519_generate_keypair` / `sc_muladd` buffers
(signing is not parallelised), `_build_ed_B_table` / `_init_ed_p` /
curve-constant init (main-thread pre-warm), the SHA-NI self-test,
`hash_file` / `hash_file_with` / `sha256_hex` / `sha512_hex` (serial
wrapper + convenience helpers), and `json_write_escaped` (audit
emission, which already runs post-join on the main thread).

## Alloc on the hot path (quirk #7)

The stdlib bump allocator and `fl_alloc` are not thread-safe. The
worker path was already alloc-free (3.2.0 pre-allocated per-artifact
`VerifyScratch` pool) **except** for one latent call: `ge_identity`
used `u256_from(0)`/`u256_from(1)`, each of which `alloc(32)`s, and it
is invoked many times per verify (via `ge_scalarmult` /
`_ge_table_select`). Harmless under the old mutex; a race once it is
dropped. Rewritten alloc-free (`memset` + two stores), which is also
faster. Confirmed no remaining `alloc`/`fl_alloc`/`vec_*` on the
worker path; crypto module inits and the TLS main-thread block are
pre-warmed single-threaded in `sv_verify_batch` before fan-out.

## Constant-time

Banking changes only the **base address** of working buffers by a
per-thread, data-independent offset; it introduces no branch on secret
data. `cbank()`'s only branch is on `_crypto_tls_inited` (thread
context — public) and the bank index derives from the worker index
(public), never from key material. The branchless primitives
(`ge_cmov`, `_ge_table_select`, `ct_select`, the constant-time
scalar-mul loop) are unchanged. Verify operates exclusively on public
data (public key, signature, content hash). Constant-time posture is
unchanged from 3.5.x.

## Key zeroization

No secret-material buffer was banked. The verify path carries no
private keys. The `secret var` buffers in the signing/keypair paths
were intentionally **not** banked: `secret var` zeroises the whole
array on scope exit, which under concurrent banking would wipe a
sibling thread's live lane. Since signing is single-threaded this is
moot today, but banking them is recorded as unsafe-by-design to
prevent a future "parallel sign" change from doing so naively.

## TLS / bounds

- `src/crypto_scratch.cyr` uses cyrius 6.0.52 `lib/thread_local.cyr`
  (slot-based TLS). Workers inherit a zeroed 4 KB TLS block from
  `thread_create`'s `CLONE_SETTLS` (lib/thread.cyr); the main thread
  installs one via `crypto_tls_main_init()` (pre-warm) with `cbank()`
  self-installing as a backstop for direct/serial callers.
- Slot usage: 1 of 16 (slot 0). Bank index masked with
  `& (SIGIL_CRYPTO_BANKS-1)` = `& 7`, so it can never index outside
  the `[0, 8)` lane range regardless of slot contents (defence in
  depth). Buffer bounds: lane `bank*N` with `bank ≤ 7` spans
  `[7N, 8N) ⊂ [0, 8N)` — in-bounds for every widened `[N*8]` array.
- `SIGIL_CRYPTO_BANKS = 8` vs default `_SIGIL_BATCH_WORKERS = 4`
  (banks 1..4) — comfortable headroom.

## Verification

- **Single-threaded correctness**: full `.tcyr` suite 47/47 (bank 0).
- **Race detection**: `tests/tcyr/batch_parallel.tcyr` mutex-off,
  **35 consecutive clean runs at 228/228**. The first mutex-off run
  *failed* (non-deterministic trust verdicts), which surfaced the
  un-banked `fp_inv` and `hash_file_into` buffers; both were banked and
  the test went green and stayed green. This is the empirical
  guarantee that the race surface is closed.
- **Performance** (`benches/history.csv` row `v3.6-parallel-crypto`):
  `sv_verify_batch_64` 422.867 ms → **123.563 ms = 3.42×** at 4
  workers (target ≥3×); batch_4 3.50×, batch_16 3.39×. `batch_1`
  (serial path) 6.897 ms shows no regression from the per-call
  `cbank()` overhead.

## Findings

- **LOW-1** — `sv_verify_batch` clamps `workers` to
  `SIGIL_CRYPTO_BANKS - 1` silently. If a future change raises
  `_SIGIL_BATCH_WORKERS` past 7 without widening `SIGIL_CRYPTO_BANKS`,
  parallelism is silently capped (no correctness impact — the clamp
  preserves the disjoint-bank invariant; the mask is a second
  backstop). Acceptable; noted for the next worker-count change.
- **INFO-1** — banked arrays grow `.bss` ~8× (notably the 4096-byte
  file buffer → 32 KB). No heap, no per-call alloc; negligible for a
  trust daemon.

Zero CRITICAL / HIGH / MEDIUM. The audit-floor LOWs from prior cycles
(bump-allocator lifetime on init paths) are unaffected and carry
forward to the v3.7 `_into` cycle.
