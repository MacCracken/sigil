## Threat model

Private-key DER/PEM is attacker-controllable in the TLS-server
key-loading path (`tls_native_new_server(cert_chain, …, key,
key_len)`): a misconfigured or hostile key file must not corrupt
memory or be silently misparsed into the wrong curve/algorithm.

## Checklist findings

1. **Input validation** — every `der_walk` / `_der_len` return is
   checked `< 0` before the returned content offset is used. The
   version INTEGER is read only after a successful `der_walk`
   (`vlen < 1` rejected). Algorithm and curve OIDs are **full-byte
   matched** (`_privkey_oid_eq` compares length *and* every byte), so
   a P-384 key cannot be accepted by the P-256 parser, an EC key
   cannot be accepted by the Ed25519 parser, and an OID prefix cannot
   spoof a longer OID. Negative tests in `privkey.tcyr` exercise all
   three cross-rejections + a truncated-length reject. **Clean.**
2. **Buffer safety** — the scalar/seed copy goes through
   `_privkey_copy_be_fixed`, which rejects `src_len > dst_len` before
   writing and left-pads the rest, so a malformed length field cannot
   overrun the caller's fixed (32 / 48) buffer. The Ed25519 inner
   OCTET STRING is required to be exactly 32 bytes (`ilen != 32`
   reject) before the `memcpy`. `pem_decode_privkey` requires
   `key_max >= 48` up front and the base64 decode is bounded by
   `_pem_b64_decode`'s `out_max` (= `pem_len`). Marker buffers are
   sized to their exact literal lengths. **Clean.**
3. **Constant-time** — N/A in the cryptographic sense: parsing
   structure/length/OID bytes is not secret-dependent (the key
   *value* is copied, never branched on). No comparison of secret
   material here. **Clean.**
4. **Key material zeroization** — the parser holds no key bytes in
   locals: the scalar/seed is copied directly from the DER buffer
   into the caller's `key_out`, which the caller owns and SHOULD
   declare `secret var` (documented in the module header). One
   residue: `pem_decode_privkey` leaves the decoded DER (which
   contains the private scalar) in the bump pool; this is the same
   one-shot-lifetime shape as `pem_decode_certs` and is covered by
   the INFO note. Callers needing zeroization of the intermediate DER
   should decode themselves and `memset` their buffer. **Acceptable;
   documented.**
5. **Syscall review** — none added. **N/A.**
6. **Pointer validation** — no raw dereference of untrusted input:
   every read is `load8(buf + off + i)` where `off` came from a
   bounds-checked `der_walk` and `i < len` (also from `der_walk`).
   `_pem_find` / `_pem_match_at` bound every compare against
   `buf_len`. **Clean.**
7. **No command injection** — none. **N/A.**
8. **No path traversal** — the parsers take buffers, not paths; the
   test inlines its PEM data (no file reads). **N/A.**
9. **Known-CVE check (key parsing)** — common ASN.1/PEM parser
   pitfalls:
   - *Length-field overflow / indefinite length* — `_der_len`
     rejects indefinite (0x80) and > 4-byte lengths and bounds the
     decoded length against `buf_len` (inherited from x509.cyr).
   - *OID confusion (curve/algo substitution)* — full-byte OID match;
     P-256↔P-384 and Ed25519↔EC substitution rejected by test.
   - *Integer/length sign issues* — scalar copied as an opaque fixed
     big-endian field; no sign interpretation. `_privkey_copy_be_fixed`
     left-pads short values rather than trusting an attacker length.
   - *PEM label confusion* — the three labels are distinct
     substrings; the bare `PRIVATE KEY` label routes by the algid OID
     (`_privkey_pkcs8_algo`), not by guessing.
   No new CVE class introduced. **Clean.**
10. **Findings filed** — this document. Zero CRITICAL/HIGH/MEDIUM/LOW.

## Correctness evidence

- `tests/tcyr/privkey.tcyr` (+33 assertions): Ed25519 PKCS#8 (seed
  parse → `ed25519_keypair` derives the known public key → sign/verify
  roundtrip), P-256/P-384 SEC1 + PKCS#8 scalar parse (vs known
  bytes), curve/OID/truncation rejection, and the PEM layer
  (label + algo auto-dispatch for all five EC/Ed25519 forms + the
  RSA-recognized sentinel). All pass.
- Vectors generated with OpenSSL; the inlined PEM blocks carry the
  same base64-encoded keys as the DER vectors, so the DER and PEM
  layers cross-check the same keys.
- Full suite: 46 files / 1264 assertions, 0 failures. No regression
  in any existing module.
