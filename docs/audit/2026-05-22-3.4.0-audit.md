## Step 1 — Input validation

### `pem.cyr`

`pem_decode_certs_into(pem, pem_len, out_chain, max_certs, der_pool, der_pool_size)`:

- `pem_len < 0` rejected (`pem.cyr:199`).
- `max_certs <= 0` rejected (`pem.cyr:200`).
- `der_pool_size < 0` rejected (`pem.cyr:201`).
- BEGIN marker not found → `count == 0` → return -1 (no silent
  empty-chain) (`pem.cyr:217-221`).
- BEGIN found without matching END → -1 (`pem.cyr:227-228`).
- Base64 invalid char → -1 (`pem.cyr:134`).
- Base64 padding (`=`) in non-final position → -1
  (`pem.cyr:142-149,157`).
- More than 2 padding chars in a final quad → -1
  (`pem.cyr:158-163`).
- Empty body (zero alphabet chars between BEGIN/END) → -1
  (`pem.cyr:181`).
- Per-quad over-fill (>4 chars) → -1 (`pem.cyr:144,148`).
- Decoded byte would exceed `der_pool_size` → -1
  (`pem.cyr:164`).
- Cert count would exceed `max_certs` → -1 (`pem.cyr:235`).

`pem_decode_certs(pem, pem_len, out_chain, max_certs)` adds the
`pem_len <= 0` guard before allocating (`pem.cyr:271`).

### `sgx_quote_verify_full` / `tdx_quote_verify_full`

- `cert_data_type != PCK_PEM (5)` rejected first (`sgx.cyr:421`,
  `tdx.cyr:380`).
- `cert_data_len <= 0` rejected (`sgx.cyr:425`, `tdx.cyr:384`).
- `root_ca_der_len <= 0` rejected (`sgx.cyr:447`, `tdx.cyr:406`).
- `pem_decode_certs` return < 1 rejected (`sgx.cyr:432`,
  `tdx.cyr:391`).
- Per-cert `x509_parse` failure rejected (`sgx.cyr:441`,
  `tdx.cyr:400`).
- Root cert `x509_parse` failure rejected (`sgx.cyr:449`,
  `tdx.cyr:408`).

### `tdx.cyr` (att_key_type=3 path)

- `att_key_type` must be 2 or 3 — anything else rejected at
  parse time (`tdx.cyr:182-184`).
- Signature/AK widths derived from `att_key_type` at parse,
  computed BEFORE any offset arithmetic (`tdx.cyr:187-191`).
- The minimum-size check `sig_min_fixed` is recomputed against
  the att_key_type-specific widths (`tdx.cyr:208-211`).

## Step 2 — Buffer safety

### `pem.cyr` base64 decoder

- `quad[4]` is a 4-byte local array (stack-resident scalar fits
  in registers — used as bytes via `&quad`). Writes bounded by
  `quad_n + 1 <= 4` (`pem.cyr:140-149`).
- Output writes bounded by `op + out_n <= out_max`
  (`pem.cyr:164`).
- `_pem_b64_table` is 256 bytes; reads indexed by `c` which is
  a single byte (`& 0xFF`), so always in-bounds
  (`pem.cyr:133`).
- Marker compare: `pos + len > buf_len` reject up front
  (`pem.cyr:103`), so the inner loop is safe.

### `*_verify_full` scratch arrays

- `chain[64]` = 4 entries × 16 bytes. Writes bounded by
  `count < max_certs (4)` (`sgx.cyr:437-439`, `tdx.cyr:396-398`).
- `parsed[4]` = 4 × 8-byte pointers. Writes bounded by `i < n`
  where `n <= max_certs = 4` (`sgx.cyr:444`, `tdx.cyr:403`).
- `inters[4]` = 4 × 8-byte pointers. Writes bounded by
  `inters_n < 4` (`sgx.cyr:471`, `tdx.cyr:430`).

### `tdx.cyr` verify scratch

- `_tdxv_binding_input` enlarged from 65600 to 65632 bytes to
  hold P-384 AK (96 B) + max auth_data (65535 B). Writes
  bounded by `i < ak_w` (with `ak_w` ∈ {64, 96}) and
  `i < aux_len` (with `aux_len <= 65536` guard) (`tdx.cyr:339-348`).
- `_tdxv_binding_hash` enlarged from 32 to 48 bytes to hold the
  full SHA-384 digest. The comparison loop reads only the lower
  32 B (`tdx.cyr:357-362`) — matches the on-wire report_data
  layout in both att_key_type=2 and =3 variants.

## Step 3 — Constant-time audit

### `pem.cyr`

The PEM decoder operates on **public** PEM bytes (X.509 cert
chains from the in-quote `qe_cert_data`, or VCEK chains fetched
from AMD KDS). There are no secrets in the input or output;
constant-time discipline does not apply. The output is
deterministic-time per byte (no early exits on success — only on
malformation).

### `*_verify_full`

The orchestrators don't introduce new comparisons on secret
data; they delegate to `x509_verify_chain`, `ecdsa_p256_verify`,
`ecdsa_p384_verify`, `sha256`, and `sha384` — all audited in
prior cycles. The `_x509_dn_eq` call inside
`_sgx_cert_is_self_issued` / `_tdx_cert_is_self_issued` operates
on **public** DN bytes; not constant-time, deliberately.

### `tdx.cyr` (P-384 path)

The AK binding hash comparison loop is the same shape used in
the P-256 path (32 bytes, byte-wise OR-accumulated by the
underlying ct_eq primitive). However, the in-place
`load8` + `!=` early-exit comparison at `tdx.cyr:357-362` is
**not** constant-time on the binding hash result. This is
intentional and matches the existing P-256 verify_with_pck path
— the binding hash itself is over PUBLIC data (AK pubkey +
qe_auth_data), so an early-exit timing leak reveals only public
bytes. Same conclusion documented in 3.2.3 audit Step 3 for the
SGX path.

## Step 4 — Key material zeroization

- `pem.cyr` handles only public PEM bytes. No secret material.
- `*_verify_full` doesn't introduce new secret state. The
  underlying `ecdsa_p256_verify` / `ecdsa_p384_verify` zeroize
  their per-call scalars (audited 3.2.1 / 3.2.4); the
  `_tdxv_binding_input` scratch is overwritten on every call
  with public inputs (AK + auth_data) — no secret residue.

## Step 5 — Syscall review

No new syscalls.

## Step 6 — Pointer validation

- All input pointers (`pem`, `root_ca_der`, `cert_data`) are
  treated as caller-owned read-only buffers with explicit
  lengths. Sigil reads exactly the documented widths.
- The `out_chain` entries store absolute pointers into the
  caller's `der_pool`. The caller is responsible for not
  reusing / freeing the pool while the chain pointers are live.
- `parsed[N]` and `inters[N]` arrays hold pointers to
  `x509_cert_alloc`-allocated structs. These persist for the
  lifetime of the process (bump allocator) — see LOW-1 below.

No `sys_system`, no `exec_*`. Static analysis: clean.

No filesystem access.

## Step 9 — Known CVE check

- **CVE-2022-0778 (BN_mod_sqrt)** — N/A: sigil doesn't do
  modular sqrt; ECDSA verify uses inversion only.
- **CVE-2020-25659 / asn1c-style heap overflow** — `_pem_b64_decode`
  bounds every write against `out_max`. `_pem_find` bounds every
  read against `buf_len + marker_len`.
- **PEM bomb (gigabyte-scale base64 → small DER)** — the input
  PEM length is bounded by `qe_cert_data_len` which `sgx.cyr` /
  `tdx.cyr` already validates against the parent quote size. A
  malicious attacker can at most consume `qe_cert_data_len`
  bytes of bump-allocator pool. Worst-case for current consumers:
  the quote's overall `buf_len` (typically < 64 KB).
- **Newline injection in PEM** — `_pem_find` finds the first
  matching marker byte sequence; subsequent markers are not
  re-scanned within a single block's body. Padding-with-`=`
  rules ensure that a BEGIN marker embedded in the base64 body
  cannot be misparsed as a new block: base64 alphabet doesn't
  include `-`, so the dash sequence in BEGIN/END markers is
  rejected as invalid base64.
- **Double BEGIN without END** — handled: `_pem_find` searches
  forward from `body_start` for END only. A second BEGIN before
  END would land in the base64 body, get rejected (dash not in
  alphabet) → return -1.

## Step 10 — Findings

### LOW-1 (carried forward) — `*_verify_full` bump-alloc drift

**Where**: `src/sgx.cyr:sgx_quote_verify_full`,
`src/tdx.cyr:tdx_quote_verify_full`.

**Note**: Each call allocates (a) a `pem_decode_certs` DER pool
sized to `cert_data_len` (~1.2 KB for our synthesised chain;
~3 KB for real Intel PCK chains), (b) up to 4 × 256-byte
`x509_cert` structs for the in-quote chain, (c) 4 × 64-byte
`raw_sig` heap buffers inside `x509_parse`, (d) 1 × 256-byte
`x509_cert` for the root, (e) 1 × 64-byte raw_sig for the root.
Per-call: ~5–8 KB bump-allocator drift.

**Action**: Roadmap 3.6 ships the unified `_into` API that
threads caller-provided scratch through `pem_decode_certs_into`,
a future `x509_parse_into`, and a `*_verify_full_into` wrapper.
For 3.4 the drift is documented but not exploitable — no
function reads back its own past scratch.

### LOW-2 (NEW) — `_pem_init` lookup-table allocation

**Where**: `src/pem.cyr:_pem_init`.

**Note**: First call to `pem_decode_certs` allocates a 256-byte
lookup table + two marker buffers (27 + 25 = 52 bytes) via the
bump allocator. Subsequent calls reuse them. Same pattern as
`_xp_init`, `_sgxv_init`, `_tdxv_init` — the LOW family
identified in the 3.2.x arc.

**Action**: Same closeout target as LOW-1 — 3.6's unified
`_into` API. The lookup table is **read-only after init** and
could equally lift to a true global with init-on-first-call;
this is the least-invasive shape and is preferred at 3.6 over
caller-scratch.

### INFO-1 — SEV-SNP verify_full deferred

**Where**: `src/sev_snp.cyr` — unchanged in 3.4.

**Note**: Roadmap 3.5 (now 3.5 again post-renumber, see roadmap
update) called for `snp_report_verify_full`. The implementation
is blocked on x509 P-384 SPKI extraction: the VCEK leaf cert
holds a 96-byte P-384 pubkey, which sigil's current x509 parser
(P-256-only) cannot extract.

**Action**: A future cycle — likely bundled with x509 P-384 SPKI
support and ARK/ASK signature-algorithm extensions — will close
this. Sigil's per-piece API (`snp_report_parse` +
`snp_report_verify` against a caller-supplied 96-byte VCEK
pubkey) remains the supported integration shape until then.

### INFO-2 — In-quote root cert treatment

**Where**: `src/sgx.cyr:_sgx_cert_is_self_issued`,
`src/tdx.cyr:_tdx_cert_is_self_issued`,
`src/sgx.cyr:sgx_quote_verify_full:466-469`,
`src/tdx.cyr:tdx_quote_verify_full:425-428`.

**Note**: Intel's DCAP PEM chain in `qe_cert_data` includes the
Intel SGX Root CA as the topmost cert. Sigil **silently drops**
this entry if it is self-issued (subject DN == issuer DN). The
chain walk then anchors on the CALLER-PROVIDED root_ca_der; the
in-quote root copy is never verified or compared against the
caller's anchor.

**Implication**: An attacker who can inject a self-issued cert
at the top of the in-quote chain doesn't gain anything — the
walk still anchors at the caller's external trust root. The DN
of the in-quote root is irrelevant to the verification outcome.

**Action**: None. This is the correct trust-establishment
posture. Document this in the function header comments (already
done).

### INFO-3 — Binding hash truncation under att_key_type=3

**Where**: `src/tdx.cyr:tdx_quote_verify_with_pck:354-356`.

**Note**: For att_key_type=3, the AK binding hash is SHA-384
(48 bytes) but the qe_report.report_data slot allocates only 64
bytes total, of which the upper 32 bytes are required-zero per
spec. The binding hash therefore truncates to the LOWER 32 bytes
of SHA-384 (the first 32 bytes of the digest, per FIPS 180-4
byte order).

**Implication**: Birthday-bound collision security for the
binding becomes 2^128 instead of 2^192 (full SHA-384). Still
exceeds the 2^128 collision-resistance bar; matches Intel TDX
quoting library v1.5+ spec. Documented in the module header.

**Action**: None — matches the on-wire spec.

### INFO-4 — Chain order assumption

**Where**: `src/sgx.cyr:sgx_quote_verify_full`,
`src/tdx.cyr:tdx_quote_verify_full`.

**Note**: Sigil assumes Intel DCAP chain order: `cert[0]` is the
PCK leaf, `cert[1..n-2]` are intermediates in leaf-towards-root
order, `cert[n-1]` may be the embedded root. This matches Intel
DCAP v3 and the TDX 1.0+ specs.

**Implication**: A chain delivered in any other order (e.g.,
root-first) would fail the issuer/subject DN match in
`x509_verify_chain` — i.e., would be rejected, not
mis-accepted. Safe behaviour.

**Action**: None.
