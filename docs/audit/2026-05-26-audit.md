## Step 1 — Input validation

`sgx_derive_seal_key`:
- `policy` must be exactly 0 or 1 (`SGX_SEAL_POLICY_MRSIGNER` or
  `_MRENCLAVE`). Anything else rejected (`src/seal.cyr:103-105`).
  An attacker who can influence the policy byte cannot smuggle in
  a value that affects the info string in unexpected ways.
- `isvsvn` range-checked to fit in u16 (`:106-107`) — matches the
  on-wire shape used by SGX reports / quotes for cross-checking.
- `key_id_len` range-checked: `< 0` rejected (defends against a
  signed-overflow propagating from an attacker-controlled u32
  somewhere upstream), `> SGX_SEAL_MAX_KEY_ID (256)` rejected
  (`:108-109`).

`sgx_seal_key` and `sgx_unseal_key` are pure aliases — they don't
add their own validation, but they delegate to
`sgx_derive_seal_key` which does.

## Step 2 — Buffer safety

One long-lived 291-byte scratch buffer (`_seal_info`, init-time
allocated via `_seal_init`). All writes are bounded:

- `store8(_seal_info + N, ...)` for N ∈ {0, 1, 2} — fixed
  offsets.
- The measurement copy loop is bounded `i < 32` (`:113-115`).
- The key_id copy loop is bounded `i < key_id_len`, where
  `key_id_len` is validated `≤ 256` upstream. Max write offset:
  `_seal_info + 35 + 255` = byte 290, within the 291-byte
  allocation. ✓

The 32-byte `out_key` is caller-allocated; sigil writes exactly
32 bytes via `hkdf` (audited at 2.9.x to honour its `out_len`
parameter).

## Step 3 — Constant-time audit

The KDF compose itself uses HMAC-SHA256 internally (via `hkdf`),
which is constant-time. The info-string construction is
straightforward `memcpy`-style writes with no branches on the
secret material. ✓

`sealing_root` itself is secret material (a CPU sealing key).
Sigil's HKDF implementation already handles this — see
`src/hkdf.cyr:183`: `secret var prk[32]` zeroizes the PRK on every
return path. The info string contains only public material
(policy, measurement, isvsvn, key_id) — no secret leakage via
the info buffer's residue after the call.

## Step 4 — Key material zeroization

- The `sealing_root` input is caller-owned; sigil does not
  retain a copy.
- The `prk` intermediate (inside `hkdf`) is `secret var` — zeroed
  on return.
- The `out_key` output buffer is caller-owned; if the caller
  wants to zero it after use, that's their responsibility.
- The `_seal_info` scratch buffer is **not** zeroed between
  calls. This is intentional and safe: info contains only
  public bytes (policy + measurement + isvsvn + key_id). A
  subsequent call overwrites the same bytes; even reading the
  residue would only reveal the previous caller's *public*
  measurement / ISVSVN / key_id, never any secret.

## Step 6 — Pointer validation

`sealing_root`, `measurement`, `key_id`, and `out_key` are all
caller-supplied. Sigil reads exactly the documented widths (32
B sealing_root, 32 B measurement, `key_id_len` bytes key_id) and
writes 32 B to out_key. No pointer arithmetic beyond
`base + bounded_index`.

## Step 9 — Known CVE check

- **CVE-2021-XXXX-style HKDF salt confusion**: sigil uses a
  fixed domain-separation salt (`SGX_SEAL_SALT` =
  `"sigil-sgx-seal-v1"`). The caller cannot override it. Same
  sealing root used in unrelated HKDF calls elsewhere cannot
  derive the same key as the seal surface. ✅
- **Cross-policy collision**: the `policy` byte is the FIRST
  byte of the info string. MRSIGNER-bound and MRENCLAVE-bound
  derivations with otherwise-identical inputs produce different
  keys (audit tests group "divergence"). ✅
- **Truncation collision (Klima-Pohan 2003)**: HKDF-SHA256 is
  not vulnerable to truncation collisions for distinct info
  strings since the underlying PRF is HMAC-SHA256. The info
  string's encoding (policy byte FIRST, fixed-position
  measurement, fixed-width ISVSVN, then key_id) is
  unambiguously parseable — no two distinct input sets produce
  the same byte sequence. ✅

## Step 10 — Findings

### INFO-1 — Sealing-root provenance is the caller's responsibility

**Where**: `src/seal.cyr` file header (`:7-13`) and the
`sgx_derive_seal_key` doc comment.

**Note**: Sigil cannot call SGX's `EGETKEY` directly — that
instruction is enclave-only and sigil runs outside the enclave
trust boundary. The caller obtains the 32-byte sealing root via
their runtime's enclave-side bridge (Gramine, Occlum, TDX
TDG_MR_REPORT, etc.) and hands it to sigil.

**Implication**: if the caller hands sigil a *bad* sealing root
(e.g. one derived under a different MRSIGNER than they intended,
or one mis-extracted from EGETKEY), sigil's derived key is
deterministically wrong but in a way sigil cannot detect.
Documented in the file header as a fundamental scope cut.

**Action**: none in sigil. The runtime that bridges EGETKEY to
sigil's seal_key is the load-bearing trust boundary; sigil's
audit recommends that runtime ships its own audit.

---

### INFO-2 — AEAD encryption is out of scope

**Where**: `src/seal.cyr` file header (`:21-24`).

**Note**: 3.2.6 derives a 32-byte key but does not encrypt /
decrypt anything. The "seal" / "unseal" semantic terminology in
the function names refers to the surrounding usage pattern: the
caller derives the key, then encrypts (seal) or decrypts (unseal)
their payload via `aes_gcm.cyr` (already in tree since 2.9.x).

**Rationale**: keeping the KDF surface narrow lets the caller
choose their own AEAD nonce scheme and AAD layout. Sigil
prescribing one would force every consumer into a single
serialisation choice that may not match their existing storage
format.

**Action**: none. If a future bite finds it useful to bundle a
default seal-and-encrypt wrapper, it slots cleanly on top of
this primitive.
