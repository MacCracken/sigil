## Load-bearing confirmations (the security positives)

The adversarial pass confirmed, with no dissent, the trust-boundary
properties of the change:

- **Leftmost-bits digest→scalar mapping is correct for all four
  hash×curve combos.** `bitlen(n_p256) = 256` and `bitlen(n_p384) = 384`
  are both whole-byte, so the spec's leftmost-*bits* selection equals a
  leftmost-*bytes* copy: P-256 + SHA-384 takes the leftmost 32 of the
  48-byte digest (MSBs); P-384 + SHA-256 uses the whole 32-byte digest
  left-zero-padded into 48. No sub-byte shift is required.
- **The single conditional subtract of n is provably exact** for every
  field-width `e` on both curves (`e < 2^(8·fw) < 2n` since
  `n > 2^(8·fw − 1)`), matching the pre-existing diagonal behavior.
- **Diagonal paths are byte-for-byte identical** to the pre-refactor
  code (the wrappers feed a field-width digest through the same
  arithmetic), and the **4-arg public verify signatures are preserved**,
  so all sgx/tdx/snp/dist callers are unchanged. Regression-checked: full
  suite 53 files / 1459 assertions, 0 failures.
- **No algorithm/curve-substitution false-accept.** The decoupled
  dispatch binds the verify primitive + signature width to the *issuer*
  curve and rejects a width mismatch before any signature bytes are
  consumed; the inner/outer sig-algo equality (RFC 5280 §4.1.1.2) and the
  SPKI→curve binding remain enforced.
- **Parse width tracks the issuer curve, not the hash** — all four combos
  store `sig_len` equal to the issuer-curve r,s width (64 or 96), the
  invariant the off-diagonal accept depends on. The
  ~2^-256 "P-384 signature whose r and s both fit in 32 bytes" edge is
  unreachable with honest signers and fail-closed (false-reject) if ever
  hit.
- **Buffer bounds + no aliasing**: the take/pad assembly loops write
  within the 32-/48-byte scalar buffers; the repurposed (previously dead)
  `_ecv_digest_padded` and the new `_ecv384_eb` / `_xvl_digest` scratch
  have no stale reader.

## Test corpus

External ground truth (OpenSSL), not self-referential:

- `ecdsa_p256.tcyr` (+4) / `ecdsa_p384.tcyr` (+4): off-diagonal primitive
  KATs — a P-256 key signing with SHA-384 and a P-384 key signing with
  SHA-256, each `openssl dgst -verify`-confirmed; accept + tampered-digest
  + tampered-signature rejects + a leftmost-bits truncation assertion
  (flipping a truncated-away tail byte must still verify).
- `x509_offdiag.tcyr` (new, +20): two real OpenSSL-issued off-diagonal
  cert chains (P-384 CA / SHA-256 leaf, and P-256 CA / SHA-384 leaf),
  each `openssl verify`-OK; link-verify, full-chain verify, the
  issuer-curve `sig_len` (96 / 64), tamper-reject, and cross-issuer
  width rejection.

## Conclusion

No CRITICAL / HIGH / MEDIUM / LOW findings outstanding. The off-diagonal
verify is a security-*positive* change (it closes a real-world chain
coverage gap) and introduces no false-accept path; the two in-cycle fixes
(dist regen, scratch reuse) were correctness/hygiene, not verify-logic
defects.
