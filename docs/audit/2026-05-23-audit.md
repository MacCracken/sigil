`buf_len` before any dereference.

## Step 1 — Input validation

`sgx_quote_parse(buf, buf_len, out)`:

- Minimum-size guard at the top (`SGX_MIN_QUOTE_SIZE = 1012` bytes
  — the smallest possible valid v3 DCAP quote with empty
  auth_data + empty cert_data). Anything smaller cannot possibly
  be a v3 quote (`src/sgx.cyr:191`).
- Version must equal 3 (`:196`). EPID quotes, future v4, and
  anything else rejected.
- `att_key_type` must equal 2 (ECDSA-P256). EPID (1) and ECDSA-
  P384 (3) rejected (`:197`).
- `signature_data_len` field bounds-checked twice:
  - `sig_data_off + sig_data_len > buf_len` → reject (`:209`)
  - `sig_data_len < sig_min_fixed` (fixed part of the signature
    section, 580 bytes) → reject (`:213`)
- `qe_auth_data_size` (u16) bounds-checked against the remaining
  signature section (`:240, :242`).
- `qe_cert_data_size` (u32) bounds-checked likewise (`:251, :253`).
- Strict-end check: `cur + 6 + cert_len != sig_section_end` →
  reject (`:255`). Any trailing bytes inside the signature section
  cause the parse to fail — defends against attacker-supplied
  hidden payloads tucked after the cert data.
- Signed-overflow defense: `u32` reads can produce values
  > 2^31-1 which are negative when stored as i64. We guard with
  `< 0` checks on `sig_data_len`, `auth_len`, `cert_len`
  (`:208, :241, :251`). Cyrius produces signed-integer overflow
  semantics for i64 + i64, so the negative guard is the right
  shape.

`sgx_quote_verify_with_pck(quote, pck_pk)`:

- Each of the three signature checks consults a known-fixed
  byte region of the quote (offsets pre-validated by
  `sgx_quote_parse`).
- `aux_len` (qe_auth_data length) is bounded by the parser, but
  we add a defense-in-depth `> 65536` check (`:296`) before the
  SHA-256 input assembly so a hostile parser-bypass would still
  fail safe.
- The required-zero padding `report_data[32:64]` is enforced
  (`:325-328`) — see INFO-2.

## Step 2 — Buffer safety

`sgx.cyr` allocates:
- One 256-byte `SgxQuote` struct per parsed quote (`sgx_quote_alloc`).
- Three long-lived scratch buffers initialised once
  (`_sgxv_init`): 65600 B for SHA-256 input assembly (sized for
  the worst-case u16 auth_data length plus AK), 32 B for the
  resulting hash, 432 B for the quote-body sig input. No
  per-call allocation in the verify path.

All offsets stored in the struct are produced by
`sgx_quote_parse` AFTER bounds checks; the verify path can
treat them as trusted. Writes to scratch buffers are bounded by
parser-validated lengths.

## Step 3 — Constant-time audit

Quote parsing is **not** constant-time and does not need to be
— the entire quote, including auth data and AK, is public. The
verify orchestrator's final-step byte comparisons on
`report_data` are early-exit (`:319-323`), but this comparison
target is public material (the QE produces it; an attacker
forging it would need to also forge the corresponding signature).

No secret-data branch is introduced. The composed
`ecdsa_p256_verify` inherits 3.2.1's constant-time discipline
for the scalar-mul substrate.

No secret material handled by `src/sgx.cyr`.

## Step 6 — Pointer validation

The caller passes the original `buf` and its `buf_len`. The
parser stores `buf` and the validated offsets in the struct;
verifiers and accessors derive absolute pointers as
`buf + off` and read at most `len` bytes after the offset —
all `(off, len)` pairs were bounds-checked at parse time.

`pck_pk` is a 64-byte buffer the caller must provide. Per the
3.2.3 scope, the caller has already validated that this key is
on-curve and chain-anchored to Intel's SGX Root CA — sigil does
not re-check ownership here (see INFO-1). If a caller passes a
garbage pointer, the resulting `ecdsa_p256_verify` call will
fail (the buffer either has on-curve bytes, in which case the
sig check fails, or off-curve bytes, in which case the on-curve
check inside `ecdsa_p256_verify` fails). Either way no memory
corruption results.

## Step 9 — Known CVE check

- **CVE-2017-5715 / Spectre v2** (relevant to SGX as a whole):
  out of scope for sigil — sigil verifies attestation, it does
  not run inside an enclave.
- **Quoting Enclave forgery**: defended by the three-step
  verify chain. An attacker forging a quote needs to either:
  - Forge the QE report signature → requires the PCK private key
  - Forge the AK binding → requires colliding SHA-256 (infeasible)
  - Forge the body signature → requires the AK private key
  None of these are achievable without breaking either ECDSA or
  the hash. ✅
- **Length-confusion / TOCTOU on signature_data_len**: the parse
  is single-pass and stores all bounds in the struct before any
  verify step reads them. No re-read of the wire bytes during
  verification → no TOCTOU. ✅
- **CVE-2024-39686 (Linux SGX kernel UAF) / CVE-2024-37281
  (Intel SGX driver buffer issues)**: kernel-side bugs in Intel
  drivers, not in attestation-verifier code. Out of scope.
- **Substitution attack via cert_data placeholder**: the 3.2.3
  parser exposes `cert_data_type` and `cert_data_ptr/_len` but
  does NOT parse the embedded PEM. The caller decides what to
  do with it. **INFO-1** tracks the deferred PEM-decoder work
  that closes this gap end-to-end.

## Step 10 — Findings

### INFO-1 — `sgx_quote_verify_with_pck` requires external chain walk

**Where**: `src/sgx.cyr:286-352` (`sgx_quote_verify_with_pck`)
and the surface API note in the file header.

**Note**: 3.2.3's verify orchestrator covers the three SIG checks
*inside* the quote (PCK→QE-report, AK-binding-via-SHA256, AK→body)
but does NOT walk the PCK certificate chain from the leaf PCK
cert (in `qe_cert_data`) up to Intel's SGX Root CA. The caller
is responsible for:

1. Decoding `qe_cert_data` (typically PEM type 5) into individual
   DER-encoded certificates.
2. Parsing each via `x509_parse`.
3. Walking the chain via `x509_verify_chain` to a known-trusted
   Intel SGX Root.
4. Extracting the leaf PCK cert's pubkey and passing it as
   `pck_pk` to `sgx_quote_verify_with_pck`.

This is documented at the top of `src/sgx.cyr` (lines 16-26).
The PEM decoder + integrated chain walk is the next bite in
this arc's roadmap (likely 3.2.3+ patch — not yet ticketed
since neither 3.2.4 nor 3.2.5 strictly need it before they ship,
both formats also externalise their cert chains).

**Action**: when an integration consumer (kavach in particular)
surfaces, ship the PEM decoder + an `sgx_quote_verify_full(quote,
intel_root)` wrapper that internalises the chain walk. Until
then, the per-piece API is the supported surface.

### INFO-2 — Required zero padding on `report_data[32:64]`

**Where**: `src/sgx.cyr:325-328`.

**Note**: Intel's spec for the QE's report_data when used to
bind an attestation key uses the lower 32 bytes as the
SHA-256(AK || qe_auth_data) hash. The upper 32 bytes are
specified as zero. Sigil enforces this — any non-zero byte in
`report_data[32:64]` causes verify to reject.

**Rationale**: defense in depth. A real Intel QE will always
zero this region; enforcing the zero-padding catches a
quoting-enclave variant or attacker who tries to stuff
additional bits there (which could otherwise be silently
ignored by sigil but interpreted differently by a downstream
consumer parsing the full 64-byte field).

**Action**: none. Documented as a deliberate strictness
choice in the source comment.
