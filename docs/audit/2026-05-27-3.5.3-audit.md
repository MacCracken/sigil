## 10-step Security Hardening checklist

1. **Input validation.** Fixed 32-byte scalar / u-coordinate /
   output by contract. The scalar is clamped (RFC 7748 §5: clear
   low 3 bits, clear bit 255, set bit 254); the u-coordinate has
   its MSB masked. Non-canonical u ≥ p is accepted and reduced by
   the `fp_*` ops, per RFC.
2. **Buffer safety.** All ladder state is in one 576-byte block
   `W` (18 × 32-byte slots); the named pointer slices reach at most
   `&W + 544` + 32 = 576 — in bounds. `ub`/`base` are 32 bytes;
   `_x25519_cswap` indexes 4 limbs (`i*8`, i < 4 → offset ≤ 24);
   `out` is 32 bytes. No overrun.
3. **Constant-time.** The ladder runs a fixed 255 iterations
   regardless of scalar value; the per-bit `cswap` is branch-free
   (arithmetic mask `0 - swap`), so the field-operation sequence is
   independent of secret scalar bits — the core X25519
   constant-time requirement. The bit extraction reads every bit
   position unconditionally. See INFO-2 for the field-reduction
   branches.
4. **Key material zeroization.** The clamped scalar `k` and the
   entire ladder state (projective coords + temporaries) share the
   single `secret var W` block, compiler-zeroized on return.
   `ub`/`base`/`a24` (public u-coordinates / curve constant) are
   plain `var`.
5. **Syscall review.** None.
6. **Pointer validation.** No untrusted deref; fixed offsets into
   `W` and fixed-size buffers.
7. **No command injection.** None.
8. **No path traversal.** None.
9. **Known-CVE check.** (a) **Scalar-bit timing** (the classic
   X25519 break) — avoided via the masked `cswap` ladder. (b)
   **Clamping errors** (off-curve / cofactor) — clamp matches RFC
   7748 byte-for-byte; verified by §5.2/§6.1 vectors. (c)
   **Small-subgroup / all-zero shared secret** — see INFO-1. (d)
   **Non-canonical u** — MSB masked per RFC; field ops reduce mod
   p.
10. **Findings filed** — below.

## Findings

### INFO-1 — All-zero shared secret is not rejected

RFC 7748 §6.1 notes that X25519 yields an all-zero output exactly
when the input is a small-order point, and that protocols **MAY**
check for and abort on the all-zero value (it indicates a
non-contributory / small-subgroup input). `x25519` returns the raw
result without this check — the deliberate choice for a primitive
library: the protocol layer (e.g. TLS 1.3, which already mandates
the check) decides whether all-zero is fatal. Documented as a
caller obligation. No code change; revisit if a consumer wants a
built-in contributory-agreement mode.

### INFO-2 — Field-reduction branches inherit the ed25519 stance

`fp_add` / `fp_sub` / `u512_mod_p` (in `src/bigint_ext.cyr`) end
with data-dependent conditional subtractions of `p` (a bounded
1–2 iterations). These operate on field-element values, not
directly on scalar bits, and are the same primitives Ed25519 has
shipped since 3.0.x — the established, accepted constant-time
level for sigil's single-tenant threat model. X25519 adds no new
secret-dependent branch of its own. Not a new finding; recorded
for completeness.

### INFO-3 — Serial-only static scratch

`W`, `ub`, `base` are function-scope arrays (Cyrius static
globals, quirk #1); each call reinitializes `W` and writes the
others before use, so serial use is correct. Concurrent callers
would race. Not yet on a multi-threaded path (v3.6
per-worker-scratch cycle). The single-block `secret var W` shape
also anticipates that cycle. Upstream cap-raise filed:
`cyrius/docs/development/issues/2026-05-27-secret-defer-block-per-fn-cap.md`.
