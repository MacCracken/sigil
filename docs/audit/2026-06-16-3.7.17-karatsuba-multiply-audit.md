# Sigil Security Audit — 2026-06-16 (3.7.17 Karatsuba u256_mul_full + u2·Q-window mixed add)

**Auditor**: Internal (Claude/Opus) + a 5-lens adversarial carry-propagation review workflow
**Version audited**: 3.7.17
**Cyrius toolchain**: 6.2.12; deps agnosys 1.4.3, sakshi 2.3.0
**Prior audit**: `2026-06-16-3.7.16-ec-inversion-mixedadd-audit.md`
**Scope**: The P-256 verify EC-squeeze levers 2b + 3:
- **Lever 3 — Karatsuba `u256_mul_full`** (`src/bigint_ext.cyr`): the 256×256→512
  multiply under **all of sigil's 256-bit ECC** — Ed25519, X25519, ECDSA P-256 —
  replaced from schoolbook (16 × 64×64) to Karatsuba (12 × 64×64 via the
  `z1 = (aL+aH)(bL+bH) − z0 − z2` identity). The schoolbook is retained as
  `_u256_mul_full_schoolbook` (KAT oracle). A new stateless `_u256k_mul128` helper
  does the 128×128 sub-products.
- **Lever 2b — `u2·Q`-window mixed add** (`src/ecdsa_p256.cyr`): affine Q-table via
  Montgomery batch inversion + `pt_add_mixed` (shipped marginal; see 3.7.16 audit
  for the mixed-add formula review).

**Method**: A 5-lens adversarial carry-propagation review (accumulator-overflow,
middle-corrections, borrow-subtraction, final-shifted-add, formula-derivation),
each lens building a **bit-exact emulation** of the cyrius code and differentially
testing it against arbitrary-precision multiply, plus the in-tree differential KAT.

## Correctness — conclusive

The crown-jewel-multiply risk (a carry bug = a wrong product = **signature
mis-accept across Ed25519 *and* ECDSA**) is closed by overlapping, mutually
reinforcing evidence:

1. **In-tree differential KAT** (`ecdsa_p256.tcyr`): `u256_mul_full` ==
   `_u256_mul_full_schoolbook` over 200 pseudo-random pairs + **14 carry-boundary
   / adversarial edges** (all-ones halves → `aL+aH` overflow, `2^128` exact
   boundary, every-limb-`2^63`, alternating `0xAA`/`0x55`, max × `2^255`,
   `2^128−2` halves). +2 assertions.
2. **Full real-vector suite (1481 assertions, 0 failures)**: Ed25519 (RFC 8032),
   X25519 (RFC 7748), ECDSA P-256 (RFC 6979), x509, SGX/TDX/SNP — every path that
   multiplies 256-bit field/scalar values exercises the Karatsuba against known
   answers and passes.
3. **Thread-safety**: the concurrent `batch_parallel.tcyr` path is green. The
   Karatsuba uses **only scalar locals + the caller's `out` + the stateless
   `_u256k_mul128`** (no static scratch), so it is race-free on the Ed25519
   parallel batch-verify path (the constraint that forced the inline-scalar design).
4. **5-lens adversarial review**: every lens returned "looks-correct," each backed
   by a bit-exact Python emulation differentially tested against arbitrary-precision
   `a*b` over **millions of cases** (1M–3M+ per lens) + exhaustive carry corners,
   plus structural proofs: `cs`/`ct` ∈ {0,1} (the second half-sum add cannot
   overflow once the first did); column carries `cm`/`cm2` reach their max and are
   captured by independent binary overflow tests; `m4` base is always 0; the borrow
   `br` can never reach 2 (the two borrow sources are mutually exclusive); `z1 ≥ 0`
   is structurally guaranteed (`m ≥ z0 + z2`); `out[7]` can never overflow (the
   result is `< 2^512`). One lens **validated its own harness** by injecting the
   exact off-by-one (z1 placed one limb too far) and confirming the harness catches
   it — yet the real code matched every time. **Zero discrepancies across all
   lenses.**

## INFO — pre-existing constant-time posture (unchanged)

`u256_mul_full` has operand-dependent carry branches (`if (cs == 1)`, `if (ct ==
1)`, and the per-limb `if ((x ^ sbit) < (y ^ sbit))` carry tests). This is the
**same class** of operand-dependent branching the schoolbook it replaces already
had (the schoolbook branches on `mid_carry` / `lo_carry` / column carries). So
sigil's 256-bit field multiply is **not strictly constant-time, and was not before
this change** — Karatsuba introduces **no new side-channel class**. The verify
paths consume only public data; the secret-nonce *point*-scalar-mul for signing
stays on the constant-time Montgomery ladder `pt_scalarmul` (no secret-dependent
*point* branches). Making the underlying field multiply strictly branchless for the
signing path is a separate, pre-existing consideration (surfaced here, not buried —
candidate backlog item if the threat model adds local timing attacks on signing).

## Verdict

The 3.7.17 Karatsuba multiply is **correct (conclusively, across KAT + real-vector
suite + a millions-of-cases adversarial review) and thread-safe**, with no new
side-channel class. It is a modest ~3–4% speedup across all 256-bit ECC (Ed25519
verify 6.90 → 6.63 ms, P-256 verify ~11.58 → 11.25 ms). The ≤ 10 ms P-256 target
is **not** reached (~10.9 ms floor — at 256-bit, fast hardware multiply caps
Karatsuba's gain); the known EC-squeeze levers are now exhausted. Audit floor stays
empty.
