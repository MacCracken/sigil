## Scope: what changed in 3.5.7

### `src/aes_gcm.cyr`

- **New** `aes_128_key_expand(key, round_keys)` — FIPS 197 §5.2 with
  Nk=4 / Nr=10, producing a 176-byte / 11-key schedule. Distinct
  from the AES-256 schedule (Nk=8 has an extra `i%4==0` SubWord-only
  branch that AES-128 does not need).
- **Refactor** the software block transform into
  `_aes_encrypt_block_sw(round_keys, in, out, nr)`; `aes256_encrypt_block`
  and the new `aes128_encrypt_block` are thin NI-dispatch wrappers
  fixing `nr`.
- **Refactor** `_aes_gcm_ctr_xor` and the GCM encrypt/decrypt drivers
  to take `nr`; `rk_size = 16 * (nr + 1)`. Public `aes_gcm_encrypt` /
  `aes_gcm_decrypt` keep their exact signatures and now delegate to
  `_aes_gcm_{encrypt,decrypt}_impl(…, 14)`; the new
  `aes_128_gcm_encrypt` / `aes_128_gcm_decrypt` delegate with `nr=10`.

### `src/aes_ni.cyr`

- **New** `aes128_encrypt_block_ni` — 9× AESENC + 1× AESENCLAST over
  round-key offsets 0x00..0xA0, same prologue/parameter shape and
  16-byte-alignment requirement as the AES-256 variant.
- **New** `_aes128_ni_self_test()` against FIPS 197 §C.1; wired into
  `aes_ni_available()` so a divergence in **either** the §C.3 or §C.1
  NI vector pins the cache to software for both key sizes.

## Checklist findings

1. **Input validation** — `_aes_gcm_{encrypt,decrypt}_impl` reject
   `aad_len < 0` / `pt_len|ct_len < 0` before any allocation, same as
   the AES-256 path. Key/IV lengths are caller-contract (16/12 bytes
   for AES-128); the GCM 96-bit-IV fast path is unchanged. **Clean.**
2. **Buffer safety** — `rk_size = 16*(nr+1)` is 176 (nr=10) or 240
   (nr=14). `fl_alloc(rk_size)` and the matching
   `memset(round_keys, 0, rk_size)` before `fl_free` use the same
   `rk_size`, so the AES-128 path allocates, writes (key expand writes
   words 0..43 → bytes 0..175), and zeroizes exactly 176 bytes. The
   key schedule's max write offset is word 43 → `dst_base = 172`,
   bytes 172..175 < 176. AES-NI globals `_aes_ni_st_rk[240]` hold the
   176-byte AES-128 schedule with room to spare; the self-test zeroes
   only the first 176. All block buffers stay 16 bytes regardless of
   key size. **Clean.**
3. **Constant-time** — no new comparisons. The tag compare remains
   `ct_eq_bytes_lens` (OR-accumulating, no early exit). The only new
   branches are on the **public** round count `nr` (in `_aes_keyexp`,
   `_aes_block_enc`, and the `nr`-bounded loop in
   `_aes_encrypt_block_sw`) and the public `key_len`-independent NI
   availability/alignment checks — none branch on key or plaintext
   bytes. **Clean.** (INFO: `nr` is a compile-fixed public constant
   per call site, not attacker-influenced.)
4. **Key zeroization** — `aes_128_*` go through the identical
   `secret var h/y/s/expected_tag/ks` buffers (compiler-zeroized on
   every return) plus the explicit `memset(round_keys, 0, rk_size)`
   before free. Error-return paths (negative lengths) hit the same
   `secret var` scope-exit zeroization. **Clean.**
5. **Syscall review** — no syscalls added. **N/A.**
6. **Pointer validation** — callers supply `round_keys`, `in`, `out`,
   `key`, `iv`, `ct_out`, `tag_out` per the documented contract;
   no new untrusted dereference. The AES-NI path gates on
   `(round_keys & 15) == 0` before issuing m128 memory operands
   (#GP guard), unchanged from AES-256. **Clean.**
7. **No command injection** — none. **N/A.**
8. **No path traversal** — none. **N/A.**
9. **Known-CVE check (AES-GCM)** — the classic AES-GCM pitfalls:
   - *IV reuse / 96-bit-IV handling* — unchanged from AES-256; same
     J0 = IV‖0x00000001 fast path. AES-128 inherits the same
     never-reuse-(key,IV) contract (documented on the public fns).
   - *Non-constant-time tag compare* — uses `ct_eq_bytes_lens`.
   - *Plaintext release before auth* — decrypt computes/verifies the
     tag first and zeroes `pt_out` on mismatch (no-leak contract),
     verified by the tag-flip test.
   - *GHASH/key-schedule table-driven timing* — the AES-128 key
     schedule uses the same data-independent S-box-lookup structure
     as AES-256 (single-tenant threat model; scatter-store remains a
     backlog item for the multi-tenant case, unchanged).
   No AES-128-specific CVE class introduced. **Clean.**
10. **Findings filed** — this document. Zero CRITICAL/HIGH/MEDIUM/LOW.

## Correctness evidence

- `tests/tcyr/aes128_gcm.tcyr` (+15 assertions): canonical GCM
  AES-128 Test Cases 1–4 (empty, single-block, 4-block no-AAD, AAD +
  60-byte partial block), decrypt roundtrip, one-bit tag-flip
  rejection + `pt_out`-zeroed checks, empty-message roundtrip. All
  pass.

  AES-256 regression from the parametrization refactor.

  gates the NI path live, so the AES-128 KAT validates the
  `aes128_encrypt_block_ni` opcode sequence end-to-end (NI and
  software produce byte-identical output, or the §C.1 self-test would
  have downgraded to software).
- Full suite: 45 files / 1231 assertions, 0 failures.
