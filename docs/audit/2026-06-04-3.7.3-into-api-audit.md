## Audit-floor reclassification (the core deliverable)

The alloc-site map showed the eight bump-allocator-lifetime LOWs split
cleanly:

**Genuine per-call drift (4) — resolved via `_into`.** Each ran the
global bump allocator on every verification:
- 3.2.2 LOW-1: `x509_parse` ECDSA `raw_sig` (≤96 B/cert)
- 3.6.5 LOW-1: `_xp_parse_spki` RSA SPKI side block (544 B/RSA-cert)
- 3.4.0 LOW-1: `sgx`/`tdx_quote_verify_full` (cert structs + sig blocks
  + PEM pool, per call)
- 3.4.1 LOW-1: `snp_report_verify_full` (same shape)

The `_into` variants route all of these through a caller arena; a
looping consumer `arena_reset`s between calls and never grows the global
heap. The one-shot bump wrappers remain (documented for one-shot use).

**Init-once singletons (4) — reclassified as correct, not drift.** All
are guarded (`if (_x_inited == 1) return 0`) and allocate once for the
process — exactly the "alloc() for init-once tables" pattern CLAUDE.md
quirk #2 endorses:
- 3.2.4 LOW-1: `_snp_v_init`; 3.4.0 LOW-2: `_pem_init` tables;
  3.2.3 / 3.2.5: `_sgxv_init` (65 KB) / `_tdxv_init` (66 KB).

They were over-conservatively flagged and never drifted; churning them
onto the freelist would add risk for no benefit.

## Regression (arena == 0 byte-identical)

`x509_parse` / `x509_cert_alloc` / `sgx`/`tdx`/`snp` `*_verify_full` are
now thin `arena == 0` wrappers; `_x509_scratch(0, n) == alloc(n)`
exactly, so the one-shot path is byte-for-byte unchanged. All pre-3.7.3
public signatures are preserved (API-additive). The full suite (52/52,
incl. the three `*_verify_full` bump-path tests) confirms it.

## Arena safety

Every arena allocation is checked for exhaustion (`arena_alloc` returns
0): `x509_cert_alloc_into`, `_x509_scratch` (raw_sig / RSA block),
`der_pool`, and `root` all reject 0 and fail closed (no null deref).
Lifetime: the arena outlives the whole `x509_verify_chain` walk (the
orchestrator holds it; no mid-verify `arena_reset`/`arena_free`).
Aliasing: `der_pool`, cert structs, and sig blocks each get a distinct
region (the stdlib `arena_alloc` bumps an 8-byte-aligned cursor) — the
DER the certs point into is never overwritten by a later cert-struct
allocation. `cert_data_len` / `vcek_chain_pem_len` is a safe upper bound
for the DER pool (base64 shrinks 4→3).

## Drift-completeness — finding fixed in-cycle (MEDIUM + LOW)

The review correctly observed that the original no-drift test asserted
only `arena_used` constancy, which **cannot** witness a residual
*global*-bump alloc (`arena_used` measures the arena; the global heap is
`alloc_used()`). Two responses, both **landed in this release**:

1. **Code-level proof**: every allocation reachable from the `_into`
   path was traced — `der_pool` (arena), each `x509_cert_alloc_into`
   (arena), each `x509_parse_into` raw_sig/RSA block (arena),
   `pem_decode_certs_into` (writes into `der_pool`, no alloc),
   `x509_verify_chain` / `_x509_verify_link` / the ECDSA/RSA verify
   primitives (static/banked scratch, no per-call alloc). **No residual
   global `alloc()` exists** on the path.
2. **Empirical proof**: the tests now capture `alloc_used()` before a
   50-iteration `arena_reset`+verify/parse loop and assert the delta is
   **0** — across the SGX orchestrator path (`sgx_verify_full.tcyr`),
   the RSA 544-byte block (`x509_rsa.tcyr`), and the P-384 raw_sig
   (`x509_p384.tcyr`). All pass, directly proving no global-heap drift
   and closing the SGX-test coverage gap for the RSA / P-384 allocations
   the original test never exercised.

## Disposition

3.7.3 ships clean — **0 outstanding findings, audit floor empty**. The
`_into` API gives looping consumers (kavach's attestation flow) a
provably drift-free path; the one-shot wrappers are unchanged. The
review's two test-assurance findings were fixed in-cycle, upgrading the
no-drift guarantee from asserted to empirically witnessed.
