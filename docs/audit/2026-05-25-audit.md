The 3.2.5 surface (~290 lines) is a TDX-shaped clone of the
3.2.3 SGX parser + verify pattern. The signature section format
is byte-identical to SGX (because the TDX QE runs on Intel SGX
hardware), so the entire SGX audit (`2026-05-23-audit.md`)
applies modulo the renamed function symbols.

What's TDX-specific:
1. Header validation (version=4, tee_type=0x81, att_key_type=2).
2. The 584-byte TD report body layout (MRTD / MRSEAM /
   RTMR0..3 / REPORT_DATA at well-defined offsets).
3. The signed body is 632 bytes (header + TD body) instead of
   SGX's 432 (header + enclave report).

## Step 1 — Input validation

`tdx_quote_parse`:
- Minimum-size check via `TDX_MIN_QUOTE_SIZE = 1148` bytes
  (`:159`). Anything smaller cannot possibly be a valid v4
  quote.
- `version != 4` rejected (`:166`). SGX v3 quotes won't be
  mis-parsed as TDX.
- `att_key_type != 2` rejected (`:167`). Only ECDSA P-256 /
  SHA-256 is accepted in 3.2.5 — see INFO-1.
- `tee_type != 0x81` rejected (`:168`). Guards against an SGX
  enclave-report being mis-fed (which has tee_type field
  positioned identically but as the `reserved` zero word).
- `sig_data_len` bounds-checked twice: against `buf_len`
  (`:181`) and against the fixed signature-section minimum
  (`:184-187`).
- `qe_auth_data_size` (u16) and `qe_cert_data_size` (u32)
  bounds-checked against the remaining signature section.
- Strict-end check: `cur + 6 + cert_len != sig_section_end` →
  reject (`:218`). Defends against attacker-supplied trailing
  payloads.

## Step 2 — Buffer safety

Same posture as 3.2.3:
- One 256-byte `TdxQuote` struct per parsed quote.
- Three long-lived scratch buffers initialised once via
  `_tdxv_init`: 65600 B for SHA-256 input assembly (auth_data
  worst case), 32 B for the binding hash, 632 B for the body-
  sig input.
- All offsets stored after parse-time bounds checks; verify
  treats them as trusted.
- `aux_len > 65536` guard at `:265` adds a defense-in-depth
  cap on the SHA-256 input even though `tdx_quote_parse`
  already gates `aux_len` against the buffer.

## Step 9 — Known CVE check

- **Quote-format confusion (SGX ↔ TDX)**: the parser enforces
  three independent header invariants (version, tee_type,
  att_key_type) — an SGX quote fed to `tdx_quote_parse` fails
  on both version and tee_type. ✅
- **Algorithm confusion**: only ECDSA P-256 / SHA-256 accepted;
  any other att_key_type rejected. ✅
- **Length-confusion on the variable signature section**: same
  bounds-check shape as 3.2.3, audited there.
- **QE substitution**: the three-step verify chain (PCK→QE,
  AK binding, AK→body) requires breaking ECDSA or SHA-256 to
  forge. ✅

## Step 10 — Findings

### INFO-1 — P-384 (att_key_type = 3) deferred

**Where**: `tdx_quote_parse` (`src/tdx.cyr:167`).

**Note**: 3.2.5 supports only `att_key_type = 2` (ECDSA P-256 /
SHA-256). The P-384 / SHA-384 variant (`att_key_type = 3`) used
by some TDX 1.5 deployments is structurally identical except
for: (a) 96-byte AK and ecdsa_sig fields instead of 64-byte,
(b) different `ecdsa_p384_verify` call, (c) SHA-384 binding
hash output is 48 bytes — but the binding only consumes the
lower 32, so still aligns with `report_data[0:32]`. The 3.2.4
P-384 primitive landed alongside SEV-SNP so this is a small
delta when surfaced.

**Action**: open as a 3.2.x follow-up when an integration
target needs P-384 TDX quotes. The 3.2.5 parser correctly
rejects `att_key_type = 3` rather than silently treating it as
P-256.

---

### INFO-2 — PCK chain walk caller-driven (consistent with 3.2.3 / 3.2.4)

**Where**: `src/tdx.cyr` file header (`:18-23`).

**Note**: TDX shares the SGX PCK cert chain (same Intel SGX
Root → PCK Platform CA → PCK Cert), so the chain walk is
caller-driven via the existing `x509_*` surface from 3.2.2 —
exact same shape as 3.2.3. PEM decoding of in-quote
`qe_cert_data` (type 5) is the same future patch already
tracked at 3.2.3's INFO-1.

**Action**: rolls into the existing PEM-decoder follow-up
already tracked.
