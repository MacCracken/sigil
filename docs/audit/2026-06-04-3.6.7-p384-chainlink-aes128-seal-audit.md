## x509 P-384 chain-link

### Correctness
Real P-384 CA (secp384r1, self-signed `ecdsa-with-SHA384`) signing a
P-384 leaf with `ecdsa-with-SHA384`, OpenSSL-cross-verified, verifies
green (`x509_p384.tcyr` Group 5); tamper rejects; `ecdsa_p384_verify`
SHA-384-hashes the raw TBS internally and consumes the 96-byte
(Qx‖Qy) issuer pubkey directly from the cert slot.

### Algorithm/curve confusion (reviewed, clean)
`_x509_verify_link` dispatches strictly on the child's `sig_algo`:
ECDSA-SHA256 → issuer must be P-256; **ECDSA-SHA384 → issuer must be
P-384** (`sig_len==96` enforced before `ecdsa_p384_verify` reads
`sig+48`); RSA-SHA256/384 → issuer must be RSA. No cross-acceptance: a
SHA-256-signed child against a P-384 issuer is still rejected
(`x509_p384.tcyr` Group 4 regression). Inner/outer signature algorithms
must still match (RFC 5280 §4.1.1.2).

### `_ecdsa_der_int_w` refactor (reviewed, behavior-preserving)
The fw=32 wrapper reproduces the original P-256 constants exactly
(`raw_len > fw+1`, leading-0x00 strip at `raw_len == fw+1`, `int_len >
fw`, `memset(dst, fw)`); fw=48 applies the same logic to P-384. All
prior callers (`ecdsa_p256_verify_der`, the x509 P-256 path) are
unchanged — confirmed by the green P-256 suite + the TEE `*_verify_full`
chains.

### LOW-A — ECDSA `sb_np` clobber (found by the review; FIXED in 3.6.7)
The ECDSA signature-value parse walked the inner `SEQUENCE { r, s }`
into `sb_np` (the BIT STRING end-position out-param), so the final
cert-length gate `sb_np != cert_cp + cert_seq_len` validated the inner
SEQUENCE end instead of the BIT STRING end. A crafted cert could
over-declare the signatureValue BIT STRING length and smuggle stray
bytes inside the accepted cert buffer past the signature.
- **Pre-existing**: the P-256 branch had the identical reuse; the 3.6.7
  P-384 work copied the pattern. The review caught it via a runnable
  probe (over-long BIT STRING + 2 trailing bytes → wrongly accepted).
- **Not a forgery**: `r`/`s` are still parsed from the genuine inner
  SEQUENCE (`s_next2 == inner_seq_end` enforced) and the signature is
  verified normally; impact was acceptance of non-canonical DER + admitted
  trailing bytes.
- **Fix**: walk the inner SEQUENCE into a distinct `inner_seq_np`,
  preserving `sb_np` for the cert-length gate, **and** assert
  `inner_seq_end == sb_np` (the inner SEQUENCE must consume the whole
  BIT STRING content). Closes both P-256 and P-384. All valid ECDSA
  certs (P-256, P-384, SGX/TDX/SNP chains) still verify.

## AES-128 seal keys

### Back-compat (reviewed, byte-identical)
`sgx_derive_seal_key` is now a wrapper calling `_n(..., 32)`; the info
layout and HKDF call are unchanged, so the 256-bit derivation is
byte-for-byte identical to pre-3.6.7 — confirmed by `seal.tcyr`'s
direct-HKDF reference KAT (still green) and an explicit "7-arg wrapper
== _n(...,32)" assertion.

### Validation + buffer safety
`out_key_len` is checked against `{16, 32}` **before** any HKDF write;
all callers size `out_key` to match. The shared `_seal_info` scratch
holds the (public) measurement + key_id, never the derived key — the
derived key goes only to the caller's buffer (unchanged posture).

### INFO-1 — HKDF prefix relationship
A 16-byte key and a 32-byte key derived from the **same** context are
prefix-related (HKDF OKM is a prefix family); the key length is not
bound into the KDF info because that layout is frozen for back-compat
(binding it would break previously-sealed data). Documented in
`seal.cyr`: a caller needing both widths should use a distinct `key_id`
per (purpose, width) — which it already should per purpose. Not a
practical weakness for the seal use case.

## Disposition

3.6.7 ships clean — **0 outstanding findings**. The adversarial review's
real finding (a latent pre-3.6.7 DER-strictness bug on the shared ECDSA
parse path) was fixed in-cycle, hardening both EC curves. Audit floor
unchanged at 8 LOW (no new per-call allocations: the P-384 raw_sig
alloc is the same per-parse pattern already counted under 3.2.2 LOW-1;
AES-128 seal adds no allocation).
