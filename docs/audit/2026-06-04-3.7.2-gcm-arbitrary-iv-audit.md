## J0 correctness (SP 800-38D §7.1)

- **96-bit fast path** unchanged: `J0 = IV ‖ 0x00000001`. The 12-byte
  consistency test asserts `aes_gcm_encrypt_iv(…, 12, …)` is
  byte-for-byte identical to `aes_gcm_encrypt(…)`, and the existing
  12-byte GCM KATs (`aes128_gcm`/`aes_gcm` 15/15) are unchanged.
- **Arbitrary length**: `J0 = GHASH_H(IV ‖ 0^s ‖ (0⁶⁴ ‖ [len(IV)·8]₆₄))`.
  `_ghash_update` zero-pads the final partial IV block (so the `IV ‖
  0^s` framing is exact), then the length block (bytes 0–7 zero, bytes
  8–15 = `_be64_store(iv_len*8)`) is GHASHed. **Interop-verified**: the
  output matches OpenSSL (EVP, `EVP_CTRL_GCM_SET_IVLEN`) byte-for-byte
  for AES-256 and AES-128 at 60/8/1-byte IVs, and the two 60-byte cases
  match the canonical McGrew-Viega GCM Test Cases 6 (AES-128) and 18
  (AES-256). A wrong J0 could not reproduce these.

## Memory safety + secret hygiene

`_gcm_compute_j0` writes exactly 16 bytes to `j0_out`; its `secret var
y[16]` (derived through the secret subkey `h`) is zeroized on scope exit
and the public `lenb[16]` is `memset` after use. The arbitrary-IV path
reads only `iv[0..iv_len)` — `_ghash_update` zero-pads internally without
reading past `iv_len`. The impls' existing `j0`/`counter`/`zero_block`/
`len_block` zeroizations are intact after the refactor. `iv_len ≤ 0` is
rejected before any use (`return -1` / `ERR_INVALID_INPUT`).

## Regression + API-additivity

The 12-byte path is byte-for-byte unchanged (the public 8-arg entries
pass `iv_len = 12` to the now-10-arg impls, which take the unchanged
fast-path J0). All pre-3.7.2 public signatures are preserved
(`aes_gcm_encrypt`/`_decrypt`, `aes_128_gcm_*`); the new surface is the
four `*_iv` entries (each doc-commented → `cyrius doc --check` stays at 0
undocumented). The only callers of the changed internal impls are the
public entries, all updated.

## CT posture

`_gcm_compute_j0` has no secret-dependent branch: the GHASH block count
derives from the public `iv_len`, and GHASH itself is data-independent.
The IV is public; the only secret is the subkey `h`, handled as before.
No new side-channel.
