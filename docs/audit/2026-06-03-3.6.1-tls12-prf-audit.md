## Correctness (load-bearing)

`PRF(secret, label, seed) = P_hash(secret, label || seed)` with
`A(0) = label||seed`, `A(i) = HMAC(secret, A(i-1))`, output =
`HMAC(secret, A(1)||seed') ‖ HMAC(secret, A(2)||seed') ‖ ...`. Verified
against the canonical 2009 IETF TLS WG PRF test vectors (the pair used
by mbedTLS / wolfSSL):

- **P_SHA256** — exact 100-byte output for secret `9bbe..db35`, label
  `"test label"`, seed `a0ba..198c`. This vector was independently
  reproduced with a stdlib-only Python `hmac`/`hashlib` reference
  **before** embedding; the reference matched byte-for-byte, which is
  the validation anchor for the SHA-384 emission.
- **P_SHA384** — exact 148-byte output for secret `b80b..67df`, label
  `"test label"`, seed `cd66..7465` (emitted by the validated
  reference; its leading bytes match the published companion vector).
- **Truncation** — a 12-byte and a 48-byte request equal the
  corresponding prefixes of the 100-byte stream (the `take =
  min(hlen, remaining)` partial-block path).
- **Determinism** and **return value** (`= out_len`, or `0` over-cap).

## Buffer safety

- `seedp[256]` holds `label || seed`; `slen = label_len + seed_len`
  is checked `> TLS12_PRF_SEED_MAX (256)` **before** the two
  `memcpy`s, so an over-cap (or malicious oversized) length returns 0
  with no out-of-bounds read/write. Confirmed by the over-cap test
  (`label_len = 300`).
- `hin[304] = 48 (max hlen) + 256`; the per-round write is `hlen +
  seedp_len ≤ 48 + 256 = 304` — in-bounds. `A[48]` / `block[48]` hold
  ≤ 48-byte digests.
- The output loop writes `take = min(hlen, out_len - filled)` bytes
  per round into `out + filled`, never exceeding `out_len`. Sizing of
  `out` to `out_len` is the caller's contract (documented), matching
  the rest of sigil's KDF surface (`hkdf_*`).

## Constant-time

No branch depends on secret data. The only conditionals are on
`is384` (public algorithm selector), the public `out_len` loop bound,
and the `slen > cap` length guard (lengths are public). `memcpy`
lengths (`hlen`, `seedp_len`, `take`) are public. Secret-dependent
timing is confined to the underlying HMAC, whose properties are
unchanged. The PRF adds none.

## Key zeroization

The intermediates that carry derived key material — `A`, `block`,
`hin` in `_tls12_p_hash`, and `seedp` in each public wrapper — are
`secret var`, so cyrius zeroizes them on scope exit (incl. the
over-cap early return). The caller owns `out` (the master secret /
key block).

## Threading

Not on the parallel batch-verify path; the PRF is a per-handshake,
single-thread KDF. Its static working buffers are therefore plain
function-scope scratch (NOT banked) — consistent with the rest of the
HMAC/HKDF surface, which is single-threaded by construction (e.g.
`hmac_sha256`'s `secret var kprime[64]`). No change to the 3.6
bank invariant.

## Findings

- **INFO-1** — the 256-byte `label || seed` cap. Acceptable; TLS 1.2
  never approaches it, and over-cap returns 0 (no silent truncation,
  no OOB). Documented in the module header and exercised by the test.

Zero CRITICAL / HIGH / MEDIUM / LOW.
