## 10-step Security Hardening checklist

1. **Input validation.** `poly1305_mac(key, msg, msg_len, out)`
   reads a fixed 32-byte key and writes a fixed 16-byte tag ‚Äî both
   caller-sized by contract (matches `hmac_sha256`,
   `aes_gcm_*`). `msg_len == 0` is handled (no blocks processed;
   tag = s). A negative `msg_len` falls through the
   `while (done < msg_len)` guard to the empty-message result ‚Äî no
   over-read, no crash.
2. **Buffer safety.** The only writable stack buffer is
   `block[16]`. The partial-block path writes `remaining`
   (`< 16`, so offset ‚â§ 15) message bytes plus the `0x01` byte at
   offset `remaining` (‚â§ 15) ‚Äî in bounds. Full blocks are read
   in place from `msg + done` where `done + 16 ‚â§ msg_len`. The
   overlapping `_p1305_u8to32` windows read at most `mp + 12 ..
   mp + 15` ‚Äî within each 16-byte block. `out` writes offsets
   0/4/8/12 (+4 each) ‚Üí 0..15. No adjacent-variable overflow.
3. **Constant-time on secret data.** The accumulator (`h0..h4`)
   and clamped key (`r*`, `s*`, `pad*`) are secret-derived. The
   multiply + carry chain is branch-free. The final freeze uses a
   sign-bit mask-select (`mask_g = sign - 1`, `mask_h = 0 - sign`
   with `sign = (g4 >> 63) & 1`) ‚Äî no branch on, and no early
   exit from, the accumulator. The `&1` isolates the sign bit so
   the result is independent of arithmetic-vs-logical `>>`. The
   one data-dependent branch (`if (remaining < 16)`) tests the
   **message length** only ‚Äî public in this threat model
   (ciphertext length is not secret), not key or accumulator
   state. `poly1305_verify` compares via the stdlib constant-time
   `ct_eq_bytes` and does not short-circuit on mismatch.
4. **Key material zeroization.** `block` (transient message
   bytes) and `computed` (recomputed tag in `poly1305_verify`)
   are `secret var` ‚Äî compiler-zeroized on every return path.
   The key-derived scalar locals (`r*`, `s*`, `pad*`, `t*`) are
   explicitly cleared before return. See INFO-1.
5. **Syscall review.** None. The module makes no syscalls.
6. **Pointer validation.** No raw deref of untrusted pointers;
   all access is via `load8`/`store8`/`load64` at
   contract-bounded offsets. No pointer arithmetic on
   external-controlled offsets.
7. **No command injection.** No `sys_system`/`exec_*`.
8. **No path traversal.** No filesystem access.
9. **Known-CVE check.** Poly1305 CVE classes reviewed: (a)
   carry-propagation / incomplete-reduction bugs that admit tag
   collisions ‚Äî the freeze adds 5, conditionally subtracts
   2^130, and selects in constant time, matching the donna
   reference exactly; verified against RFC ¬ß2.5.2. (b) Variable-
   time tag comparison (the classic AEAD forgery oracle) ‚Äî avoided
   via `ct_eq_bytes`. (c) **One-time-key reuse** (RFC ¬ß2.6) ‚Äî the
   single most dangerous Poly1305 misuse: authenticating two
   messages under one key leaks `r`. Mitigated by contract, not
   code: the security contract is documented in a boxed module
   header, and the AEAD bite will derive a fresh one-time key per
   nonce from the ChaCha20 keystream.
10. **Findings filed** ‚Äî below.

## Findings

### INFO-1 ‚Äî Best-effort scalar zeroization

`secret var` (Cyrius 5.3.5+) guarantees zeroization for stack
**arrays** only. The clamped-key scalar locals (`r0..r4`,
`s1..s4`, `pad0..pad3`, `t0..t3`) are cleared by explicit
`= 0` stores at function end, which a future DCE-enabled release
build could in principle elide. This is the same accepted stance
as the mldsa module-global workspace buffers (3.4.3 audit). The
secret here is a **caller-owned, one-time** key; residual
register/stack lifetime is low-risk. No action; tracked as a
defense-in-depth note should Cyrius gain scalar `secret`.

### INFO-2 ‚Äî Single-call static scratch (serial-only)

`block[16]` is a function-scope `var` array, which Cyrius
allocates as a static global (CLAUDE.md quirk #1), not a per-call
stack array. `poly1305_mac` fully writes it (memset ‚Üí memcpy ‚Üí
0x01) before reading, so **serial** use is correct. Concurrent
callers would race on the shared buffer. Poly1305 is not yet on a
multi-threaded path; the per-worker-scratch discipline arrives
with the v3.6 parallel-verify cycle. Documented in the module
header.
