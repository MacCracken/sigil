# Sigil 3.2 — Scope Tracker

Working document for the 3.2 cycle. Each item gets checked off
as it lands on the `main` branch. When the 3.2.0 batch ticks
clean, sigil tags `3.2.0` and the patch series in
[`3.2-tee-arc.md`](3.2-tee-arc.md) opens for 3.2.1 → 3.2.6.

**Branch strategy:** 3.2 work lands incrementally on `main`,
not on a long-lived branch. The 3.0 branch model was load-
bearing for the parallel-batch infrastructure (a single
breaking-API cut); the 3.2 cycle is additive (NI structural
fix, alloc-free hot path, then a new module surface for TEE
attestation) so a feature-branch-per-bite cadence suffices.

**Cyrius pin:** `6.0.1` (set in `cyrius.cyml` and
`.github/workflows/ci.yml` `CYRIUS_VERSION` env during the 3.1.2
ship-cut). cc6's "REAL TYPE SYSTEM" annotation pass + stdlib-
resolution path bug fixes are the load-bearing toolchain
features for 3.2 — they implicitly resolved
[`issues/archive/2026-05-10-ed25519-verify-aarch64-accepts-wrong-pk.md`](issues/archive/2026-05-10-ed25519-verify-aarch64-accepts-wrong-pk.md)
and silenced the x86_64 surface of
[`issues/2026-05-10-cyrius-510-asm-stack-frame-drift-breaks-ni-paths.md`](issues/2026-05-10-cyrius-510-asm-stack-frame-drift-breaks-ni-paths.md).

## Carry-over from the v3.1 backlog

The 3.1.x line shipped a stdlib-annotation pass (3.1.1) plus the
cyrius 6.0.1 / sakshi 2.2.5 / agnosys 1.2.7 dependency arc
(3.1.2). The originally-scoped "alloc-free verify hot path"
rewrite from `roadmap.md` § "Road to v3.1" did **not** land in
3.1.x — that arc reframed as toolchain modernisation before
the rewrite could start. It carries forward into 3.2.0 below.

## 3.2.0 work — defense + hot-path

### Core items

- [ ] **NI dispatch: parameters off the stack frame.** Migrate
      the three asm-block parameter loads in `src/aes_ni.cyr`
      (lines 60, 88–90) and `src/sha_ni.cyr` (lines 64,
      244–246) off hardcoded `[rbp-N]` byte literals. Pattern:
      a small non-asm thunk captures parameters into module-
      level globals (`_aes_ni_arg0`, `_aes_ni_arg1`,
      `_aes_ni_arg2`, mirrored for sha_ni), the asm block loads
      from `[rip+...]` instead. Same shape sigil already uses
      for the `_aes_ni_cache` / `_sha_ni_cache` probe sites.
      Tracked under
      [`issues/2026-05-10-cyrius-510-asm-stack-frame-drift-breaks-ni-paths.md`](issues/2026-05-10-cyrius-510-asm-stack-frame-drift-breaks-ni-paths.md).

      **Risk note:** the bug *does not currently reproduce*
      under cyrius 6.0.1 (24/24 tests green on the 3.1.2 ship-
      cut). This is a defense-in-depth fix — the structural
      defect (asm offsets coupled to a moving cyrius prologue
      contract) remains; cc6 just happens to emit a frame
      layout that matches the hardcoded offsets. A future
      cyrius prologue change re-breaks it identically. Lowest-
      risk migration; sigil's perf budget absorbs the
      indirection.

- [x] **Alloc-free verify hot path
      (`sv_verify_artifact_into` + `VerifyScratch`).**
      Landed 2026-05-21. The 3.0 path ran ~10 `fl_alloc` calls
      per artifact (`trust_check_new` × ~6, `trusted_artifact_new`,
      `verification_result_new`, `vec_new`, `hex_decode`,
      `hex_encode`); 3.2.0 collapses all of that into one
      `fl_alloc(count * 768)` pool on the main thread, with
      each worker writing intermediate state into its own
      slot. New helpers shipped alongside the existing API:
      `hex_decode_into`, `hash_file_into`, `sha256_init_into`,
      `verify_scratch_new`. Public API preserved 100% — every
      pre-3.2.0 accessor still works.

      **Mutex stayed.** The crypto modules use module-level
      globals for working state (`_sha_ctx`, `_sha_a..h`,
      `_sha256_W` in `sha256.cyr`; equivalents in `ed25519.cyr`
      / `aes_gcm.cyr`) as a cyrius local-clobbering workaround
      (CLAUDE.md quirk #1). Concurrent workers race on those
      globals. Verified empirically: mutex-off → 30/228
      `batch_parallel.tcyr` fail; mutex-on → 228/228 pass.

      Bench (cyrius 6.0.1, 4-worker pool):
      `sv_verify_batch_64` 423 ms (≈ 6.6 ms / artifact),
      essentially identical to per-artifact `sv_verify_batch_1`
      6.7 ms. Same throughput as 3.0's mutex-wrap; recorded
      in `benches/history.csv` under `v3.2.0-allocfree`. The
      original "3×+ at 4 workers" target slips to 3.3 — see
      "Towards v3.3" in roadmap.md.

      **What 3.2.0 delivers:** ~10× lower allocator churn,
      ~768 B per-artifact contiguous memory footprint, cleaner
      thread-safety story (the worker body is verifiably
      allocator-free; the remaining serialisation point is the
      crypto modules' global state — a single bottleneck to
      remove next cycle).

- [x] **Three-way bench: serial / mutex-wrap / alloc-free.**
      Landed 2026-05-21. Mutex-wrap (3.0) and alloc-free
      (3.2.0) rows added to `benches/history.csv`. The two
      run within noise of each other under the current crypto-
      global-serialisation regime — the throughput gap closes
      when the 3.3 per-worker crypto rewrite lands.

### Secondary items (ship-when-touched, not blocking the tag)

- [x] **`cyrlint` cleanup + workflow gate.** Landed 2026-05-21.
      Cyrlint baseline against `src/*.cyr` under cyrius 6.0.1
      surfaced 11 warnings (down from 35 in 2.9.5):
      - 7 × line-length warnings in `src/sha_ni.cyr` —
        decorative box-drawing-character separators inside the
        asm block; the unicode chars are 3 bytes each so a
        69-char visual line was 159 bytes physical. Replaced
        with ASCII `--` separators, dropping to 0 warnings.
      - 2 × multiple-blank-lines warnings in `src/trust.cyr`
        and `src/verify.cyr`. Trimmed the extra blank.
      - 2 × line-length warnings remaining in `src/aes_gcm.cyr`
        (`_aes_sbox_hex`, 512-char FIPS 197 §5.1.1 S-box hex
        literal) and `src/mldsa_ntt.cyr` (`_mldsa_zetas_hex`,
        2048-char NTT twiddle-factor table). Both are
        indivisible data tables — splitting would require
        30+ lines of init scaffolding per table for cosmetic
        compliance. Per-file exempted in CI workflow with a
        narrow rule: line-length warnings from these two
        files don't gate; any other warning class still does.

      CI gate added in `.github/workflows/ci.yml` as the
      `lint:` job, matching the agnosys / yukti pattern but
      with the documented exemption. Job runs `cyrius lint`
      against every `src/*.cyr` and `programs/*.cyr`;
      non-exempt warnings fail the build.

- [ ] **`secret var` ambient adoption.** ~20 `memset(..., 0,
      ...)` sites across `src/hkdf.cyr` / `src/aes_gcm.cyr` /
      `src/mldsa.cyr`; roughly 8 are true private-key / PRK /
      round-key / GHASH-state buffers, the rest intermediate
      scratch. `secret var` (cyrius 5.3.5) gives compiler-
      guaranteed zeroization. Land in-place when a 3.2 edit
      touches one of those call sites — no separate pass.

- [x] **Downstream re-test sweep.** Landed 2026-05-21:
      - phylax — `tlsh_distance(h, h)` SIGSEGV
        ([issue](issues/2026-05-11-tlsh-distance-segfault-phylax.md)
        updated 2026-05-21 with a 3.2.0 recheck footer).
        Sigil-side SHA-256 byte semantics are unchanged
        across 3.1.1 → 3.1.2 → 3.2.0; the only sha256 delta
        is the additive `sha256_init_into` helper + the
        `_sha_ni_self_test` gate, both verified against FIPS
        180-4 §B.1. A phylax-side bisect against either pin
        sees identical hash output. No sigil-side action owed
        until phylax confirms whether the segfault still
        reproduces against the new floor.
      - argonaut + libro — aarch64 cross-arch trust paths
        ([archived issue](issues/archive/2026-05-10-ed25519-verify-aarch64-accepts-wrong-pk.md)).
        Verified resolved under qemu-user 6.0.1 + sigil 3.1.2
        (right pk → 1, wrong pk → 0); 3.2.0 doesn't touch
        the ed25519 verify path. Real-silicon aarch64
        confirmation from a downstream production deploy
        will close out the issue permanently; not blocking
        the 3.2.0 tag.

## 3.2.x arc — TEE attestation

Tracked in [`3.2-tee-arc.md`](3.2-tee-arc.md). Each minor in the
arc lands as an independent patch on `main`, opening the next
work item only when the previous ships:

| Tag    | Module surface                                  | Unblocks |
|--------|-------------------------------------------------|----------|
| 3.2.1  | ECDSA P-256 verify (P-384 deferred)            | foundation |
| 3.2.2  | Minimal X.509 cert-chain walker                | foundation |
| 3.2.3  | `src/sgx.cyr` — quote parse + verify           | kavach SGX backend |
| 3.2.4  | `src/sev_snp.cyr` — VCEK chain + report verify | kavach SEV backend |
| 3.2.5  | `src/tdx.cyr` — TD-quote (shares SGX chain)    | kavach TDX backend |
| 3.2.6  | `src/seal.cyr` — SGX sealing (MRSIGNER/ISVSVN) | kavach SGX persistence |

Origin: [`issues/2026-05-10-kavach-sgx-sev-tdx-attestation-modules.md`](issues/2026-05-10-kavach-sgx-sev-tdx-attestation-modules.md).
Kavach rates the ask P1 on its side ("attestation is the load-
bearing primitive that turns the TEE-tier scoring claim from
'trusts the runtime started' into 'verifies the guest's
identity'"); sigil-side rating is **P3** (enhancement, no
current forcing function — kavach ships fine today). The arc
is independently shippable per-bite and each minor is useful
to any future first-party consumer of TEE runtimes (daimon,
AgnosAI, shakti).

## Out of scope for 3.2

- **ML-KEM-768 (PQC KEM).** Sigil is a verification / signing
  library; key agreement belongs in a sibling module (potential
  future `kem.cyr`) if a consumer surfaces a need.
- **Scatter-store for the fixed-base comb.** Single-tenant
  AGNOS doesn't have the threat model that requires it; queue
  if multi-tenant ever lands.
- **CLMUL-assisted GHASH.** GCM 1 KB encrypt / decrypt sits
  around 700 µs after AES-NI; GHASH bit-by-bit GF(2^128)
  multiply dominates. Hardware path attractive but the NI
  structural fix above is the prerequisite — once params-via-
  globals lands cleanly, a CLMUL dispatcher slots in on the
  same pattern. Not gating 3.2.
