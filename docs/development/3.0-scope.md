# Sigil 3.0 — Scope Tracker

Working document for the 3.0 cycle. Each item gets checked off
as it lands on the `3.0` branch. When all boxes are ticked, the
branch merges to `main` as a single PR and tags `3.0.0`.

**Branch strategy:** `3.0` branch cuts from `main` at `3b25549`
(post-2.9.1). All 3.0 work lands here. `main` stays shippable
for any 2.9.x emergency-hatch release (`-N` suffix pattern —
see sigil release-cadence notes).

**Cyrius pin:** bumped to `5.5.35` on the `3.0` branch (was
`5.5.30` on main). The 5.5.31 atomic primitives + 5.5.32
thread-safety audit are the load-bearing unblock for the
parallel batch-verify pillar. Active toolchain dispatches to
`5.5.35`; the pin is the minimum-compatible floor.

## 3.0 work

### Core features

- [x] **Parallel `sv_verify_batch` fan-out — infrastructure.**
      Landed in `src/verify.cyr`. Ships the spawn / join / shard /
      mutex scaffolding, the fixed 4-worker pool, and the
      `_SIGIL_BATCH_PARALLEL_THRESHOLD = 4` gate for the serial
      fast path on small counts. Test coverage in
      `tests/tcyr/batch_parallel.tcyr` (228 assertions, batch
      sizes 0 / 1 / 4 / 32, mixed signed/unknown/revoked, full
      determinism) passes.

      **Honest status on throughput:** the mutex-wrap approach
      wraps the full `sv_verify_artifact` call (because its
      downstream call chain — `hash_file`, `trust_check_new`,
      `hex_decode`, `verification_result_new`, `vresult_add_check`
      — hits `alloc` / `vec_push` / `map_get`, none thread-safe
      under cyrius (quirk #7). That means the dominant cost
      (`ed25519_verify` at ~6.4 ms per artifact under SHA-NI on
      the dev host) also runs under the lock, so workers fully
      serialise. Measured during the 5.5.35 bring-up: 0.96x – 1.04x
      across counts 1/4/16/64, i.e. no speedup, slight thread-
      overhead regression at count=64. Correctness is preserved;
      scaffolding is in place; **actual throughput gains defer to
      3.1** (Option 1 rewrite — see roadmap).

- [x] **Ungate `SIGIL_BATCH_PARALLEL` (originally a 3.1 task).**
      Landed on `3.0` branch (2026-05-01) once the upstream
      fixup-cap raise (cyrius 5.5.37, scheduled when filed
      2026-04-22) shipped and the active toolchain advanced to
      5.7.48. Removed the `#ifdef SIGIL_BATCH_PARALLEL` guards
      around `lib/thread.cyr`'s include (`src/lib.cyr`) and the
      worker fn / globals / parallel branch (`src/verify.cyr`).
      Default `cyrius build` compiles clean (one pre-existing
      benign agnosys-1.0.4 `sys_stat` x86_64-stub warning,
      unrelated); `tests/tcyr/batch_parallel.tcyr` 228/228
      green; full suite 24/24 green. Issue file
      `docs/development/issues/archive/2026-04-22-cyrius-fixup-cap-raises.md`
      flipped from "scheduled" to "resolved".

      **Remaining upstream cyrius dependency** (rolls forward to
      3.1):
      - Alloc / vec / hashmap thread-safety (quirk #7) forces the
        mutex-wrap shape rather than true lock-free worker
        isolation. The 5.5.32 design discussion deferred upstream;
        status on 5.7.48 needs an upstream check before the
        Option 1 rewrite starts.

### Breaking changes (3.0 bump)

- [x] **Remove `TRUST_COMMUNITY` enum variant.** Landed on
      `3.0` branch. Consumer audit across the 9 AGNOS repos
      (daimon, kavach, ark, aegis, phylax, mela, stiva,
      argonaut, takumi) turned up a single reference — an
      argonaut `lib/sigil.cyr` vendored bundle frozen at sigil
      v2.0.1, not active usage. Removed the variant, rank
      branch, and name branch from `src/types.cyr`; adapted
      `tests/tcyr/types.tcyr` (4 assertions removed,
      2 new assertions covering legacy-value → "unknown"
      degradation); swapped the one call site in
      `tests/tcyr/sigil.tcyr` to `TRUST_VERIFIED`. Numeric
      slot 2 intentionally unassigned (no renumbering) so
      persisted state keyed on values 3/4 decodes identically
      pre- and post-3.0. argonaut's vendored bundle picks up
      the change when its sigil pin is refreshed — not a 3.0
      blocker.
- [x] **Rename `alog_append_to_file` → `alog_save`** (and
      `alog_load_from_file` → `alog_load`) to match the
      `rl_save` / `crl_save` / `sv_save_trust_store` vocabulary.
      Landed on `3.0` branch. Both `fn` definitions in
      `src/audit.cyr` renamed; one stale comment in
      `src/policy.cyr:213` updated; test call sites in
      `tests/tcyr/sigil.tcyr` (4 usages) updated. Module-header
      comments retain "Renamed from <old> in 3.0" migration
      notes for one cycle. Historical CHANGELOG entries (under
      shipped 2.x versions) left intact — do not rewrite
      history. `sigil.tcyr` still 96/96 green.
- [x] **Rewrite the stale "drop `hmac.cyr`" roadmap entry.**
      Landed on `3.0` branch. `docs/development/roadmap.md`
      entry struck-through with "superseded" framing (2.9.0's
      HKDF landing pulled `hmac_sha256` into sigil's internal
      call graph, invalidating the "no sigil-internal caller"
      premise). `src/hmac.cyr` module header gains a new
      status block documenting hmac as a sigil-internal HKDF
      implementation detail — public consumers should use
      `hkdf_*` rather than calling `hmac_sha256` directly.
      Tests retain hmac-level vectors as unit coverage of the
      primitive. No public surface or binary-layout change.
- [x] **Re-evaluate the `SIGIL_PQC` feature flag.** Empirically
      tested on cc5 5.5.32. Cap is still 1 MB (unchanged from
      cc3-era `src/frontend/lex.cyr:1436`). Measured expansions:
      default 930 KB, `-D SIGIL_PQC` cmdline 1047 KB (squeaks
      under cap by ~1 KB), unconditional-include 1,049,596 B
      (over cap by 1020 B). Moving the define from cmdline into
      source (via `#define SIGIL_PQC 1` at top of lib.cyr) also
      overflows — the IFDEF pass accounts bytes differently for
      cmdline-`-D` vs in-source `#define`. PQC therefore stays
      a cmdline-opt-in for the 3.0 cycle. `src/lib.cyr` header
      comment updated with the actual measurements + the "until
      cyrius raises the buffer" trigger. Documented in CLAUDE.md
      as quirk #8. Revisit when cyrius bumps
      `preprocess_out` beyond 1 MB.

### Ambient — adopt when touching the relevant code

- **`secret var` key-material qualifier.** Available in cyrius
  since 5.3.5 (already used at `src/hmac.cyr:19` for the
  HMAC key-prime buffer). Not a standalone pillar — adopt
  in-place whenever a future edit touches one of the remaining
  `memset(sk, 0, N)` patterns on private-key / PRK / round-key
  buffers (ed25519 signing, ML-DSA signing, HKDF expand,
  AES-GCM). `secret var` gives the compiler-guaranteed
  zeroization the manual `memset` patterns approximate, so
  the replacement is strictly better. When parallel
  `sv_verify_batch` and other 3.0 pillars touch key-material
  call sites, swap the manual zeroization to `secret var` as
  part of the same commit.

### Deferred — still blocked on cyrius

- **Alloc-level thread safety.** Cyrius 5.5.32 "deferred" the
  design (per-thread arena vs full-`alloc` spinlock vs
  CAS-bump-ptr are all in flight upstream). Sigil's batch-verify
  design sidesteps this by pre-allocating worker scratch
  upfront. Revisit only if a future consumer needs mid-verify
  allocation from worker threads.

## Not in scope

Out-of-scope for 3.0 (would bloat the cycle unless a consumer
specifically needs it):

- **Scatter-store for the fixed-base comb.** Cache-timing
  hardening. AGNOS is single-tenant; queue only if the threat
  model shifts.
- **Dedicated `sigil_batch_verify_parallel` entry point.**
  Not needed if `sv_verify_batch` fans out by default at the
  threshold-`count`. The single entry point is the simpler API.
- **PQC KEM (ML-KEM-768).** Sigil is a verification / signing
  library; key-agreement belongs in a sibling module
  (potential future `kem.cyr`) if any AGNOS consumer needs it.

## Post-3.0 follow-ups (recorded so they don't evaporate)

The 3.0 parallel `sv_verify_batch` landing uses **mutex-wrap
approach** (single sigil-internal mutex around `alloc` / `vec_*`
/ `map_*` hot spots so worker threads can safely run the verify
path concurrently). This is the pragmatic "ship the pillar now"
choice given that the expensive work (`ed25519_verify` at ~8.4ms,
`hash_file` at 250µs for 4 KB) dwarfs the alloc bookkeeping —
mutex contention on short alloc critical sections is real but
bounded.

Two deeper refactors were evaluated and **deliberately deferred**
out of 3.0 scope. They remain valuable future work:

- **Pure alloc-free verify hot path ("Option 1").** Refactor
  `sv_verify_artifact` and its call chain to accept caller-
  provided scratch instead of allocating internally. Touch list
  from the 3.0 spike:
  - `hash_file` returns a caller-provided hex buffer rather
    than `alloc`-ing one per call
  - `hex_decode` writes into a caller-provided 32-byte buffer
  - `trust_check_new` + `vresult_add_check` → replace the
    per-check vec with a bounded fixed-size check array on
    `verification_result` (the standard verify path emits ≤ 6
    checks; size accordingly with room to grow)
  - `verification_result_new` → allow in-place construction into
    a pre-allocated output slot
  - `map_get` on the trust store — read-only contract documented
    (stdlib `hashmap` needs an explicit "read-only is safe under
    threads" statement, or we wrap in a rwlock)

  Expected wins: better parallel scaling (no mutex contention on
  bookkeeping), lower per-artifact latency, cleaner thread-safety
  story, smaller attack surface for the security audit. Worth
  taking when we have a consumer actually bottlenecked on
  batch-verify throughput, or as a 3.x minor after we've watched
  production workloads on the mutex-wrap version.

- **Per-worker arena allocator ("Option 3").** Give each worker
  its own bump-allocator arena; shard the batch N-ways; merge
  vecs on the main thread. Middle ground between mutex-wrap and
  pure pre-alloc. Requires a small arena primitive we don't have
  yet — could live in sigil as an internal helper or be proposed
  upstream to Cyrius stdlib. Not needed if Option 1 lands, since
  Option 1 removes the need for a per-worker arena entirely.

Neither is blocked on Cyrius. Schedule when a consumer surfaces
batch-verify throughput as the bottleneck.
