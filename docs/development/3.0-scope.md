as it lands on the `3.0` branch. When all boxes are ticked, the
branch merges to `main` as a single PR and tags `3.0.0`.

**Branch strategy:** `3.0` branch cuts from `main` at `3b25549`
(post-2.9.1). All 3.0 work lands here. `main` stays shippable
for any 2.9.x emergency-hatch release (`-N` suffix pattern —

## 3.0 work

### Core features

- [ ] **Parallel `sv_verify_batch` fan-out.** `src/verify.cyr:827`
      currently serialises. Design constraint: `lib/alloc.cyr`
      is NOT thread-safe on its Linux bump-pointer path (5.5.32
      audit finding). Approach: pre-allocate every per-worker
      scratch buffer in the main thread before spawn; workers
      read read-only globals (fixed-base comb, round-key
      tables) and write to their own output slots. No shared
      mutable state during verify. Single-thread fast-path
      stays for small `count`. `lib/thread.cyr` mutex is
      race-free as of 5.5.31 — use it for the shared result
      counter if needed, or skip mutexes entirely with atomic
      slot writes from `lib/atomic.cyr`.

### Breaking changes (3.0 bump)

- [x] **Remove `TRUST_COMMUNITY` enum variant.** Landed on
      `3.0` branch. Consumer audit across the 9 AGNOS repos
      (daimon, kavach, ark, aegis, phylax, mela, stiva,
      argonaut, takumi) turned up a single reference — an
      argonaut `lib/sigil.cyr` vendored bundle frozen at sigil
      v2.0.1, not active usage. Removed the variant, rank
      branch, and name branch from `src/types.cyr`; adapted
      `tests/tcyr/types.tcyr` (4 assertions removed,
      2 new assertions covering legacy-value → "unknown"
      degradation); swapped the one call site in
      `tests/tcyr/sigil.tcyr` to `TRUST_VERIFIED`. Numeric
      slot 2 intentionally unassigned (no renumbering) so
      persisted state keyed on values 3/4 decodes identically
      pre- and post-3.0. argonaut's vendored bundle picks up
      the change when its sigil pin is refreshed — not a 3.0
      blocker.
- [x] **Rename `alog_append_to_file` → `alog_save`** (and
      `alog_load_from_file` → `alog_load`) to match the
      `rl_save` / `crl_save` / `sv_save_trust_store` vocabulary.
      Landed on `3.0` branch. Both `fn` definitions in
      `src/audit.cyr` renamed; one stale comment in
      `src/policy.cyr:213` updated; test call sites in
      `tests/tcyr/sigil.tcyr` (4 usages) updated. Module-header
      comments retain "Renamed from <old> in 3.0" migration
      notes for one cycle. Historical CHANGELOG entries (under
      shipped 2.x versions) left intact — do not rewrite
      history. `sigil.tcyr` still 96/96 green.
- [ ] **Rewrite the stale "drop `hmac.cyr`" roadmap entry.**
      Original plan was to drop hmac entirely, but `src/hkdf.cyr`
      uses `hmac_sha256` internally (4 call sites). New stance:
      hmac stays as sigil-internal, documented as an HKDF
      implementation detail. Public surface exposes `hkdf_*` /
      `hkdf_extract` / `hkdf_expand`. Update roadmap + module
      header in `src/hmac.cyr`.
- [x] **Re-evaluate the `SIGIL_PQC` feature flag.** Empirically
      tested on cc5 5.5.32. Cap is still 1 MB (unchanged from
      cc3-era `src/frontend/lex.cyr:1436`). Measured expansions:
      default 930 KB, `-D SIGIL_PQC` cmdline 1047 KB (squeaks
      under cap by ~1 KB), unconditional-include 1,049,596 B
      (over cap by 1020 B). Moving the define from cmdline into
      source (via `#define SIGIL_PQC 1` at top of lib.cyr) also
      overflows — the IFDEF pass accounts bytes differently for
      cmdline-`-D` vs in-source `#define`. PQC therefore stays
      a cmdline-opt-in for the 3.0 cycle. `src/lib.cyr` header
      comment updated with the actual measurements + the "until
      cyrius raises the buffer" trigger. Documented in CLAUDE.md
      as quirk #8. Revisit when cyrius bumps
      `preprocess_out` beyond 1 MB.

### Deferred — still blocked on cyrius

- **`secret var` key-material qualifier.** Not in cyrius 5.5.32.
  If it ships during the 3.0 cycle, adopt it mechanically
  across the 8 hot-path `memset(sk, 0, …)` call sites. If not,
  carry forward to 3.1.
- **Alloc-level thread safety.** Cyrius 5.5.32 "deferred" the
  design (per-thread arena vs full-`alloc` spinlock vs
  CAS-bump-ptr are all in flight upstream). Sigil's batch-verify
  design sidesteps this by pre-allocating worker scratch
  upfront. Revisit only if a future consumer needs mid-verify
  allocation from worker threads.

## Not in scope

Out-of-scope for 3.0 (would bloat the cycle unless a consumer
specifically needs it):

- **Scatter-store for the fixed-base comb.** Cache-timing
  hardening. AGNOS is single-tenant; queue only if the threat
  model shifts.
- **Dedicated `sigil_batch_verify_parallel` entry point.**
  Not needed if `sv_verify_batch` fans out by default at the
  threshold-`count`. The single entry point is the simpler API.
- **PQC KEM (ML-KEM-768).** Sigil is a verification / signing
  library; key-agreement belongs in a sibling module
  (potential future `kem.cyr`) if any AGNOS consumer needs it.
