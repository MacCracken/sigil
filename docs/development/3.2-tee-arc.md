# Sigil 3.2.x — TEE Attestation Sub-Arc

Working plan for the 3.2.1 → 3.2.6 patch series that lands SGX /
SEV-SNP / TDX attestation primitives. The arc opens once 3.2.0
ships (see [`3.2-scope.md`](3.2-scope.md)); each minor below
tags independently on `main` as it lands.

**Origin:** [`issues/2026-05-10-kavach-sgx-sev-tdx-attestation-modules.md`](issues/2026-05-10-kavach-sgx-sev-tdx-attestation-modules.md).
**Reporter:** kavach (AGNOS sandbox-execution framework).
**Reporter severity:** P1. **Sigil-side severity:** P3
(enhancement — kavach ships fine today; the gap is a tier-
claim-vs-reality drift, not a blocker).
**Sequencing rationale:** the format-specific parsers compose
against ECDSA + X.509, so those land first. SGX lands ahead of
SEV-SNP / TDX because TDX shares the SGX cert chain (smaller
delta once SGX is in tree) and SEV-SNP's chain (ARK → ASK →
VCEK) is the cleanest "second chain shape" to validate the
abstraction against.

## Crypto primitives already in tree

The 2.9.x crypto pillar already covers the bulk of what an
attestation verifier needs:

| Primitive       | Module                   | Used for                          |
|-----------------|--------------------------|-----------------------------------|
| SHA-256/512     | `src/sha256.cyr` / `src/sha512.cyr` | Quote body hash         |
| HMAC-SHA256     | `src/hmac.cyr`           | DICE / sealing key derivation     |
| Constant-time   | `lib/ct.cyr` (stdlib)    | Cert-chain comparisons            |
| HKDF-SHA256     | `src/hkdf.cyr`           | Sealed-key derivation             |
| Ed25519         | `src/ed25519.cyr`        | (Not directly used — SGX/SEV/TDX rely on ECDSA P-256) |
| TPM / IMA / SB  | `src/tpm.cyr` etc.       | Host-side attestation             |

**The gap:** ECDSA P-256 verify, minimal X.509 cert-chain
parsing, and three quote-format parsers. The patch series below
lands them in dependency order.

## 3.2.1 — ECDSA P-256 verify  ✅ SHIPPED (2026-05-21)

Landed in `src/ecdsa_p256.cyr` (~680 lines). Test surface in
`tests/tcyr/ecdsa_p256.tcyr` covers all 9 groups with 49 assertions;
bench row `v3.2.1,ecdsa_p256_verify,136.283,ms` in
`benches/history.csv`. Audit doc: `docs/audit/2026-05-21-audit.md`
(0 CRITICAL/HIGH/MEDIUM, 2 LOW). Solinas reduction deferred to the
bench-tuning follow-up — slow field reduction is documented at
INFO-2 in the audit and is not a security defect.

**Surface:** `src/ecdsa_p256.cyr` with two public fns —

```
fn ecdsa_p256_verify(
    pk: i64,        # 64-byte uncompressed pk (Qx || Qy)
    msg: i64,
    msg_len: i64,
    sig: i64        # 64-byte sig (r || s, big-endian)
): i64              # 1 = valid, 0 = invalid
```

…and a sig-length-tolerant DER-decoder helper for the X.509
signature shape consumers hit (`ecdsa_p256_verify_der`).

**Constant-time discipline.** Reuse the OR-accumulation pattern
from `ed25519.cyr` for the final R comparison; lift the field
arithmetic from RFC 6979 + SEC 2 §2.7.1. No external dep — the
bigint surface in `lib/bigint.cyr` plus the existing modular
inverse pattern in `src/ed25519.cyr` carry the load.

**Test vectors:** NIST CAVS ECDSA P-256 SigVer vectors
(P-256 SHA-256 SigVer.rsp), at minimum the first 15 pass cases
+ all 4 fail-class cases (BadSig, BadHash, BadKey,
ModifiedMessage). One round-trip test pairs `ecdsa_p256_sign`
(internal-only sign for vector generation) with the public
verify.

**Decision deferred to 3.2.x:** P-384 surface. TDX quotes
sometimes carry P-384 attestation keys; if the kavach
integration confirms the deployment targets need P-384, slot a
`src/ecdsa_p384.cyr` between 3.2.5 and 3.2.6. Otherwise P-256
is sufficient.

## 3.2.2 — Minimal X.509 cert-chain walker  ✅ SHIPPED (2026-05-22)

Landed in `src/x509.cyr` (~720 lines). Test surface in
`tests/tcyr/x509.tcyr` covers all 8 groups with 35 assertions;
openssl-generated test certs hex-embedded in the test file (sigil
has no sign yet — sign deferred per 3.2.1 audit INFO-1). Audit
doc: `docs/audit/2026-05-22-audit.md` (0 CRITICAL/HIGH/MEDIUM, 1
LOW + 2 INFO). LOW-1 from 2026-05-21 closed — `pt_is_on_curve`
landed alongside in `src/ecdsa_p256.cyr` and is now invoked by
both verify entry points (ECDSA and X.509 SubjectPublicKeyInfo).

**Surface:** `src/x509.cyr` —

```
fn x509_parse(der: i64, der_len: i64, out_cert: i64): i64
fn x509_verify_chain(
    leaf: i64,        # parsed cert
    intermediates: i64, intermediates_len: i64,
    root: i64         # parsed trust root
): i64
```

**Scope:** **minimal** — just enough to walk IAS root →
intermediate → leaf (SGX EPID) and ARK → ASK → VCEK (SEV-SNP).
Not a general PKIX library: no policy mapping, no name
constraints, no CRL fetching, no general TLS chain walk.

The TBS-cert hash + signature check uses the 3.2.1 ECDSA verify.
Key usage bits and basic constraints are parsed but only
checked against a small allow-list (CA bit must be set on
intermediates and the root; leaf must be end-entity). Time
validity uses `chrono` from the stdlib (`lib/chrono.cyr`,
already in deps).

**Test vectors:** the Intel IAS-root + SGX intermediate +
known-good attestation report (from Intel's published quote
samples); the AMD ARK-root + ASK + a synthetic VCEK signed
under a test key for the unit suite (real VCEKs are per-CPU and
not redistributable).

## 3.2.3 — `src/sgx.cyr` (SGX quote parser + verifier)

**Surface:**

```
struct SgxQuote {
    version: u16, sign_type: u16, qe_svn: u16, pce_svn: u16,
    fmspc: u8[6], qe_vendor_id: u8[16],
    user_data: u8[20],
    report_body: SgxReportBody,    # MRENCLAVE, MRSIGNER, ISVPRODID, ISVSVN, ATTRIBUTES, report_data
    signature_data: u8[*],          # variable-length
}

fn sgx_quote_parse(buf: i64, buf_len: i64, out: i64): i64
fn sgx_quote_verify(quote: i64, ias_root: i64): i64
```

**Composition:** parser is straight DER + Intel quote-format
spec (linear scan, bounds-checked at every offset). Verifier
composes against:
- SHA-256 over the report body (existing `sha256.cyr`).
- ECDSA P-256 signature check (3.2.1) on the quote signature.
- IAS / DCAP cert chain walk (3.2.2) up to the IAS root.

**Test vectors:** Intel's published SGX quote samples (in the
SGX SDK source tree at `external/sgx-tee/sample_attestation/`).
At minimum: one DCAP quote (current path) plus one EPID quote
(legacy path) if any kavach deployment still uses it. EPID can
be deferred to a later 3.2.x bite if not actively needed.

**Unblocks:** kavach SGX backend attestation verify
(`kavach/src/attestation.cyr` populates the existing
`SgxAttestationReport` placeholder once the parser ships).

## 3.2.4 — `src/sev_snp.cyr` (SEV-SNP guest attestation)

**Surface:**

```
struct SevSnpReport {
    version: u32, guest_svn: u32, policy: u64,
    family_id: u8[16], image_id: u8[16],
    vmpl: u32, signature_algo: u32,
    current_tcb: u64, platform_info: u64,
    report_data: u8[64],
    measurement: u8[48],          # SHA-384 guest measurement
    host_data: u8[32], id_key_digest: u8[48], author_key_digest: u8[48],
    report_id: u8[32], report_id_ma: u8[32],
    reported_tcb: u64,
    chip_id: u8[64],
    signature: u8[*],
}

fn snp_report_parse(buf: i64, buf_len: i64, out: i64): i64
fn snp_report_verify(report: i64, ark_root: i64): i64
```

**Cert chain shape:** ARK (AMD root) → ASK (AMD signing) → VCEK
(per-CPU). The VCEK is fetched per-host once; sigil's verifier
accepts a caller-provided VCEK rather than fetching itself
(networking is not sigil's job).

**Decision in scope:** measurement is SHA-384. The 2.9.x crypto
pillar shipped SHA-512 (`src/sha512.cyr`); SHA-384 is the
truncated variant. Decision: extend `sha512.cyr` with a
`sha384` entry point that shares the compression core, or add
a small `sha384.cyr` wrapper. Latter is cleaner (parallel to
sha256 / sha512 / hkdf separation). Slot the wrapper as a
single-commit prerequisite at the head of 3.2.4.

**Test vectors:** AMD's published SEV-SNP report samples (in
the AMD SEV firmware repo). At minimum one current-firmware-
version report with a synthetic VCEK chain.

## 3.2.5 — `src/tdx.cyr` (TDX TD-quote)

**Surface:**

```
struct TdxQuote {
    version: u16, attestation_key_type: u16,
    tee_type: u32, tcb_evaluation_data_number: u16,
    qe_vendor_id: u8[16], user_data: u8[20],
    td_report: TdReport,           # MRTD, RTMR0..3, TD attributes
    signature_data: u8[*],
}

fn tdx_td_report_parse(buf: i64, buf_len: i64, out: i64): i64
fn tdx_quote_verify(quote: i64, ias_root: i64): i64
```

**Cert chain shape:** TDX quotes are signed by the SGX quoting
enclave on the same host — the chain from the QE-signing key
up to the IAS root is identical to SGX. Verifier composes
against the 3.2.3 SGX chain walker and only owns the TD-specific
report-body parsing + TDX-specific TCB evaluation.

**Test vectors:** Intel's published TDX quote samples.

## 3.2.6 — `src/seal.cyr` (SGX sealing)

**Surface:**

```
fn sgx_seal_key(
    mrsigner_or_mrenclave: i64,     # 32-byte measurement
    isvsvn: i64,                    # 16-bit security version
    key_id: i64, key_id_len: i64,
    out_key: i64                    # 32-byte derived key
): i64

fn sgx_unseal_key(...): i64         # symmetric inverse
```

**KDF:** sealing key derivation is a constrained KDF against the
enclave's CPU-resident sealing root (the EGETKEY result). sigil
doesn't have access to EGETKEY directly; this surface assumes
the caller has already obtained the sealing root (from the
runtime's enclave entry) and treats it as the KDF input. HKDF-
SHA256 (`src/hkdf.cyr`, 2.9.x) covers the derivation step.

**Sequencing:** last in the arc because sealing is rarely the
gating feature for downstream — quote verify is. Defer cleanly
if any of the higher-priority items overruns its budget.

## Closeout criteria — TEE arc

Each minor in this arc ships independently. No global tag
gates the arc; the arc closes when 3.2.6 lands. Kavach's
`src/attestation.cyr` integration is the canonical downstream
test bed for each minor — every tag in this arc should clear
the corresponding `kavach/tests/tcyr/backend_*.tcyr` group.

If the arc stalls (kavach's TEE roadmap shifts, or a higher-
priority sigil item like ML-KEM-768 jumps the queue), the
incomplete arc parks with a clean cutoff at the last shipped
minor — no half-implemented module surface in tree.
